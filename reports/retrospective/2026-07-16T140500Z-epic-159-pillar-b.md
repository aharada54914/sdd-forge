## Friction Patterns

### FP-01: `design.md`'s Constraint Compliance table needed two separate Critical-severity rounds to reach completeness — one row per *distinct repeated requirement constraint*, not one row per generically-named threat category

- **Evidence:** Impl review round 1 (`reports/impl-review/epic-159-pillar-b/attempt-1/round-1/reviewer-b.json`, `NO-REQ-CONTRADICTION`, Critical, `verdict: BLOCKED`): `requirements.md`'s single most-repeated explicit constraint — "no bypass / no weakened-gate escape hatch" (stated independently in REQ-001's goal text, AC-004, the Security Boundaries table's B2 row, OQ-007, and the Risks section's own "High" item) — had **no corresponding row** in `design.md`'s 10-row Constraint Compliance table. Fixed by adding that row. Round 2 (`.../round-2/reviewer-b.json`, `NO-REQ-CONTRADICTION`, Critical, `verdict: BLOCKED` again): the round-1 fix correctly added the "no bypass" row, but a **second, distinct, equally-repeated** constraint — the fail-closed pre-mutation **ordering and zero-mutation-on-failure** guarantee (REQ-001 body, Edge Cases, AC-002/AC-003, AC-005) — was still uncited by any of the now-11 rows. Round 2's finding explicitly frames these as different properties ("a future edit that reorders `bump-version.sh`... would violate REQ-001 without triggering any Constraint Compliance mismatch, since TEST-004's grep self-check only scans for conditional-wrapping, not line position"), i.e. the first fix closed one repeated-constraint gap but a table-completeness check applied only to the newly-added row, not swept against every other repeated constraint in `requirements.md`. Round 3 (`.../round-3/`) reached a clean PASS.
- **Frequency:** 2 consecutive Critical findings in the same task/feature, both targeting the same underlying table (`design.md`'s Constraint Compliance section) but two different specific constraints — a within-feature recurrence of the identical *shape* of gap.
- **Phase:** design-review (impl-review stage).
- **Confidence:** High (both findings are reviewer-authored, file:line-cited, and independently corroborated by an integrated-verdict field flip from `NEEDS_WORK` to `PASS` only after both rows existed).
- **Do Not Overfit:** both instances were caught cleanly by the review loop itself before implementation began (impl-review gates design.md, not code) — zero task attempts, zero quality-gate rework, zero review tickets. The lesson is a **design-authoring** principle, not a review-process failure: "Constraint Compliance table completeness" must be checked against every *individually repeated* constraint in requirements.md (each may need its own row), not against a coarser per-threat-category checklist where fixing one exemplar row is mistaken for closing the whole category.

## Data Notes

- **Good deviation example — GitHub Actions permission-inheritance gap caught and closed by the implementer, not deferred to code review (T-002):** `design.md`'s literal "Planned shape" snippet for the new `loop-gate` job omitted a job-level `permissions:` override. Because GitHub Actions job-level permissions silently inherit the workflow-level grant (here, `contents: write` / `id-token: write` / `attestations: write`) unless a job declares its own narrower block, implementing the snippet verbatim would have left `loop-gate` silently over-permissioned — directly contradicting `security-spec.md` B4 and its Secrets Management section, both of which explicitly asserted "no elevated permissions" as a hard requirement. The implementer added `permissions: contents: read` directly on the job (`release.yml:32-39`) rather than shipping the gap for a human reviewer to catch at "code review" time as the spec's own control column had proposed — closing exactly the "Elevation of Privilege" risk `security-spec.md`'s OWASP Mapping table named. Recorded as a positive pattern: implementers should reconcile a literal design snippet against the spec's own stated security intent, not just against the snippet's literal text, when the two could plausibly diverge under a platform's own default-inheritance behavior.
- **Two BSD/macOS-portability self-fixes, both matching already-catalogued classes:** (1) T-001's `build_fixture()` appended to `CLEANUP_ROOTS` from inside a command-substitution subshell, so the array mutation never propagated to the parent shell — combined with `"${CLEANUP_ROOTS[@]:-}"`'s empty-array-safety idiom, this silently forced the suite's exit code to 1 under `set -e` inside its own `EXIT` trap regardless of actual pass/fail counts; fixed by moving the append to each calling test function's own non-subshell scope. (2) T-002's scratch python3 checker used `mktemp "...checker.XXXXXX.py"` (a literal suffix *after* the `XXXXXX` run) — BSD/macOS `mktemp`, unlike GNU `mktemp`, only randomizes a trailing run of `X` characters when it is the very last thing in the template, so this returned the un-randomized literal path, succeeding once by accident and then colliding ("File exists") on every subsequent run; fixed by `mktemp -d`-ing a directory with a bare `XXXXXX` suffix and writing a fixed-name file inside it. **Both defect classes are already recorded in this session's own coder memory** (`feedback-bash-cleanup-roots-subshell.md`, `feedback-bsd-mktemp-suffix-after-x.md`) from prior features; both were self-caught and self-fixed during implementation this period, before any review or gate stage saw them — direct evidence the catalogued lessons are being applied proactively, not just referenced reactively.
- **Follow-up-issue candidate (not filed by this retrospective): BSD/macOS `sed` incompatibility in `scripts/bump-version.sh`'s unedited mutation section.** T-001 discovered (via its own TDD, not introduced by it) that `scripts/bump-version.sh`'s pre-existing mutation section uses GNU-only `sed -i "<script>" "<file>"` syntax, incompatible with BSD/macOS `sed` (which requires a backup-extension argument after `-i`). This had never been exercised by any automated suite before this task (a pre-task grep over `tests/*.tests.sh` and `.github/workflows/*.yml` found zero references to `bump-version`).
