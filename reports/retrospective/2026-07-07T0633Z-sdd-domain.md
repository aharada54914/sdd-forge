## Friction Patterns

Patterns observed across two or more tasks in this period.

### FP-01: Placeholder-scan false positives on prose uses of marker words

- **Evidence:** T-002, T-009, T-011 implementation reports each carry a
  `## Quality-Gate Fix (placeholder-scan false positives)` section (confirmed
  in gate reports `T-002-sdd-domain.md:19-20,28`, `T-009-sdd-domain.md:18,25`,
  `T-011-sdd-domain.md:19`) — prose mentioning the word "placeholder"/"todo"
  required meaning-preserving rewording before `check-placeholders` exited 0.
  T-008's first gate run FAILED outright on 2 pre-existing prose lines in
  `quality-gate/SKILL.md` (RT-20260706-001), where rewording was impossible
  because the flagged text is the scanner's own script name.
- **Frequency:** 4 tasks (T-002, T-008, T-009, T-011)
- **Phase:** quality-gate (Step 6 deterministic scripted checks)
- **Confidence:** High (4 tasks, 2 evidence types: gate reports + RT ticket)
- **Do Not Overfit:** This is not one unlucky file — a skill-authoring feature
  necessarily writes documentation *about* placeholders and TODOs, and any
  future documentation-heavy feature will re-trip it. The human-applied
  RT-20260706-001 fix (case-sensitive marker keywords in both script twins)
  addresses the largest class; whether it fully suppresses recurrence is
  exactly the kind of claim the next retrospective should verify against
  fresh evidence rather than assume.

### FP-02: Stale evidence in implementation reports (hashes/prose drift after fixes)

- **Evidence:** Minor findings of the same class in five tasks' gate reports:
  T-005 (stale secondary SKILL.md hash, cycle 1 *and* another stale hash in
  cycle 2 — `T-005-sdd-domain.md:49,68`), T-006 (report prose stale relative
  to the applied hook change — `T-006-sdd-domain.md:75`), T-007
  (stale historical hash/count text — `T-007-sdd-domain.md:60`), T-008
  (manifest `identity_ledger_sha256` no longer matching on-disk ledger —
  `T-008-sdd-domain.md:60`), T-011 (unverifiable "62/62" claim about T-005's
  suite — `T-011-sdd-domain.md:68`).
- **Frequency:** 6 occurrences across 5 tasks
- **Phase:** implementation-report authoring / quality-gate evidence check
- **Confidence:** High (5 tasks, single evidence type but uniform)
- **Do Not Overfit:** Every occurrence has the same mechanism — a report
  written before a late fix commit keeps pre-fix hashes or counts. This is a
  process property (reports are hand-maintained snapshots), not a property of
  any one task's content.

### FP-03: Tests encoding incidental environment/working-tree state force second critical-review cycles

- **Evidence:** T-007 cycle-1 `NEEDS_WORK` (2 Major): a regression test was
  anchored to the *working tree* and broke once the edits were committed
  (`T-007-sdd-domain.md:37-43`). T-005 cycle-1 `NEEDS_WORK` (1 Major): the
  jq-wall tests hard-coded the assumption that jq is absent, which broke when
  jq 1.7.1 + MSYS shim were installed mid-feature under RT-20260706-002
  (`T-005-sdd-domain.md:41-49`). Both were fixed and re-verified by fresh
  evaluators in cycle 2.
- **Frequency:** 2 tasks (the only 2 multi-cycle tasks of the period — 100%
  of extra evaluator cycles trace to this pattern)
- **Phase:** implementation (test design) surfacing at quality-gate
- **Confidence:** Medium (2 tasks, per the two-task recurrence rule)
- **Do Not Overfit:** Different surface (git state vs. tool availability),
  same root cause: a test asserting the *current environment* rather than the
  contracted behavior. Both failure modes are reproducible hazards for any
  feature whose gate window spans commits or environment changes.

### FP-04: Windows tooling wall in deterministic gates (jq / MSYS / provenance cascade)

- **Evidence:** T-008 first gate run BLOCKED: `check-workflow-state` reported
  "registry is not valid JSON" when the actual failure was `jq: command not
  found` (exit 127) — a misleading diagnostic (RT-20260706-002, major). The
  authorized fix (jq 1.7.1 + sha256-verified MSYS-safe shim) then unmasked a
  second, previously hidden finding: sdd-domain's task-review stage
  provenance tripped `check-workflow-state`, resolved by a human-decided
  registry grandfather (RT-20260707-001, major; registry + schema edits).
  RT-20260707-001's resolution also records a known follow-up: the pinned
  `tests/workflow-state-registry.tests.sh` contract predates the sdd-domain
  entry and was already failing.
- **Frequency:** 1 task directly (T-008) but 3 evidence artifacts (2 RTs + 1
  BLOCKED gate report), and it reproduces the jq/MSYS failure class already
  on record from earlier feature history.
- **Phase:** quality-gate (environment prerequisites)
- **Confidence:** Medium (single task this period; multi-evidence + prior
  history)
- **Do Not Overfit:** The gate scripts assume a POSIX toolchain; every
  Windows host that has not installed the shimmed jq will reproduce the exact
  misleading failure. The masking behavior (one failing gate hiding another
  finding) is structural, not incidental.

### FP-05: Gate-tooling contract drift — the documented workflow and the enforcing scripts disagree

Four latent gaps between what gate documentation/templates promise and what
the enforcing tools actually check were discovered while running this
feature's gates:

- **Evidence:**
  1. `deterministic-check-policy.md` documents a `required:false` +
     `waiver_reason` escape for brownfield placeholder false positives, but
     `check-contract.py`'s `_pass4_risk_tier` enforces `placeholder-scan` as
     `required:true` at every risk tier with no waiver path — "the documented
     escape does not actually work today" (`reports/quality-gate/T-008.md:41-47`).
  2. The canonical implementation-report template does not carry the
     `- Task ID: T-NNN` line the sdd-evaluator launch boundary requires;
     every one of this feature's 11 gate reports had to explicitly attest the
     field's presence (e.g. `T-001-sdd-domain.md:16-18` "…`- Task ID: T-001`
     field present"), i.e. each report was manually brought into the format
     the evaluator authorization check demands.
  3. Quality-gate report format drift: older gate reports lack the
     `Feature:` identity line that scoped, multi-feature metric collection
     (including this retrospective and T-009's drift roll-up) depends on;
     this feature's reports carry it, but nothing enforces it.
  4. The impl-review round>1 structural blocker: `validate-review-context-set`
     and the round-2 precheck contradict each other, so a `NEEDS_WORK` →
     round-2 continuation cannot be validated. Observed directly this period:
     impl-review attempt-2 round-2 and task-review attempt-1 round-3 both
     aborted after precheck with no contract produced; both loops recovered
     only by restarting a **fresh attempt at round 1** (impl-review reached
     PASS at attempt-3 round-1; task-review at attempt-2 round-2).
- **Frequency:** 4 independent gaps, 2 of them observed as live blockers this
  period
- **Phase:** quality-gate / review-loop tooling
- **Confidence:** High (multiple tasks and multiple independent artifacts)
- **Do Not Overfit:** Each gap is a divergence between a written contract and
  its mechanical enforcement — a class, not a coincidence. Left alone, every
  future feature pays the same tax (manual format compensation, unusable
  waivers, attempt restarts).
