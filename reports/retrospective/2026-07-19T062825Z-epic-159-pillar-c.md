### FP-02: PowerShell twin-authoring platform gotchas recurring beyond the already-tracked case-sensitivity/detection-suite classes (WFI-012)

- **Evidence:**
  - T-002: PowerShell's `$Host` is a read-only automatic variable — a
    naive `-Host` flag mirroring the `.sh` twin's `--host` throws "Cannot
    overwrite variable Host because it is read-only or constant" (verified
    directly); worked around with `-HostName`
    (`reports/implementation/epic-159-pillar-c/T-002.md:85-89`).
  - T-003: pwsh 7's default colorized/line-wrapped `ConciseView` error
    formatter "breaks any caller doing a plain substring match against the
    error text (including this suite's own subprocess-based assertions)."
    Fixed by wrapping the script's dispatch body in a top-level try/catch
    that prints `$_.Exception.Message` via `[Console]::Error.WriteLine()`
    and exits 1 (`reports/implementation/epic-159-pillar-c/T-003.md:105-113`).
  - T-003: native `-File` argv binding "does not reliably collect multiple
    bare tokens into an array parameter across a nested subprocess
    invocation" — an initial `[string[]]` array-parameter design broke
    under the test harness (reproduced directly: "the harness's
    `Invoke-Render` helper reproduced the exact `Count -ne 2` failure");
    fixed by switching to one scalar flag per value
    (`reports/implementation/epic-159-pillar-c/T-003.md:56-62,464-481`).
  - T-004: the ANSI/substring-assertion defect recurred a **second time**
    within the same feature — "a second, T-003-precedented fix was needed:
    the two new `Write-Error` call sites were replaced with
    `[Console]::Error.WriteLine()` + `exit 1`, because pwsh 7's default
    ConciseView terminating-error formatter... breaks any caller doing a
    plain substring match against the error text — exactly the defect
    T-003's report already documented and fixed the same way"
    (`reports/implementation/epic-159-pillar-c/T-004.md:96-101`).
- **Frequency:** 3 distinct gotcha classes across 3 of 6 tasks (T-002,
  T-003, and T-004's direct reuse of T-003's fix).
- **Phase:** implementation (PowerShell twin authoring). Every instance was
  self-caught by the implementer while building that task's own test
  harness — none reached quality-gate as a Critical/Major finding, and
  none is a case-sensitivity-class defect (WFI-012's existing scope).
- **Confidence:** High — each instance is directly evidenced with a
  verified reproduction (not merely suspected), and T-004's explicit
  "T-003-precedented" citation shows the ANSI/substring gotcha already
  recurring a second time within a single feature, prevented from becoming
  a fresh rediscovery only because the same session had direct access to
  T-003's own report.
- **Do Not Overfit:** zero downstream cost (self-caught during
  implementation; zero quality-gate cycles beyond cycle 1 attributable to
  any of these). `WFI-012` (Status: Draft, drafted in the
  epic-159-pillar-a2 retrospective, `Audit-Status: Not-Started`) already
  targets the adjacent PowerShell-twin-authoring-gap territory
  (case-sensitivity two-layer sweep + detection-suite self-reference) with
  the identical target file and Category/Mechanism
  (`app-dev-efficiency`/`instructions`, `AGENTS.md § Rules`). These 3 new
  gotchas are a natural broadening candidate for WFI-012's own Proposed
  Change rather than a competing new WFI — see Proposed Improvements.
