## Friction Patterns

### FP-01: An AC's own exhaustive-branch language was not fully mirrored by acceptance-tests.md's TEST-ID coverage — twice within one feature's spec-review loop

- **Evidence:**
  - **Round 1 (Major, `EDGE-CASE-COVERAGE`):** `requirements.md` AC-009
    explicitly claims the locking suite "locks the fetch-failure (AC-006),
    diff-detected (AC-007), no-diff, and dedup branches" — four named
    branches.
    `reports/spec-review/epic-159-pillar-d/attempt-1/round-1/reviewer-b.json`,
    `EDGE-CASE-COVERAGE`, Major, FAIL: "acceptance-tests.md's TEST-009 (the
    sole TEST-ID mapped to AC-009) describes only CI-resilience assertions
    ... it contains no assertion for the 'no-diff' branch ... No TEST-ID in
    the table anywhere exercises 'successful fetch, no divergence -> no
    issue filed.' Downstream failure mode: an implementation bug causing
    the script to file an issue unconditionally regardless of divergence
    ... would ship with all stated acceptance tests green." Remediated at
    round 2 by adding AC-020 (no-diff zero-invocation assertion), confirmed
    by round 2 reviewer-a's `AC-OBSERVABLE` finding: "New AC-020 specifies a
    concrete zero-invocation assertion."
  - **Round 2 (Major, `EDGE-CASE-COVERAGE`):** `requirements.md` AC-006
    requires fail-soft behavior "when either the Anthropic or the OpenAI
    fetch fails" (REQ-002: "on ANY fetch failure").
    `reports/spec-review/epic-159-pillar-d/attempt-1/round-2/reviewer-b.json`,
    `EDGE-CASE-COVERAGE`, Major, FAIL: "acceptance-tests.md TEST-006 scopes
    its fixture only to 'both vendor sources' failing together. No AC/TEST
    fixture exercises the asymmetric case (one vendor succeeds, the other
    fails) ... an implementation that only fail-softs on total dual-vendor
    failure would violate AC-006 with no test to catch it." Remediated at
    round 3 (commit `86cec1f`, "TEST-006 three-scenario fail-soft fixture"),
    confirmed by T-003's own implementation report's Test Evidence: "AC-006
    (3 fetch-failure scenarios x2 lanes = 6 scenario runs)."
- **Frequency:** 2 independent instances within this one feature's one
  spec-review loop, at 2 consecutive rounds (round 1, round 2), always the
  same check ID (`EDGE-CASE-COVERAGE`) and the same underlying mechanism:
  an AC/REQ statement names or implies an exhaustive set of
  branches/conditions ("locks ... four ... branches"; "on ANY fetch
  failure"), but the TEST-ID(s) acceptance-tests.md actually maps to that
  AC cover a narrower subset than the AC's own stated scope.
- **Phase:** spec-review (pre-implementation; caught before any task
  attempt began; zero downstream task-attempt or quality-gate cost).
- **Confidence:** High — both instances are reviewer-authored, directly
  quoted, the identical check ID, an identical branch-language-vs-fixture-
  scope mechanism, and both were remediated by adding the missing branch's
  own fixture/assertion (AC-020; TEST-006's 3-scenario split) rather than by
  narrowing the AC's own claim — confirming the gap was genuinely in
  acceptance-tests.md's coverage, not in the requirement's own wording.
- **Do Not Overfit:** both instances were fully caught and resolved inside
  spec-review itself, before implementation began, at zero task-attempt or
  quality-gate cost — this is not a review-quality failure; task-review and
  impl-review both later passed cleanly with no trace of either gap
  resurfacing. The genuine gap is structural: acceptance-tests.md's TEST-ID
  table is drafted by paraphrasing each AC's stated scope, and nothing in
  the authoring process cross-checks that every branch/condition an AC's own
  language enumerates or implies (a named list; "any"/"either") has a
  corresponding TEST-ID row — here it took two independent reviewer passes
  to converge. See Proposed Improvements (WFI-014).
