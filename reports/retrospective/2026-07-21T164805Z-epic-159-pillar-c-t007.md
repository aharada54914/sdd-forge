### FP-01: A test/verification suite's platform-SKIP branch, dormant since it landed, executed for real for the first time at release and failed on an environmental assumption the SKIP had never let anyone exercise — corroborated by an independent second instance in the immediately preceding feature

- **Evidence:** `tests/bump-version-gate.tests.sh`'s TEST-001 (AC-001) is
  the fixture that proves `scripts/bump-version.sh`'s real `CHANGELOG.md`
  mutation path works — it requires a leading `## Unreleased` heading to
  rewrite. On the implementing host, TEST-001 SKIPs itself with a named
  reason (`tests/bump-version-gate.tests.sh:144`: "this host's sed is
  BSD-style; `scripts/bump-version.sh`'s unedited mutation section ...
  requires GNU sed"), so this specific assumption had never been exercised
  for real, locally, since the gate suite landed (`306f2b2`, pillar-b
  #148, 2026-07-16T21:51:03+09:00). T-007 is the first — and, per
  `tasks.md`'s own text ("T-001..T-006 introduce no version-mutation path
  at all; T-007 is the ONLY task that executes a real
  `scripts/bump-version.sh` invocation"), the ONLY — task in this feature
  that runs `scripts/bump-version.sh` for real; the prior release
  (v1.10.0, `c5f1655`, 2026-07-09) predates the gate suite by a week, so
  T-007's v1.11.0 release is genuinely the first real `bump-version.sh`
  run since TEST-001 started existing. CI's GNU sed lane runs TEST-001 for
  real, and the release-branch CHANGELOG.md (carrying the v1.10.0-era
  convention of no standing heading) would have failed it. Fixed by commit
  `c59a247` ("chore(release): keep a standing Unreleased heading in
  CHANGELOG"), landed on the PR branch before the squash-merge (`eea908c`,
  PR #205); the fix's own commit message states the mechanism directly:
  "Undetected locally because BSD sed makes TEST-001 SKIP on macOS; CI's
  GNU sed lane runs it for real." Confirmed live: `CHANGELOG.md` line 3
  reads `## Unreleased` at HEAD, and
  `reports/quality-gate/20260721T153827Z-epic-159-pillar-c-T-007.md`'s
  AC-046/AC-050 evidence row records `git diff 23d8453..HEAD`: zero
  version-string mutation outside `bump-version.sh`, `CHANGELOG`
  `## Unreleased` unrenamed.
- **Frequency:** 1 instance in this period (T-007's own release), but a
  SECOND, independent, mechanically identical instance exists in the
  immediately preceding completed feature: `quality-loop-fixes` T-004
  (issue #179, commit `fe7dc81`) fixed a CRLF-contamination defect in
  `validate-review-context-set.sh`'s `jq -r` reads, which flipped
  `tests/lib/loop-driver.sh`'s `loop_validator_capability_probe` from
  "unhealthy" (silently gating `tests/loop-consistency.tests.sh`'s
  impl/task legs off on `windows-latest`, a status quo dating to before
  `#199`) to "healthy" for the first time — and those newly-reachable legs
  immediately hit "the same CRLF consumption class one layer deeper" on
  real Windows CI (commit `ac26cac`'s own message, issue #203), because
  nothing had ever exercised those specific legs' own Windows behavior
  while the probe was masking them. Both instances share the identical
  shape: a change alters the condition gating an existing suite's
  environment/platform-SKIPped branch, the newly-reachable branch executes
  for real for the first time, and it fails on an environmental assumption
  the gate's own historical silence had let go unverified — discovered
  only in the real target environment (Windows CI; GNU-sed release CI),
  never on the author's own local host. 2 independent instances, 2
  different features, 2 different suites, 2 different platform axes (OS;
  `sed` implementation) — the mechanism, not the surface, repeats.
- **Phase:** release / CI (this instance surfaced only in CI's GNU sed
  lane at release time, after the quality-gate itself had already passed
  T-007 — the quality-gate's own `check-*` scripts do not exercise
  `bump-version.sh`'s mutation path, only `tasks.md`'s own release-leg
  design does, at the actual release step); the `quality-loop-fixes`
  corroborating instance surfaced in CI's `windows-latest` lane, also
  post-quality-gate.
  Do-Not-Overfit note continues below (WFI drafted, see Proposed
  Improvements).
- **Confidence:** Medium — 2 corroborated instances meet this lineage's
  standing 2-instance recurrence bar (`workflow-retrospective/SKILL.md`'s
  "Medium for recurring patterns across two tasks"), but each individual
  feature contributes exactly 1 instance, and unlike every prior WFI in
  this lineage (WFI-009, WFI-011, WFI-013, WFI-014 were each drafted from
  2 instances found WITHIN one single feature), this is the FIRST WFI here
  drafted from 2 instances corroborated ACROSS two independent,
  already-completed features rather than within one — a genuinely
  cross-feature pattern, not merely a repeated observation inside one
  feature's own review loop. A third instance in a future feature would
  meaningfully raise confidence further.
- **Do Not Overfit:** the two instances are NOT the same bug re-appearing
  (one is Windows `jq` CRLF output; the other is BSD-vs-GNU `sed`
  behavior, in unrelated scripts) — merging them into "fix the CRLF bug
  better" or "fix the CHANGELOG heading better" would miss the actual
  recurring root cause, which is procedural: neither author's session had
  a checkpoint prompting "this test's SKIP/capability gate is about to
  open for the first time in a real environment — has anyone verified
  that environment's own assumption?" WFI-015 (drafted below) targets that
  procedural gap directly, at the generic mechanism level, not either
  script.

    TEST-044) as an honest, named-reason SKIP — the implementer judged it

    `effort_applied: "high"` (non-null, genuine, independently reproduced
    by the evaluator's own selector re-run) — AC-044 satisfied, gate

    `codex-sync` wrapper (`~/.zshrc:14`) that runs unrelated session-start/
    end hooks around every real `codex` invocation; a routine `codex

    in a SEPARATE, unrelated personal repository (`~/SKILL_SYNC`, branch
    `feature/skill-harness-engineering`) as a side effect of that
    wrapper's own hook — outside this task's scope and not authorized by
    this session (T-007.md, Unresolved Items). This is an operator-host
    environment hazard entirely outside this repository's own control
    surface (no project file causes or can prevent it); it is recorded
    here for visibility, not as a WFI candidate, since no `AGENTS.md`/

- **Confidence:** Low — single instance, first-of-its-kind (no

- **Do Not Overfit:** the account-vocabulary rejection was caught,
