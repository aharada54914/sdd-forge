### FP-01: Installed-plugin agent-definition schema drifts from the repo-side validator at independent-review launch boundaries

- **Evidence:** ci-mcp task-review attempt-2 round-1's first reviewer launch (identity ledger sequence 101/102) emitted the plugin-cache 1.7.0 agent-definition schema for `task-reviewer-a`/`task-reviewer-b`, which this repository's post-issue-86 validator rejects; outputs archived as `reports/task-review/ci-mcp/attempt-2/round-1/*.launch-1.json`. Reviewers were relaunched at sequence 103/104 with the repo's role-file canonical schema (`task-reviewer-a`: `stage`/`task-review`/`role reviewer-a`/`checks[].status`/flat manifest; `task-reviewer-b`: `manifest.allowed_inputs`/`checks[].result`; both with `findings[]`) quoted verbatim in the prompt, and passed clean (commit `8575df5`, which records in its own message: "Agent-definition vs validator drift is WFI-005 material"). This is the **second** occurrence of this exact drift class: the sibling knowledge-mcp feature's Phase-2 independent-evaluator launches hit the same installed-plugin-schema-vs-repo-validator contradiction in another repository.
- **Frequency:** 2 occurrences (this feature's task-review launch-1, and the sibling knowledge-mcp evaluator launches)
- **Phase:** task-review / independent-evaluator launch boundary (cross-plugin handoff — installed plugin cache vs repo-side deterministic validator)
- **Confidence:** Medium — within this period the drift touches exactly one review-loop invocation (task-review is feature-level, not per-task, so the "≥2 tasks" bar does not literally apply); confidence is raised to Medium rather than Low because the same failure mode is independently attested in a second repository's evidence and because the affected commit's own author explicitly attributes it to the same root cause as an already-drafted WFI.
- **Do Not Overfit:** This is not a one-off prompt mistake — it is the same defect class WFI-005 already documents (deterministic enforcement tools hardened without updating the artifact producers/consumers that must agree with them), just surfacing at a different launch boundary (task-reviewer role-file schema) than WFI-005's originally-documented gaps (implementation-report template, quality-gate report template, placeholder-scan waiver policy). Left unaddressed, every future reviewer/evaluator launch risks the same reject-and-relaunch tax.

**Disposition:** `docs/workflow-improvements/WFI-005.md` (`Status: Draft`, `Category: plugin-improvement`, `Meta-Change: true`, `Audit-Status: Human-Pending`, `GitHub-Issue: https://github.com/aharada54914/sdd-forge/issues/96`) already covers this defect class — its Root Cause Hypothesis is generic ("deterministic enforcement tools... hardened in separate changes without updating the artifact templates... template/validator drift is only discovered at gate time") and its own commit trail for this exact ci-mcp event states it is "WFI-005 material." Per the read-only mandate for this retrospective, **no new WFI is drafted for FP-01** — this occurrence is recorded here as corroborating Problem Evidence for WFI-005 and should be folded into WFI-005's audit revision (or its GitHub Issue #96) by a human-authorized follow-up, since WFI-005's current Proposed Change table targets report/template artifacts specifically and does not yet name the task-reviewer role-file schema as a fourth affected artifact class.

- **Evidence:** T-003 QG report: "`server.ts` は zero tools" claim was accurate at T-003's own commit but stale now that T-005/T-012/T-013 added 5 tools (`reports/quality-gate/2026-07-08T030000Z-ci-mcp-T-003.md`). T-012 QG report: report prose names `sizeInBytes`/`createdAt` while the shipped code/contract use `sizeBytes`+`createdAt` (`...T-012.md`).
- **Frequency:** 4 tasks this period (T-003, T-005, T-006, T-012); this is the **third consecutive feature** to show this exact class — local-env-mcp's retrospective recorded it as FP-002 (3 tasks: T-005, T-006, T-010) and proposed an un-drafted "WFI-005 (candidate)"; sdd-domain's retrospective recorded the same mechanism inside its broader FP-05 (5 tasks) and again proposed an un-drafted "WFI-005 (candidate)". Neither candidate was ever drafted — the WFI-005 number that was eventually drafted addresses a different theme (template/validator contract drift, see FP-01 above).
- **Phase:** implementation-report authoring (prose written before later same-feature tasks touch the described surface) / quality-gate evidence check
- **Confidence:** High (4 tasks this period alone, exceeding the 3-task threshold; plus 2 independent prior-feature evidence sets)
- **Do Not Overfit:** Every occurrence has the same mechanism — a report describes the state of a shared file *as of the authoring task's own commit*, and a later task in the same serial-dependency chain edits that file again. This is a process property of narrative reports coexisting with cumulative multi-task features, not any one author's error, and it has now reproduced on three separate features without ever having a WFI actually drafted for it despite being proposed twice.

**Disposition:** recorded as a candidate for a human-authorized WFI-006 in Proposed Improvements below; not drafted in this read-only retrospective (out of the orchestrator's explicit scope for this run, which authorized WFI-005 referencing for FP-01 only). Given the 3-occurrence history with two prior un-actioned proposals, a follow-up session should prioritize actually drafting and auditing this one.
