## Specification Differences

One design-time architectural finding, recorded here because it changed how
the test suite is structured relative to a literal (if under-specified)
reading of design.md's Test Strategy item 3 ("a PATH-prepended jq shim
script appends \r to every -r invocation's stdout") -- not a deviation from
an explicit design.md instruction, since design.md does not specify whether
the shim's corruption scope is global-per-run or site-scoped-per-assertion,
and both AC-022/023/024's own wording (each names a DIFFERENT, narrow site
group) is consistent with either reading:

A single shim that corrupts every `-r` call in one invocation cannot reach
TEST-023's ledger-batch site at all on the unfixed script, because the
manifest `stage`/`role` reads (lines 178-179, part of AC-022's own site
group) are consumed by the `case "$stage:$role"` authorization check
BEFORE the ledger-batch loop executes -- verified directly by running the
unfixed script under a naive global shim before authoring any assertion
(see Summary). This task's suite therefore shims exactly ONE targeted jq
program string per assertion (`CRLF_SHIM_QUERY`), isolating each of
AC-022/023/024's site groups precisely as their own text already implies,
and reserves the sentinel `CRLF_SHIM_QUERY=__ALL__` (corrupt everything at
once) for TEST-026's non-regression sweep only, where simultaneous
corruption of every site and continued fail-closed behavior is the exact
property being proven.

A second, smaller implementation-time decision: `tests/review-contract-foundation.tests.sh`
was chosen over a new `tests/validate-review-context-crlf.tests.sh` file
(AC-022's own task-time-decision note names both as options) specifically
because this feature's own Global Constraints forbid any of its 4 tasks
from adding a new entry to either `tests/run-all.sh` or
`tests/run-all.ps1`'s suite array, and `tests/review-contract-foundation.tests.sh`
is the one option already registered in both `tests/run-all.sh:16` and
`.github/workflows/test.yml:325` (INV-025) -- a brand-new file would either
need exactly the forbidden registration edit or would silently never run in
CI.
