## Specification Differences

- **Scope adjudication: driver dispatch extension is T-003 scope, not a
  Planned Files violation.** tasks.md T-003's Planned Files list does not
  name `tests/lib/loop-driver.sh`/`.ps1`, but T-002's Out of Scope note
  explicitly assigns "Driving impl/task/domain loops or the escalation
  chain (T-003, T-004)" to this task and T-004, and T-003's own Goal
  ("driving spec, impl, task, and domain review loops through rounds 1->3
  ... via the loop driver") is structurally unachievable without extending
  `drive_review_round`'s dispatcher (T-002's own implementation report
  flagged this exact seam as a forward-compatibility risk requiring a human
  decision at T-003 planning time). The orchestrator adjudicated this in
  the task prompt: the extension is authorized, strictly scoped to adding
  the three REVIEW-STAGE dispatch functions (impl/task/domain), explicitly
  excluding the escalation chain (quality-gate/terminal-tier remain T-004
  scope, untouched), and conditioned on T-002's smoke suite staying green
  (verified: `tests/loop-driver.tests.sh` 22/0 and `tests/loop-driver.tests.ps1`
  15/0, unchanged from T-002's own reported counts).
- **Fixture synthesis extension (additive only).** Driving impl/task rounds
  against the REAL `impl-review-precheck.sh`/`task-review-precheck.sh`
  requires `design.md`, four layer specs (the fixture's registry entry is
  unconditionally `profile: "full"`, set by T-002's `loop_fixture_init`),
  and (for task) `traceability.md` plus a REQ-001 id in `requirements.md`
  (`validate-layer-traceability.py`/`.ps1` requires at least one `REQ-\d{3}`
  match). These are synthesized unconditionally in `loop_fixture_init`,
  additively -- no existing T-002 fixture file or behavior is removed or
  altered, and T-002's own suite (which only ever reads `requirements.md`/
  `acceptance-tests.md`) is unaffected. `tasks.md` is the one exception:
  synthesizing it unconditionally would make `check-workflow-state.sh`'s
  task-lifecycle gate require BOTH `Spec-Review-Status` and
  `Impl-Review-Status` to already read `Passed` (a hard, unconditional
  requirement the moment `tasks.md` exists at all -- see
  `check-workflow-state.sh` around its task-lifecycle checks), which would
  break a fixture mid-drive of the impl leg itself (where only Spec is
  Passed). `tasks.md` is therefore synthesized lazily, by
  `_loop_task_fixture_prepare`/`Initialize-LoopTaskFixture`, called only
  after both upstream statuses are genuinely `Passed`.
- **A `domain/` tree is synthesized unconditionally per fixture.**
  domain-review is not feature-scoped (its precheck takes `<attempt>
  <round>`, no feature argument, and reads a single repo-root-relative
  `domain/` tree), so `_loop_fixture_init_domain`/`Initialize-
  LoopFixtureDomain` builds one canonical `domain/` tree (six Markdown
  files, `domain-contract.json`, one aggregate card) per fixture,
  unconditionally, mirroring how `specs/<feature>/` is already synthesized
  unconditionally for every profile.
- **`assert_bidirectional_invariant` is a genuine read-only re-validation,
  not a re-derivation.** `_loop_reserve_review_context` (T-002, unchanged
  4-arg signature/behavior) and the new `assert_bidirectional_invariant`
  now share `_loop_review_context_call`, which composes the identical
  `review-context-invocation/v2` envelope and always calls the REAL
  `validate-review-context-set.sh`/`.ps1`; only the `--reserve` flag
  differs (present for driving a round -- extends the identity-ledger
  chain; absent for the invariant check -- read-only, safe to call any
  number of times, never advances ledger state). This satisfies
  security-spec.md B1/B2 (real binaries driven read-only) for the new
  assertion the same way the existing driving path already does.
- **TEST-010 is deliberately decoupled from TEST-008's driven rounds.**
  Early implementation used TEST-008's own genuinely-driven round
  directories as TEST-010's fixture. Discovered while porting to PowerShell
  (see next item): this coupling would make TEST-010 inherit TEST-008's
  degradation on any lane where a precheck script is missing, even though
  `assert_bidirectional_invariant` only exercises `validate-review-context-
  set.sh`/`.ps1` (a script that IS present on every lane) and does not need
  a genuinely-driven round to prove the invariant -- it only needs a real,
  correctly-hashed file at the exact `cross_gates`-authorized path (the
  same principle T-002's own TEST-007 already established with a synthetic,
  non-driven `SCHEMA_DIR`). TEST-010 was redesigned to build its own
  self-contained fixture with hand-placed (not genuinely precheck-produced)
  round-result placeholder files, so it now runs for real, independent of
  TEST-008, on both lanes. This is a strictly stronger design than what the
  task's Scope section anticipated (reusing TEST-008's driven manifests);
  documented here as a deviation from the most literal reading of that
  Scope sentence, not as a scope reduction -- the assertion coverage (four
  positive stage:role checks plus the negative self-check) is identical or
  greater.
- **Pwsh spec/impl/task/domain leg degradation (AC-015 recorded SKIP).**
  `spec-review-precheck.ps1` does not exist upstream (T-002's own
  documented finding). Discovered during this task: `impl-review-
  precheck.ps1` and `task-review-precheck.ps1` DO exist upstream (verified,
  this checkout), and driving them in isolation succeeds -- but both
  scripts' own first, unconditional precondition is a genuine on-disk
  spec-review PASS (`impl-review-precheck.ps1` calls `Require-Pass` for
  stage "spec" before anything else; `task-review-precheck.ps1` requires
  both spec and impl), which can only be produced by driving
  `spec-review-precheck.ps1` itself. The missing spec twin therefore
  transitively blocks the impl and task legs too, even though their own
  precheck scripts are present. `domain-review-precheck.ps1` is
  independently absent (#147; domain-review has no dependency on the spec
  twin). Both gaps are named, reasoned SKIPs in
  `tests/loop-consistency.tests.ps1`; each self-heals independently the
  moment its respective upstream twin lands (SKIP conditions check file
  presence at runtime, not a hardcoded flag). AC-006's bash-lane-carries-
  the-proof precedent from T-002 continues to apply, extended to the three
  new legs.
- **Impl/task round-3 severity simplified to a clean PASS (`none`), not a
  Minor-only merge.** requirements.md/REQ-003 names the "spec round-3
  Minor-only merge to PASS" nuance specifically for spec; the inventory's
  `terminal.state` for impl-review and task-review is the literal string
  `"PASS"` (not `"PASS-with-warnings"`), so driving round 3 with severity
  `none` (zero findings) reaches that literal terminal state directly and
  is simpler to verify than replicating spec's warningCount/Minor-only
  merge-rule nuance for two more stages with no requirement text asking for
  it. The domain leg's round 3 uses `Major` severity (not `none`), because
  the inventory's domain `terminal.condition` text ("round 3 -> BLOCKED
  (round-cap behavior only)") states no Minor-only-PASS exception, matching
  `design.md`'s account of domain review having no such carve-out.
- **check-risk.sh/.ps1 asymmetry across lanes, both accepted as HEAD-observed
  behavior.** `task-review-precheck.sh` treats a missing `check-risk.sh` as
  a WARNING + SKIP (does not fail); `task-review-precheck.ps1` treats a
  missing `check-risk.ps1` as a hard failure (`Fail 'shared risk gate is
  missing'`). `check-risk.ps1` is therefore symlinked into the pwsh
  fixture (needed for the task leg to ever succeed there in principle);
  `check-risk.sh` is left unsymlinked for bash (harmless, already proven

  `Risk Rationale: ...` on both lanes (harmless on bash, required on pwsh

  identical even though only one lane currently drives task rounds for
  real.
