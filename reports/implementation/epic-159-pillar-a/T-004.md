  own permit leg), an unchanged-task-contract rejection, a forged
  `blocked_task_contract_sha256` rejection, a fractional-second timestamp
  rejection, a parent-directory symlink-escape rejection, and a forged
  diagnosis-hash rejection. That suite predates this task (visible at HEAD
  before any T-004 change) and was not written by, or attributed to, this
  epic. This is a specification defect, not an implementation one: the
  spec's premise was wrong at authoring time, not invalidated by this
  task's own work. Per `plugins/sdd-implementation/skills/implement-task/
  SKILL.md` (spec is hash-frozen; gaps are returned to
  `sdd-bootstrap-interviewer`, never silently resolved by guessing), this
  report records the correction rather than editing requirements.md/
  design.md, and this task's own suite comments were updated to state the

  `tests/loop-escalation.tests.sh`'s TEST-011 header comment). **What this

  `tests/agent-model-routing.tests.sh` fixture for this script starts from
  a tasks.md section that ALREADY carries `Approval: Approved` /
  `Terminal Reapproval: ...` and mutates the EVIDENCE JSON to probe

  where the evidence JSON is well-formed but tasks.md itself was never
  actually reapproved (missing those two lines entirely) -- the specific
  "no human approval record persisted" deny case AC-011 describes. TEST-

  quality-stage manifest builder used instead.** This task's Planned Files
  list does not include `tests/lib/loop-driver.sh`/`.ps1`, and the shared

  green at their T-002/T-003 counts, proving non-destructiveness.
- **`select-agent-model.sh` is committed without the executable bit** (mode
  100644; discovered via the RED run -- `Permission denied`, rc=126). The
  repository's own `.github/workflows/test.yml` already documents this class
  of issue ("Invoked via bash explicitly so the step does not depend on the

  consistency, every other driven `.sh` script) via `bash "$SCRIPT" ...`

- **`select-agent-model.sh`'s successful (non-BLOCKED) path performs no
  native/external command and never calls an explicit `exit N`.** On bash
  this is harmless (the underlying `python3` subprocess's own exit code
  becomes the observable `$?`). On the pwsh twin, which is a pure PowerShell
  reimplementation with zero external command invocations in that path,
  `$LASTEXITCODE` after `& $selectModelPs1 ...` is simply whatever a prior
  native command last set it to -- not a meaningful signal. This matches the
  existing `tests/agent-model-routing.tests.ps1` convention, which likewise
  never checks `$LASTEXITCODE` for this script's successful/JSON-emitting
  call shape. TEST-011.5/.6 on the pwsh lane therefore validate the
  escalation decision by JSON content only, consistent with that precedent
  (discovered via the RED run showing content-correct output alongside a
  stale, misleading `rc=1`).
- **`check-terminal-tier-resume.ps1` has no python3 (or any external
  deterministic-runtime) dependency at all** -- confirmed by a full read and
  a `grep` for "python3" over the file (zero matches). It is a pure
  PowerShell reimplementation of `check-terminal-tier-resume.sh`'s contract,
  including its own hashing (`[Security.Cryptography.SHA256]`) and JSON
  parsing (`ConvertFrom-Json`). INV-017 (investigation.md) cites only
  `check-terminal-tier-resume.sh:29-32` for this degradation, never the

  precise, not an oversight. TEST-013.1 on the pwsh lane is therefore an
  unconditional, reasoned SKIP explaining the finding rather than a
  fabricated restricted-PATH scenario the script cannot actually exhibit.
- **`validate-review-context-set.ps1` reports failures via
  `[Console]::Error.WriteLine`, which bypasses PowerShell's own stream
  redirection (`2>&1`) when the script is invoked in-process (`& $script
  ...` within the same pwsh session).** Discovered when TEST-012.4/.5's
  negative self-checks initially printed the expected
  `REVIEW_CONTEXT_PATH: ...` message directly to the console instead of
  populating the captured `$out` variable. Fix: those two calls invoke the
  validator as a genuine child process (`& pwsh -NoProfile -File
  $validatorPs1 ...`), which makes stdout/stderr real OS-level streams that
  `2>&1` captures reliably; the always-`exit 0`-on-success positive check
  (TEST-012.3) does not need this and still uses the in-process `&` form
  (its `$LASTEXITCODE` check already worked correctly in-process, since
  `exit N` -- unlike `[Console]::Error.WriteLine` -- does propagate through
  `&` invocation).
- **TEST-018.2's PowerShell mutation is a plain literal `.Replace()`, not a
  `-replace`/regex-based substitution.** The first implementation attempt
  used `[regex]::Escape(...)` on both the search AND replacement arguments
  to a `-replace` call, producing an invalid/empty effective pattern (RED
  evidence: the mutated copy crashed with "Cannot bind argument to parameter
  'Pattern' because it is null"). Fixed by using .NET's plain
  `[string]::Replace(oldLiteral, newLiteral)`, which performs pure literal
  substring substitution with no regex-metacharacter or `$`-backreference
  interpretation, exactly matching the intent (replace the source snippet
  `"\b" + [regex]::Escape($TaskId) + "\b"` with `[regex]::Escape($TaskId)`,
  removing the word-boundary anchors).
- **AC-011's `select-agent-model.sh` schema-validation coverage stops at a
  structural jq/PowerShell replicated check against
  `contracts/terminal-tier-blocked-state.schema.json`, without an additional
  negative self-check for that specific assertion.** design.md's Test
