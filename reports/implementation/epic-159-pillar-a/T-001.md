## Specification Differences

- **Cap-source finding (impl-review / task-review)** — investigation.md
  INV-001/INV-003 and the requirements assumed six script-enforced loops
  whose caps are grep-able from driver sources. The mandated implementation-
  time grep measurement (design.md / INV-003: "cap values taken from the
  driver sources") found that **neither
  `plugins/sdd-review-loop/scripts/impl-review-precheck.sh` nor
  `task-review-precheck.sh` (nor their `.ps1` twins) contains a numeric
  round ceiling** — both validate only `^[1-9][0-9]*$` (impl line 230, task
  line 243). The round<=3 policy for those two loops lives in skill text
  (`impl-review-loop/SKILL.md`: "Runs up to three rounds", round "<N> of 3";
  `task-review-loop/SKILL.md` likewise). Only `spec-review-precheck.sh:35`,
  `domain-review-precheck.sh:39` (`round -le 3`) and
  `check-quality-gate-cycle-limit.sh:54` (`count -ge 3`) carry grep-able
  numeric caps. Encoding impl/task as `cap_source: script` +
  `cap_kind: numeric` would have forced a fabricated grep target (a false
  green of exactly the kind REQ-001 exists to prevent), so the inventory
  registers impl-review and task-review with
  `cap_source: skill-instruction` while keeping their real precheck scripts
  in `driver_scripts` (the INV-002 registration requirement is independent
  of cap_source; TEST-001 still forces both prechecks to be registered).
  This is a faithful application of the requirements.md Field Definitions
  ("`script` means the cap is enforced by a repository script"; cap_kind
  "required for `cap_source: script` entries, absent for
  `skill-instruction`"). AC-002's enumerated grep targets ("round <= 3
  guards, `count >= 3`") are exactly the three that exist. ADR-0010's
  Decision 3 was amended (minimal edit) to record this finding.
- **terminal-tier cap_kind** — terminal-tier keeps
  `cap_source: script` + `cap_kind: state` per the requirements (its
  enforcement script `check-terminal-tier-resume.sh` is real; the cap is
  qualitative), with `cap.value: null` per the design schema.
- **spec-review terminal.state** — the inventory records `PASS` (round-3
  Minor-only merges to PASS; the BLOCKED alternative is in the condition
  text). design.md's example JSON shows `BLOCKED` for the same condition
  string; requirements.md Field Definitions make `assert_terminal` compare
  `state` only and A3 (T-003) is the consumer — if T-003's legs need the
  cap-reached-BLOCKED encoding instead, flipping this value is a one-line
  reviewed inventory diff. Recorded here so the T-003 implementer decides
  with the driver in hand.
- **domain-review terminal** — `BLOCKED` (round 3 → BLOCKED) per the
  requirements.md "Domain-review terminal disambiguation" note;
  post-approval drift detection is NOT encoded.
