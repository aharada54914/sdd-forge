## Specification Differences

- **`spec-review-precheck.ps1` does not exist (pwsh TEST-006 degradation)**
  — design.md's Components table lists the real gate surfaces as
  `{spec,impl,task,domain}-review-precheck.sh` "+ existing `.ps1` twins
  where present". Implementation-time measurement: `impl-review-precheck.ps1`
  and `task-review-precheck.ps1` exist; `spec-review-precheck.ps1` and
  `domain-review-precheck.ps1` do NOT (repo-wide find, this checkout).
  The specs pre-authorize an explicit pwsh degradation for the domain gap
  (#147) but are silent on the spec gap, and T-002's smoke suite drives
  exactly spec-review. Following the same AC-015 recorded-SKIP convention,
  `tests/loop-driver.tests.ps1` emits a named SKIP for TEST-006's seven
  driving checks explaining the missing upstream twin, while TEST-005/007/017 run
  fully on both lanes (the `.ps1` library's fixture init, ledger synthesis,
  and assertions are exercised against the REAL
  `validate-review-context-set.ps1` semantics via the inventory and hash
  formula). AC-006's cross-lane proof is therefore carried by the bash lane
  until a spec-review-precheck.ps1 twin lands upstream (out of T-002 scope;
  authoring it would be an unrequested new gate surface). Flagged for the
  quality gate and the humans as the principal deviation of this task.
- **`drive_review_round` stage coverage** — the design contract names
  `drive_review_round <stage> ...` generically; T-002's Scope/Done When and
  Out of Scope confine proven behavior to spec-review rounds 1->3
  (impl/task/domain driving is T-003/#143). Shipping unverified stage code
  would be a false green of the kind REQ-002 exists to prevent, so the
  dispatcher fully implements `spec` and REFUSES `impl|task|domain` with an
  explicit error naming the owning task. Forward risk recorded for T-003:
  its Planned Files list does NOT include `tests/lib/loop-driver.*`, so
  extending the dispatcher will require either a tasks.md Planned Files
  amendment (human-reviewed) or acceptance of driver edits under T-003's
  scope; the seam is isolated (per-stage `_loop_drive_<stage>_round`
  functions) to keep that extension additive.
- **`assert_prior_round_complete` naming** — the tasks.md Goal sentence
  describes "manifest composed exclusively from the previous round's
  actually-emitted outputs". In the real spec-review protocol the
  reviewer-launch manifest lists the CURRENT round's precheck output plus
  the spec inputs (the REAL validator rejects prior-round artifacts for
  spec reviewers), while the previous round's outputs gate the precheck's
  own round>1 transition. The driver therefore implements the requirement
  as: (a) every manifest entry is hashed from a file proven to exist on
  disk at compose time (`_loop_manifest_entry` fails on absence—the
  "nonexistent artifact fails" leg), and (b) round-N refuses to start
  unless the round-(N-1) output set is complete on disk
  (`assert_prior_round_complete`, TEST-006.6/.7). No hand-written hash or
  path enters any manifest.
- **Genesis ledger stage/role vocabulary** — INV-005/INV-006 fix the record
  structure and hash formula but not the genesis record's field values; the
  driver uses obviously-synthetic constants
  (`genesis|loop-driver-fixture|fixture-genesis-run|fixture-genesis-session`),
  consistent with security-spec.md Data Classification ("obviously-synthetic
  constant values") and accepted by the REAL validator's ledger grammar.
