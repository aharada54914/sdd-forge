## Summary

Implemented the three local-env-mcp MCP tools and the server/entrypoint,
conforming to `contracts/local-env-mcp-tools.v1.schema.json`, following the
mandatory TDD order (report skeleton + WFI-001 preflight, then failing tests,
then implementation to green).

- `src/tools/env.ts` — `getOsInfo()` (os/process facts only; never emits
  hostname/username/home directory/env values), `getToolchainVersions(input)`
  (strict zod `names` enum over the 14-name allowlist; invalid input →
  `invalid-input` envelope before any probe; delegates probing to the existing
  probe-engine; whole response stays `ok:true` with per-entry
  available/version/probeError), and `listAvailableClis()` (availability-only
  over the full allowlist, sharing the probe-engine TTL cache). Exposes the
  strict input schema + raw shape for introspection.
- `src/server.ts` — builds `new McpServer({ name: "local-env-mcp",
  version: "0.1.0" })`, registers exactly the 3 tools, and returns each
  `Result` envelope as MCP text content (`content[0].text`); envelope errors
  are ok MCP text, never MCP protocol errors (sdd-forge-mcp convention). Only
  `get_toolchain_versions` has an input schema, and it advertises only the
  `names` enum filter.
- `src/index.ts` — stdio transport startup with NO probing at startup; the only
  startup diagnostic is a single trivial JSON line on stderr (fixed server
  name/version/ready flag), carrying no env value, username, hostname, or home
  path. Full redaction logger remains T-003's scope.

The probe-engine, allowlist, and envelope (T-001) were used unchanged. No new
npm dependency was added; contract conformance is checked with a hand-rolled
structural validator reading the committed schema JSON (ajv is not available
and was not added).

## Working Notes

- MCP SDK 1.29.0 `registerTool` validates a strict zod input schema at the
  boundary and THROWS an McpError on failure (mcp.js `validateToolInput`,
  lines ~166-179). To keep envelope-error semantics (invalid input →
  `invalid-input` envelope as ok text, never a protocol error), the tool handler
  validates `names` internally with a strict zod object and returns the envelope;
  the registered raw shape advertises only the `names` enum for `tools/list`.
  Files examined: `node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js`.
- zod 4.4.3 (`z.enum`, `.strict()`, `z.toJSONSchema`) verified to reject
  out-of-enum members, extra properties, and non-`names` fields, and to emit
  `additionalProperties:false` — the basis for the AC-003 introspection test.
- Determinism for AC-002: reused T-001's probe-engine `pathOverride` affordance
  (fake `node`/`git` shims in a temp dir) plus `__clearProbeCache` so results do
  not depend on the host's real toolchain and do not bleed across tests.
