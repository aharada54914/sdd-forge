## Summary

Implemented `src/diagnostics.ts` (a small redaction-safe stderr logger) and
wired it into `src/index.ts`, following the mandatory TDD order (report skeleton
+ WFI-001 preflight, then failing no-secrets/diagnostics tests, then
implementation to green).

- `src/diagnostics.ts` — `logStartup(info)` writes ONE single-line JSON object
  with a fixed field allowlist `{event:"startup", name, version, transport}`;
  `logFatal(err)` writes ONE single-line JSON object `{event:"fatal", code,
  message}` where `message` passes through a conservative scrubber. The scrubber
  replaces verbatim occurrences of `os.homedir()`, `os.userInfo().username`,
  `os.hostname()`, and any `process.env` VALUE longer than 3 chars with a fixed
  `[redacted]` token. The module NEVER serializes `process.env`, `os.userInfo()`,
  `os.homedir()`, or `os.hostname()` into output; only caller-supplied
  allowlisted primitives (and a scrubbed message string) are emitted.
- `src/index.ts` — the previous ad-hoc startup stderr line was replaced with
  `logStartup({ name, version, transport: "stdio" })`; the fatal `main().catch`
  handler now routes through `logFatal`. Startup performs no probing and stdout
  stays reserved for the MCP JSON-RPC transport (behavior otherwise identical).

No new npm dependency was added. The no-secrets suite compiles the sources with
the existing `tsconfig.test.json` and spawns the compiled entrypoint
(`dist-test/src/index.js`) under a canary env, drives one `tools/list` and one
`tools/call` over stdio JSON-RPC, and grep-asserts absence of secrets from both
stdout and stderr.

## Working Notes

- MCP stdio framing verified before writing the integration test: the SDK's
  `serializeMessage` returns `JSON.stringify(message) + "\n"` and `readMessage`
  reads up to `indexOf("\n")`
  (`node_modules/@modelcontextprotocol/sdk/dist/esm/shared/stdio.js`). So the
  canary test writes newline-delimited JSON-RPC on stdin and reads
  newline-delimited responses on stdout — no Content-Length framing, no client
  library needed.
- Red→Green: the red capture shows `tsc -p tsconfig.test.json` failing with
  `TS2307: Cannot find module '../../src/diagnostics.js'` (exit 2), which blocks
  the whole suite (the canary test also requires the compiled
  `dist-test/src/index.js`). After adding `src/diagnostics.ts` and wiring
  `index.ts`, all 48 tests pass.
- One green-phase fix on the TEST side only: the initial sanity assertion looked
  for the literal `"os-info"` (with quotes), but the os-info payload rides inside
  the MCP `content[0].text` field as escaped JSON, so on the wire the marker is
  `\"os-info\"`. The assertion was relaxed to match the `os-info` substring
  without assuming quote framing. src/ was not changed for this — a manual driver
  run confirmed startup stderr and every tool response were already leak-free.
- Scrubber limits (documented in `src/diagnostics.ts`): `scrubMessage`
  removes only VERBATIM substrings of the known-secret set (homedir, username,
  hostname, and every `process.env` value of length > 3) present at scrub time.
  It does not catch transformed/encoded/partial secrets or values set after the
  scrub, and it never inspects arbitrary error properties (stack/cause) — those
  are dropped entirely, not scrubbed. The fixed field allowlist (not the
  scrubber) is the primary non-leakage control; the scrubber is defense in depth
  for `logFatal`'s single free-form field. `logStartup` has no free-form field.
- Files examined: snapshot specs (design.md, security-spec.md,
  acceptance-tests.md), current `src/index.ts` / `src/server.ts` /
  `src/probe-engine.ts` / `src/allowlist.ts` / `src/tools/env.ts` /
  `src/envelope.ts`, the existing `tests/readonly/static-check.test.ts` (to stay
  within its no-exec/no-fs-write/no-placeholder denylist), and the T-002 report
  for the validator-passing report format.
