## Amendment (gate cycle 1 Critical fix)

An independent quality-gate evaluator found one Critical: line 146's
feature-slug check used PowerShell's default case-INSENSITIVE `-notmatch`,
so `^[a-z0-9][a-z0-9-]*$` also accepted uppercase-containing slugs
(`Test-Feature`, `NOFEATURE`, `Epic-159-Pillar-A2`) that the `.sh` original's
always-case-sensitive bash `=~` rejects -- a weakened-port gap that could
route evidence to a differently-cased path on a case-sensitive filesystem,
breaking the replay-forbidden guarantee (security-spec.md B2). Fixed with a
single-token change (`-notmatch` -> `-cnotmatch`), then swept the entire
file for the same vulnerability class: one more genuine divergence in the
literally-requested `-match`/`-notmatch`/`-replace` operator set (line 447,
the `--reset` previous-round directory scan) plus ten more in an extended
sweep of `-eq`/`-ne`/`-contains`/`-notcontains` string comparisons against
JSON-sourced evidence fields (all likewise case-insensitive by PowerShell
default, all narrowed to their `-c` case-sensitive variant). T-003's
`domain-review-precheck.ps1` was diagnosed (not edited, per the amendment's
scope) and found to have the same class of gap at two sites (its own status
regex and `--reset` directory scan); reported for the host to route
separately. Full sweep table, byte-for-byte reject-path comparisons, and
regression evidence are in
`specs/epic-159-pillar-a2/verification/T-004/fix-slug-case.log`.

## Amendment (gate cycle 2 Major fix)

A second independent quality-gate evaluator found one Major: cycle 1's
sweep was OPERATOR-LEVEL only and missed CMDLET-level case-insensitive
defaults. Line 187's `Select-String -Pattern '^Spec-Review-Status:\s*(.*)$'`
used PowerShell's default case-insensitive matching, so it also matched a
`SPEC-REVIEW-STATUS:`/`spec-review-status:` header the `.sh` original's
always-case-sensitive `sed` never matches. Fixed by adding `-CaseSensitive`
to that call. Followed with a full cmdlet/language-feature-level sweep (as
distinct from the operator sweep already done) covering `Select-String`,
`Get-ChildItem -Filter`/`-Include`/`-Exclude`, `-like`/`-split`,
`[regex]::Match`/`IsMatch`/`Replace`, raw String instance methods,
`Sort-Object`, and `switch`: one genuine fix (`Select-String`, above);
`[regex]::Replace`'s existing use was verified empirically (not just
assumed) to already be case-sensitive by .NET default, matching `sed`;
`Sort-Object`'s 9 call sites were reviewed individually and found safe
because every site's actual accept/reject decision is enforced by a
separate ordinal comparison after sorting, not by sort order itself; and
`switch ($Role)`'s confirmed case-insensitivity is not exploitable since
`$Role` is always an internal literal, never JSON-sourced. Full mechanism
table and mis-cased-header parity verification are in
`specs/epic-159-pillar-a2/verification/T-004/fix-header-case.log`;
`fix-slug-case.log` carries a correction note pointing to it (its own
sweep was accurate for what it covered, but did not claim to be
cmdlet-complete).

## Working Notes

- The initial `Test-ValidateContract` port failed round 2's precheck call
  with `prior round contract is malformed or does not require work`. Because
  the .sh original's `validate_contract()` has ~15 distinct jq-based
  assertions with no single obvious failure point, debugging via ad hoc
  `pwsh -Command` probes against the real fixture (mktemp-scoped, driven
  through `tests/lib/loop-driver.ps1`'s own `Initialize-LoopFixture`/
  `Invoke-DriveReviewRound` functions -- never a real repository path) was
  insufficient on its own. Built an instrumented copy of the extracted
  function-definitions block (never the real `*-precheck.*` file itself,
  which the protected-basename write-verb/redirect rule forbids combining
  with Bash pipes/redirects) with a `Write-Host` inside every `{ return
  $false }` branch, confirmed via the PowerShell AST parser that the
  instrumentation preserved valid syntax, then ran it against the real
  round-1 contract/precheck fixture files with the exact parameters round-2's
  call site uses. First instrumentation attempt printed the marker text
  UNCONDITIONALLY (a Python string-insertion bug placed the diagnostic
  BEFORE the `if` statement rather than inside its brace block), producing a
  misleading trace; corrected to insert strictly inside the `{ ... }` block,
  after which the trace pinpointed the exact failing line:
  `if ($summary.generated_at -isnot [string]) { return $false }`. Confirmed
  via `$summary.generated_at.GetType().FullName` that PowerShell 7's
  `ConvertFrom-Json` had silently parsed the ISO-8601 `generated_at` value as
