## Amendment (gate cycle 2 fix)

A quality-gate finding on the sibling T-004 (`spec-review-precheck.ps1`,
fixed in 2a7eef5) identified a "weakened port via PowerShell's default
case-insensitive comparison operators" defect class (security-spec.md B2):
bash/jq are always case-sensitive, but PowerShell's `-match`/`-eq`/etc. are
case-insensitive unless the `-c` variant is used. The same diagnosis was
applied to this task's `domain-review-precheck.ps1` and confirmed correct
by an independent full-file sweep: 2 genuine sites (`:137` the
`Domain-Model-Status` value check, `:255` the `--reset` previous-round
directory scan) silently accepted wrong-case input the `.sh` original
rejects (e.g. `pending`/`APPROVED`, `ROUND-1`), fixed to `-cnotmatch`/
`-cmatch` respectively, plus one adjacent site (`:256`) hardened to
`-creplace` for defense-in-depth though provably unreachable with
mismatched case post-fix. Full sweep table, root-cause counterfactuals, and
end-to-end `.sh`-vs-`.ps1` parity verification (named parameters and the
production array-splat calling convention, for both the status check and
the `--reset` directory scan) are recorded in
`specs/epic-159-pillar-a2/verification/T-003/fix-case-sensitivity.log`; all

### Root cause of the TEST-008.12 FAIL (full diagnostic in `green-ps1.log`)

Isolated, step-by-step reproduction against the unmodified
`tests/lib/loop-driver.ps1` proved: (1) `domain-review-precheck.ps1` itself
succeeds cleanly for round 1, producing a well-formed `precheck-result.json`;
(2) `Publish-LoopDomainRoundA` succeeds, producing valid
`integrated-summary.json`/`reviewer-a.json`; (3) inside
`Publish-LoopDomainRoundBContract`, the `integrated-verdict.json` and
`reviewer-b.json` jq calls both succeed, but the final
`domain-review-contract.json` jq call FAILs with `invalid JSON text passed to
--argjson`. Root cause: `$manifestAJson = Invoke-LoopJq @("-r",
".allowed_input_manifest") reviewer-a.json` returns jq's PRETTY-PRINTED
(multi-line) array text, which PowerShell's `&` external-process capture
turns into a STRING ARRAY (one PowerShell string per output line) rather
than a single string; re-interpolating that array variable as `--argjson
manifest_a $manifestAJson` into a subsequent `& jq ...` call causes
PowerShell to expand the array into SEPARATE command-line arguments to jq,
so jq receives only the first line (`"["`) as the `--argjson` value instead
of the full JSON text. This pattern is unique to
`Publish-LoopDomainRoundBContract`: the spec/impl/task equivalents build
each reviewer's `allowed_input_manifest` inline within the jq filter from
scalar `--arg` values and never re-read a previously-emitted array back
through PowerShell, so they are not exposed to this bug. Because
`domain-review-precheck.ps1` never existed before this task, this code path
had never executed and the defect had no way to surface until now.
Independent confirmation that the defect is unrelated to this task's file:
`reviewer-a.json`/`reviewer-b.json` are generated entirely from
loop-driver-computed hashes and fixed paths, none of which depend on
`domain-review-precheck.ps1`'s specific output fields; the failure is
deterministic and would occur for any conforming domain-review precheck
implementation, since `allowed_input_manifest` always has >= 3 entries for
domain review (forcing jq's multi-line pretty-printed output).

Secondary, unrelated observation: `Test-LoopTerminal` (`tests/lib/loop-driver.ps1`)
only compares the STATIC `loop-inventory.json` `.terminal.state` value
against the caller-supplied `-Observed` argument -- it never reads back any
state the fixture actually produced -- which is why TEST-008.13 reports `ok`
even though TEST-008.12 (the actual round-drive) FAILed. This is a
pre-existing, wave-1-era weak assertion, not introduced by this task, and is
noted for completeness only.
