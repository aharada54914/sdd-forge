## Summary

1. **Permission minimization (AC-010).** Verified against the pinned
   `anthropics/claude-code-action@558b1d6cab4085c7753fe402c10bef0fbb92ac7a`
   (v1.0.165) `action.yml` that `id-token: write` is required ONLY for
   workload-identity federation (`anthropic_federation_rule_id` +
   `anthropic_organization_id`). This workflow authenticates with
   `claude_code_oauth_token` and sets no federation inputs, so no OIDC exchange
   occurs. **Verdict: `id-token: write` removed** from the `permissions:` block,
   with a comment citing the exact `action.yml` evidence. The three
   demonstrably-used permissions (`contents`, `pull-requests`, `issues`: write)
   are retained.
2. **Deterministic post-session guard (AC-011, AC-014).** Added
   `self-improvement-pr-guard.sh` (staging): usage
   `self-improvement-pr-guard.sh <changed-paths-file>`; empty or absent file =>
   vacuous pass (exit 0); any path matching the enforcement-chain denylist =>
   list every violating path, exit 1; otherwise exit 0. Pure and side-effect
   free (reads one file, writes stdout/stderr, sets exit code).
3. **Workflow guard wiring (AC-011, AC-014).** The updated
   `self-improvement.yml` records the head refs of already-open PRs before the
   session (baseline), then after the session enumerates open PRs whose head
   ref is new (session-created), collects their changed paths with
   `gh pr list --json headRefName,files`, and runs the guard on that list.
   Guard failure fails the run; no session-created PR => empty list => vacuous
   pass. `gh`/`jq` fetch failure fails the run (fail-closed).

## OIDC Necessity Verdict (AC-010)

**Verdict: REMOVE `id-token: write`.** The pinned action performs a GitHub OIDC
token exchange only when workload-identity federation inputs are supplied; this
workflow supplies none.

Source of truth (single, per requirements.md Assumptions and design.md External
Integrations): the pinned SHA
`558b1d6cab4085c7753fe402c10bef0fbb92ac7a` (v1.0.165).

Evidence retrieved via WebFetch of
`https://raw.githubusercontent.com/anthropics/claude-code-action/558b1d6cab4085c7753fe402c10bef0fbb92ac7a/action.yml`.

Verbatim `action.yml` input descriptions:

- Input `anthropic_federation_rule_id`: "Workload identity federation rule ID
  (fdrl_...). When set with anthropic_organization_id, the action authenticates
  to the Claude API by exchanging the workflow's GitHub OIDC token instead of
  using a static API key. Requires `id-token: write` permission."
- Input `anthropic_oidc_audience`: "Audience to request on the GitHub OIDC token
  used for workload identity federation. Defaults to https://api.anthropic.com."
- Input `anthropic_organization_id`: "Anthropic organization UUID used for
  workload identity federation".
- `runs.using`: `composite`.

Applied reasoning: the only documented consumer of `id-token: write` is the
`anthropic_federation_rule_id` federation path ("Requires `id-token: write`
permission"), which is gated on that input being set together with
`anthropic_organization_id`. The `Run Claude self-improvement session` step's
`with:` block sets only `claude_code_oauth_token`, `prompt`, and `claude_args`
-- no federation inputs -- so the action authenticates via
`claude_code_oauth_token` and never requests an OIDC token. Therefore
`id-token: write` is unused and is removed (AC-010). Per design.md External
Integrations, if this were wrong the action would fail fast at its own step
with a missing-permission error (a loud, non-silent failure), not a bypass. The
staging workflow's `permissions:` comment records this verdict and the exact
requires-phrase, and instructs re-verification when the SHA is bumped.
