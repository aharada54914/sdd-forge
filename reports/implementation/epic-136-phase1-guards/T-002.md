## Target

Fix the working-directory bypass of the R-10 protected-write analysis in the
Python and Node guard twins (`sdd-hook-guard.py`, `sdd-hook-guard.js`).
`has_protected_path` matched a protected path only as a literal substring of the
command text, so `cd plugins/sdd-quality-loop/scripts && rm sdd-hook-guard.py`
(and `pushd` equivalents) resolved the write target below the protected prefix
by basename and escaped denial — a claimed sudo-unbypassable control failing
open on the guard itself (REQ-002; security-spec.md B1).

## Summary

Added a working-directory-aware R-10 pass to both twins. It tokenizes the
command, splits it into compound-command segments (`&&`, `||`, `;`, `|`), tracks
a current-directory prefix updated by `cd`/`pushd` arguments, and resolves each
write verb/redirect target (`rm`/`tee`/`touch` every non-flag arg; `cp`/`mv` the
final arg; redirect targets) against that tracked directory before comparing to
`PROTECTED_GATE_SUFFIXES`. Unresolvable transitions (`cd -`, `cd "$VAR"`, bare
`cd`, `popd`) taint the tracked directory; a subsequent write verb on a
protected basename then fails closed. Read-only segments produce no hit, so the
existing read-only short-circuit and every prior decision are preserved. The new
pass is additive (an early `deny` before the unchanged substring + issue-#62
logic), so it can only add denials for genuine resolved-protected writes — it
never flips a prior allow. The `.py` and `.js` implementations are line-for-line
equivalent to keep decision parity.
