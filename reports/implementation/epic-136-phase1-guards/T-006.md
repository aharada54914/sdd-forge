## Target

Extend the Claude Code PreToolUse hook matcher in
`plugins/sdd-quality-loop/hooks/claude-hooks.json` so Bash tool calls are
routed to `sdd-hook-guard.js`, closing the issue #160 / #116 gap in which a
protected-file write launched through Bash was never intercepted under stock
Claude Code (REQ-006, AC-012, AC-013, security-spec B1). The live file is
enforcement-chain protected (R-10), so the corrected content is staged under
`specs/epic-136-phase1-guards/human-copy/claude-hooks.json` for a human to
copy into place.

## Summary

The Codex config (`hooks/hooks.json`) already matches
`Edit|Write|MultiEdit|apply_patch|Bash|bash|shell|exec_command|exec`, but the
Claude config matched only `Edit|Write|MultiEdit|apply_patch`. The guard's
Node twin (`sdd-hook-guard.js`) already classifies Bash payloads correctly
(denying protected-file writes, allowing read-only commands, failing closed on
malformed input) -- the defect was purely that Claude Code never routed Bash
to it. The fix changes ONLY the sdd-hook-guard entry's matcher string to match
the Codex coverage. The kill-switch PreToolUse entry (matcher `*`) and the
exec-form Node invocation are byte-for-byte unchanged; the `description` is
unchanged because it does not enumerate matcher coverage (minimal diff).

RED-first (TDD): `tests/claude-bash-matcher.tests.sh` drives the matcher gap to
a failing assertion against the live file, then passes against the staged file,
while independently proving (in both runs) that the guard already denies Bash
writes, allows read-only Bash over protected paths, fails closed on malformed
payloads, and that the Claude/Codex/Copilot hook paths agree on the shared
Bash corpus.
