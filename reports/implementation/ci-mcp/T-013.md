## Cycle 2 Fix

The independent quality-gate evaluator confirmed a **Major**: security-spec.md
B2 declares the DoS mitigation for huge job logs as "Ńā¬Ńā│Ńé░ŃāÉŃāāŃāĢŃéĪ + 256 KiB
õĖŖķÖÉ(µ£½Õ░ŠÕä¬Õģł)" (ring buffer + 256 KiB cap, tail-priority), mapped to
REQ-008 / AC-004 / TEST-004. The Cycle-1 implementation did not bound memory:
`githubGetText` (`github-client.ts`) called `await response.text()`,
materializing the *entire* upstream job-log body in memory before
`truncateLogTail` (`tools/actions.ts`) made a second full copy via
`Buffer.from(log, "utf-8")`. Peak memory scaled with the full log size
(~2x), even though only the final *response* was capped ŌĆö the fetch path
itself was unbounded, and this Goal/Summary language ("ring buffer") did not
match the implementation. TEST-004 also only exercised response truncation
against an in-memory fake body, never a streaming one, and never a multibyte
fixture crossing the cut boundary.

Fix (implements what security-spec.md already mandated ŌĆö no spec change):

- `github-client.ts` gained `readBoundedTail`, a bounded streaming tail
  reader: it consumes a `ReadableStream<Uint8Array>` (`response.body`, which
  every real `fetch` `Response` exposes) chunk-by-chunk, retaining chunks in
  a small deque and evicting whole chunks from the front whenever
  `retainedBytes - firstChunk.byteLength >= CAP + MARGIN` (`CAP` = 262144,
  `MARGIN` = 65536 bytes of slack). It tracks `totalBytesRead` (the true
  upstream size) and `maxRetainedBytes` (exposed for direct test
  verification of memory-boundedness). `githubGetText` now uses this as its
  primary read path whenever `response.body` is present, falling back to
  `.text()` only when it is absent (some hand-written test fakes) ŌĆö that
  fallback is not memory-bounded, but the real `defaultTextFetch` (`fetch`)
  always provides a `body`. Error-normalization behavior (401/403/404/429/
  5xx/network) is unchanged ŌĆö errors are still detected from `status` before
  any body streaming begins, via the same `normalizeUpstreamResponse` /
  `normalizeNetworkError` reuse as before.
- `tools/actions.ts`'s `truncateLogTail` is unchanged in its UTF-8-safe
  boundary logic and response contract (`{kind:"job-log", jobId, log,
  truncated, returnedBytes}`, `returnedBytes` <= 262144), but now operates
  on the already-bounded tail `githubGetText` returns instead of the full
  body. `getJobLog` now derives the final `truncated` flag by comparing
  `githubGetText`'s `totalBytesRead` (true upstream size) against the final
  `returnedBytes` ŌĆö true iff the source was longer than what is returned,
  regardless of whether the reduction happened during streaming, during the
  final trim, or both.
- New regression tests (all green, see Test Evidence below):
  `tests/tools/get-job-log.test.ts` gained 3 cases using a genuinely

## Specification Differences

Three items, disclosed honestly:

1. **`githubGetText` addition beyond the task's Scope line.** T-013's Scope
   in tasks.md lists only `mcp/ci-mcp/src/tools/actions.ts` (get_job_log
   implementation + zod schema) and `server.ts` registration ŌĆö
   `github-client.ts` is not listed. In practice, `get_job_log`'s upstream
   GitHub Actions job-logs endpoint returns a plain-text body (log content),
   not JSON, and `githubGet` (T-002) only supports `.json()`-parsed bodies.
   Rather than parsing text as JSON (which would break) or duplicating
   `githubGet`'s host-fixing/GET-only/error-normalization logic inline
   inside `actions.ts` (which would violate the "GET-only, host-fixed
   request path lives in github-client.ts" boundary T-002/T-006 establish
   and check statically), `githubGetText` was added as a text-bodied sibling
   of `githubGet` in `github-client.ts`, sharing every behavior except the
   body-parsing method. This is a minimal, boundary-respecting extension of
   an existing file rather than new Scope; it is called out explicitly here
   because it technically exceeds the Scope line's file list. **Cycle-2
   update:** this placement turned out to be exactly right per design.md's
   component table (`mcp/ci-mcp/src/github-client.ts` | "GET Õ░éńö© fetch
   Ńā®ŃāāŃāæŃā╗õĖŖµĄüŃé©Ńā®Ńā╝µŁŻĶ”ÅÕī¢Ńā╗**ŃéĖŃā¦Ńā¢ŃāŁŃé░ truncation**Ńā╗ŃāłŃā╝Ńé»Ńā│ŃāśŃāāŃāĆõ╗śõĖÄ"),

2. **`truncateLogTail`'s UTF-8 byte-boundary nuance.** The 256 KiB cap is a
   byte cap (`Buffer.byteLength`), not a character-count cap. A naive
   byte-offset slice at exactly 262144 bytes could land mid-way through a
   multibyte UTF-8 character (if the log contains non-ASCII content),
   producing either a decode exception or replacement characters when
   converted back to a string. `truncateLogTail` therefore scans backward
   from the 262144-byte offset for a byte that is a valid UTF-8 lead byte
   (not a `10xxxxxx` continuation byte) and slices there instead ŌĆö meaning
   `returnedBytes` can be a handful of bytes fewer than 262144 on logs whose
   cap boundary falls mid-character, but is never truncated mid-character
   and never exceeds the cap. AC-004's Done-When text ("õĖŖķÖÉ 256 KiB")
   doesn't spell out this edge case; it is a faithful, conservative reading
   of "at most 256 KiB, tail-priority" that avoids emitting corrupt text,
   not a deviation from intent. **Cycle-2 update:** a dedicated multibyte
   fixture crossing the cut boundary is now covered by
   `tests/tools/get-job-log.test.ts` (see Cycle 2 Fix), closing the gap this
   item originally disclosed.
3. **Ring-buffer deviation from security-spec.md B2 ŌĆö RESOLVED in Cycle 2.**
   security-spec.md B2 declares "Ńā¬Ńā│Ńé░ŃāÉŃāāŃāĢŃéĪ + 256 KiB õĖŖķÖÉ(µ£½Õ░ŠÕä¬Õģł)"
