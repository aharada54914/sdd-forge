## Target

Feature: ci-mcp / Task T-006 (Risk: high, Required Workflow: tdd).
Enforce the write boundary with two verification layers: (a) an input-schema
check that all 5 tools' zod schemas reject write-inducing fields
(action/method/body/command-family) with `invalid-input`, and (b) a static
(grep-shaped) check that `src/` contains zero write-method `fetch` calls
(POST/PATCH/PUT/DELETE), zero fs-write API references, zero `child_process`
usage in any form, and zero `eval` usage, with HTTP method fixed to GET.

## Summary

T-006 is a pure verification task: it adds no new `src/` production code (all
5 tools and their supporting modules were already implemented and already
compliant, landed by T-001ŌĆōT-005/T-012/T-013). Its deliverable is two new
test suites plus their shared support modules, TDD (Red -> Green):

- `mcp/ci-mcp/tests/no-write/support/write-inducing-fields.ts` (new) ŌĆö the
  shared denylist of write-inducing field names (`action`, `method`, `body`,
  `command`, and near-synonyms) used by both new no-write test files, so the
  probed field list and the schema-shape assertions share one source of
  truth rather than two independently-typed literal lists that could drift.
- `mcp/ci-mcp/tests/no-write/input-schema-fields.test.ts` (new, AC-006 part
  1): asserts every one of the 5 tools' zod input schemas declares only
  read-scoped allowlisted field names, that each schema has at least one
  field, and that the probed denylist is drawn from the shared support
  module (not a second, possibly-drifted copy).
- `mcp/ci-mcp/tests/no-write/no-write-behavior.test.ts` (new, AC-006 part
  2): for each of the 5 tools, asserts that passing a write-inducing field
  (`action`/`method`/`body`/`command`) is rejected with `invalid-input`
  *before* any GitHub call is made (the injected fetch is asserted
  never-called) ŌĆö 20 cases (5 tools x 4 denylisted fields).
- `mcp/ci-mcp/tests/readonly/support/source-scan.ts` (new) ŌĆö a small,
  dependency-free source-text scanner (reads every `.ts` file under a given
  root, applies a set of regex probes) shared by `static-check.test.ts`,
  following the same pattern T-001's `tests/readonly/static-check.test.ts`
  established for `local-env-mcp`.
- `mcp/ci-mcp/tests/readonly/static-check.test.ts` (new, AC-007): 7 static
  assertions over `src/`: zero fs-write API references, zero
  `child_process` usage in any form (no `execFile` exception ŌĆö ci-mcp is
  stricter than `local-env-mcp`, see Specification Differences), zero
  `eval` usage, zero write-method HTTP literals (POST/PUT/PATCH/DELETE),
  the only `fetch(...)` call sites are in `github-client.ts`,
  `github-client.ts` fixes every HTTP method literal to `GET`, and the only
  `scheme://host` literal anywhere in `src/` is the fixed GitHub API base
  URL.

Tests were written first and captured failing (Red: pretest `tsc` fails ŌĆö

## Specification Differences

One item, disclosed honestly: T-006's static check bans `child_process`
usage in **any** form in `src/`, with **no** `execFile` exception. This is
stricter than `local-env-mcp`'s equivalent static check
(`mcp/local-env-mcp/tests/readonly/static-check.test.ts`), which explicitly
permits `execFile` (its probe-engine's entire purpose is to `execFile`
allowlisted CLI version probes). ci-mcp has no legitimate subprocess use
case anywhere in its design (T-004's `repo-resolve.ts` explicitly rejects
git-remote-inference specifically to avoid needing `exec`, per its OQ-001
resolution), so T-006's tasks.md Goal text ("`child_process`(exec/spawn/
execFile)... Ńüī 0 õ╗Č") is read literally as zero tolerance for the whole
family including `execFile`, rather than importing local-env-mcp's
execFile-shaped carve-out by default. This is a deliberate, tighter
interpretation consistent with ci-mcp's read-only, no-subprocess design ŌĆö
not an accidental copy-paste gap from the local-env-mcp precedent.
