## OQ-004 Resolution

**Question:** Which environment variable(s) supply the read-only GitHub
token, and in what priority order, when more than one is set?

**Confirmed priority order:** `CI_MCP_GITHUB_TOKEN` â†’ `GH_READONLY_TOKEN` â†’
`GITHUB_TOKEN`, first non-empty value wins (an empty-string value is treated
as unset and the resolver falls through to the next variable). This matches
design.md's proposed order verbatim; T-003 confirms it as final rather than
provisional.

**Rationale:**
- `CI_MCP_GITHUB_TOKEN` is checked first because it is ci-mcp-specific: a user
  or CI environment that wants to hand ci-mcp a different (or scoped-down)
  token than other tools use can do so without disturbing anything else that
  reads `GITHUB_TOKEN`.
- `GH_READONLY_TOKEN` is checked second because it already exists as a
  cross-project convention in this repo (see the
  `github-mcp-read-write-split` decision: a dedicated read-only PAT variable
  name shared with other MCP servers here) â€” falling back to it lets ci-mcp
  pick up the same read-only credential other read-only tooling in this
  workspace already uses, with no extra configuration.
- `GITHUB_TOKEN` is checked last because it is the most generic/ambient name
  (commonly auto-populated by CI systems such as GitHub Actions itself) and
  the most likely to be a broader-scoped token than intended for a read-only
  MCP server; it is a reasonable last resort but not the first choice.
- No variable resolving to a non-empty value returns `auth-missing`
  (REQ-005) â€” the process never terminates and no tool ever calls GitHub
  without a token.

**Implementation:** `mcp/ci-mcp/src/auth.ts` â€” `resolveToken()` walks
`TOKEN_ENV_PRIORITY` in this exact order; `withToken()` is the shared gate a
tool handler calls through (starting at T-005), short-circuiting to
`auth-missing` without invoking the handler when no token resolves.

## Note on "every registered surface" (AC-008/AC-009 wording)

`acceptance-tests.md` describes AC-008/AC-009 in terms of "all tools"
returning `auth-missing` / not leaking the canary. No tool surfaces are
registered yet â€” `server.ts` still registers zero tools (T-001 scope; the 5
read-only Actions tools are added starting T-005/T-012/T-013), which is
independently verified by the pre-existing
`buildServer() constructs a server with zero tools registered` test. T-003's
tests therefore validate the acceptance criteria at the one place that gates
every future tool call: `withToken()` in `auth.ts`. Every tool handler added
from T-005 onward is expected to route its GitHub-touching logic through
`withToken`, so proving `withToken` returns `auth-missing` without invoking
its handler (and never terminates the process) proves the criterion will
hold for each concrete tool once they exist.
