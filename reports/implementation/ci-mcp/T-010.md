## Target

Feature: ci-mcp / Task T-010 (Risk: high, Required Workflow: tdd).
Extend `uninstall.sh` / `uninstall.ps1` so a placed `ci-mcp` is removed and
the installer-managed `ci-mcp` entry is deregistered from Claude / Codex /

## Summary

- `uninstall.sh` (modified) -- `VALID_MCPS`/`MCP_LIST` += `ci-mcp`; usage/help
  text and the Node-absent Cursor/VS Code warning updated to name `ci-mcp`.
- `uninstall.ps1` (modified) -- same, `ValidateSet`/default array parity.
- `tests/uninstall.tests.sh` (modified) -- `seed_installed_mcps` extended to
  3 MCPs incl. ci-mcp Codex marker block + Cursor/VS Code entries; new
  `--mcp "sdd-forge-mcp,local-env-mcp,ci-mcp"` invocations; ci-mcp-specific
  removal assertions added to scenarios (o)/(p).
- `tests/uninstall.tests.ps1` (modified) -- ps1 parity for all of the above.

No new removal *logic* was required: `uninstall.sh`/`uninstall.ps1` already
implement a fully generic `MCP_SELECTION`/`-Mcp`-driven loop (established
when local-env-mcp was added) for payload removal
(`remove_mcp_payload`/`Remove-McpPayload`), Claude deregistration
(`unregister_claude_mcp`/`Unregister-ClaudeMcp` -- `claude mcp remove
<name>`), Codex config.toml marker-block stripping
(`unregister_codex_mcp`/`Unregister-CodexMcp` -- matches `# >>> <name> ...
>>>` / `# <<< <name> <<<`), and Cursor/VS Code `mcp.json` managed-key-only
deletion (`remove_mcp_json_keys`/`Remove-McpJsonKeys`). Adding `"ci-mcp"` to
the 4 selection-list locations was sufficient for it to be deregistered
through every one of those paths.

Two safety properties were specifically re-verified (not re-derived) to also
cover ci-mcp:

- **Broken-JSON failsafe**: `remove_mcp_json_keys`/`Remove-McpJsonKeys` exit 3
  on invalid JSON, leave the file byte-identical, print an "invalid JSON"
  notice, and continue with the other client -- verified for ci-mcp's 3-name
  list by scenario (q), which still corrupts only the Cursor file and
  confirms VS Code (carrying the ci-mcp key too) is still cleaned.
- **Delete-only-managed-name boundary**: `remove_mcp_json_keys`/
  `Remove-McpJsonKeys` only `delete section[name]` for names in the selection;
  scenario (o) asserts the pre-seeded user-defined `"my-server"` key and the
  unrelated `"telemetry"`/`"inputs"` top-level keys survive alongside the
  sdd-forge-mcp/local-env-mcp/ci-mcp removals, and the Codex config.toml's
  unrelated `[some_other_section]` block survives scenario (p)'s full 3-MCP
  removal.

Red evidence (`T-010-red.txt`) discloses the same deviation pattern as T-009:
`uninstall.sh`/`uninstall.ps1` were reverted to their last-committed
pre-ci-mcp content (`git show HEAD:uninstall.sh > uninstall.sh`,
non-destructive, backed up first) while the new 3-MCP test scenarios were
kept in place; the resulting run produced genuine `FAIL:` lines (managed keys
not removed, `claude mcp remove ci-mcp` not invoked, Codex ci-mcp block not
removed, corrupt-JSON scenario failing to continue) before the ci-mcp-extended
source was restored and the suite went green (24 passed / 0 failed).
