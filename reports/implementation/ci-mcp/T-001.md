## Summary

Established the `mcp/ci-mcp/` TypeScript project scaffold and implemented the
three T-001 source modules acceptance-first (acceptance tests written first
and captured failing, then implementation to green):

- `src/envelope.ts` — `OkEnvelope<T>`/`ErrorEnvelope`, the 10-code `ErrorCode`
  type (the 7 codes shared with sdd-forge-mcp/local-env-mcp plus
  `upstream-error`/`rate-limited`/`auth-missing`), and `ok()`/`err()`/
  `isOk()`/`isErr()` helpers. A test asserts the TS `ErrorCode` union equals
  the contract `error.code` enum exactly (10 codes) and that all 7
  pre-existing codes plus the 3 ci-mcp additions are present.
- `src/server.ts` — `buildServer()` constructs an `McpServer` (name `ci-mcp`,
  version `0.1.0`). At T-001, zero tools are registered (the 5 read-only
  Actions tools land starting T-005/T-012/T-013); construction performs no
  I/O, no environment-variable read, and no network call, keeping the <=1s
  startup SLO.
- `src/index.ts` — entrypoint: builds the server and connects it over stdio
  via `StdioServerTransport`. At T-001 this used a minimal inline
  `JSON.stringify` stderr write for fatal errors (later replaced by the
  redaction-safe `diagnostics.ts` logger added in T-003 — see Specification
  Differences below).

Acceptance tests were written first and captured failing (Red: pretest `tsc`
fails with module-not-found against `src/envelope.js`/`src/server.js`, exit
2), then all 6 pass after implementation (Green, exit 0). A manual stdio
startup smoke (recorded in the Green evidence file) additionally confirmed:
server replies to an `initialize` JSON-RPC request within ~1.5s with
`capabilities:{}` (zero tools, matching T-001 scope) and produces no stderr
output — i.e. no token read and no network call attempted at startup,
satisfying the Done-When "stdio 起動が成功し...トークン検証を行わない" criterion.
`tsc --noEmit` is clean.
