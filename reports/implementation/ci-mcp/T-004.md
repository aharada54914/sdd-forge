## OQ-001 Resolution

**Question:** How does a caller specify the target repository — explicit
tool arguments, an environment variable, or git-remote inference — and what
is the canonical priority when more than one is available?

**Confirmed canonical priority:**

1. Explicit `owner` / `repo` tool arguments, when **both** are provided
   together. If only one of the two is given, that is treated as an
   ambiguous/invalid explicit specification (`invalid-input`) — it does NOT
   silently fall back to step 2.
2. The `CI_MCP_REPO` environment variable, in `owner/repo` form, used only
   when no explicit arguments were given at all.
3. `invalid-input` when neither resolves to a valid pair.

Git-remote inference is explicitly excluded: ci-mcp never execs a subprocess
and never reads `.git/config` or invokes `git remote` to guess the
repository. This matches design.md's proposed order verbatim; T-004
confirms it as final rather than provisional.

**Rationale:**
- Explicit tool arguments are the least ambiguous way for an MCP client to
  say "run this against repository X" and must always be honored exactly,
  including honoring their invalidity — if a caller passes a malformed
  `owner`/`repo`, silently substituting a different repository from
  `CI_MCP_REPO` would be a worse failure mode (wrong-repo data returned
  looking like a success) than a clear `invalid-input`.
- `CI_MCP_REPO` gives a convenient single-repository default for a client
  session that always targets the same repo (the common case for an IDE
  workspace), without requiring every tool call to repeat `owner`/`repo`.
- Git-remote inference was rejected outright: it would require an exec-based
  code path (`git remote -v` or reading `.git/config`), which is
  security-spec.md's B2 "exec avoidance" Non-goal for this server (any
  `child_process` usage is treated as a static-check-failing pattern per
  design.md's Test Strategy). Avoiding exec entirely also avoids a whole
  class of injection/path-confusion risk for no material convenience gain
  over `CI_MCP_REPO`.

**Implementation:** `mcp/ci-mcp/src/repo-resolve.ts` — `resolveRepo()`
implements exactly this 3-step priority, validates `owner`/`repo` against
conservative GitHub-style name allowlists (defense-in-depth; the SSRF-relevant
guarantee that the host cannot be substituted is independently enforced by
`github-client.ts`'s fixed base URL + per-segment percent-encoding), and never
imports Node's subprocess-spawning module.
