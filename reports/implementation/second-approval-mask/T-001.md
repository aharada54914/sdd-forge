## Summary

Taught the task-stage normalization in BOTH `check-workflow-state` twins to
DELETE column-0 `Second Approval:` lines (the whole line including its
terminator) before hashing, so a critical-tier second-approval recorded in
`tasks.md` after the task-review freeze normalizes to the byte-identical stream
that was frozen and no longer trips `stage-provenance: task plan hash is stale`.
Every other post-freeze edit to `tasks.md` still trips staleness. The live
protected twins were NOT modified; the fixed twins are staged under
`specs/second-approval-mask/human-copy/` for a human to apply.

## The change (one line per twin)

- `check-workflow-state.sh`, `normalized_hash()` task branch: added
  `-e "/^Second Approval:/d"` to the existing `sed` (after the three value
  masks). `sed d` removes the whole pattern space including its terminator; a
  CRLF line matches because the pattern is a column-0 prefix match.
- `check-workflow-state.ps1`, `Get-NormalizedHash` `"task"` branch: added
  `$text = [regex]::Replace($text, "(?m)^Second Approval:[^\r\n]*\r?\n?", '')`
  after the three masks. The `\r?\n?` tail consumes the terminator, so a deleted
  final line without a trailing newline matches `sed` byte-for-byte.

The two patterns are column-0 anchored and disjoint from the three existing
value masks (`Task-Review-Status:`, `Approval:`, `Status:`), so ordering does
not affect the result.
