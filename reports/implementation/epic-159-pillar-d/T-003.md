## Specification Differences

- **`.ps1` twin: native re-implementation, not a `gh.ps1`-shaped PATH
  stub**: design.md's own test-technique sketch for T-003 says the `.ps1`
  twin uses "PowerShell functions + a `gh.ps1`-shaped stub on PATH." This
  report deliberately used an IN-PROCESS mock (a script-scoped invocation
  log plus parameters controlling the mocked "already-open issue" lookup)
  instead of a literal external-process PATH stub, for robustness on the
  Windows CI lane (PowerShell script-file PATH resolution for a bare `gh`
  invocation is less portable than an in-process function call). No AC or
  TEST row in `acceptance-tests.md` mandates the stubbing MECHANISM, only
  that the SAME fixture-driven assertions are exercised natively -- which
  they are, scenario-for-scenario, against the twin's own native port of
  `fetch_source_or_unavailable`/`compute_divergence`/
  `file_or_dedupe_issue`. Recorded here as a disclosed implementation-detail
  deviation, not a silent scope change.
- **`compute_divergence`'s conservative-heuristic token filter (both the
  real script and its native `.ps1` port) requires a candidate token to
  contain at least one digit, in addition to the charset allowlist
  (`[A-Za-z0-9.-]`)**: design.md's own function-shape comment does not
  specify an exact extraction algorithm beyond "the description is built
  EXCLUSIVELY from model-ID tokens matching the charset allowlist" and
  Non-goals explicitly authorizes a conservative, imprecise heuristic
  ("false negatives acceptable... not a precise parser"). The added
  digit-presence filter is a deliberate noise-reduction choice: without it,
  ordinary prose words with internal dashes (e.g. a marketing page's
  "state-of-the-art") would satisfy the charset allowlist and generate
  spurious divergence issues on every run. The accepted, documented
  trade-off is that this repository's own no-digit alias model names
  (`anthropic/haiku`/`sonnet`/`opus`) can never be DETECTED as divergent
  from vendor-page prose by this heuristic -- an accepted false-negative
  (Non-goals), not a defect; D1's manual capability-refresh checklist
  remains the primary defense for those names (requirements.md User
  Stories).
- **TEST-009's CI-resilience self-check technique deviates from
  `tests/bump-version-gate.tests.sh`'s exact style** for the
  "no possibly-empty bash array expanded under `set -u`" check: rather than
  searching for an "unguarded" variant of an array-expansion pattern (which
  would self-collide with this suite's OWN check-line source, since any
  textual representation of the guarded idiom `${arr[@]+"${arr[@]}"}`
  necessarily contains the "bad-looking" fragment as a sub-expression -- see
  Working Notes), this suite instead makes the STRONGER, collision-free
  positive claim: it declares NO bash array anywhere in its own source, so
  no possibly-empty array can be expanded under `set -u` by construction.
  Verified directly: `grep -c '\[.*\]=\|declare -a\|local -a' tests/
  model-freshness-check.tests.sh` and manual read-through confirm zero
  array declarations.
