Two implementation-time findings, both disclosed as deviations:

1. **A design-intent gap identified and closed during implementation**
   (not a bug in existing code, but an omission the literal design.md
   "Planned shape" snippet would have introduced): `release.yml`'s
   top-level `permissions:` block (`release.yml:25-28`) grants
   `contents: write` / `id-token: write` / `attestations: write` to the
   *entire workflow*. GitHub Actions job-level permissions inherit from the
   workflow level unless a job declares its own, narrower `permissions:`
   key. design.md's Planned shape code snippet for the `loop-gate` job did
   not include a `permissions:` override, which — despite the Risk
   Rationale, security-spec.md B4, and security-spec.md's Secrets
   Management section all explicitly asserting "the new loop-gate job
   requests no elevated permissions" / "no elevated GitHub Actions
   permissions beyond the default contents: read" — would have left the
   job silently inheriting the full write/attestation grant. Added
   `permissions: contents: read` directly on the `loop-gate` job
   (`release.yml:32-39`) to make the implementation match the design's own
   stated security posture. This is the exact "Elevation of Privilege" risk
   security-spec.md's OWASP Mapping table names ("the loop-gate job being
   granted release-upload permissions it does not need"); closing it here
   removes the gap rather than deferring it to "code review" as the spec's
   control column proposed.
2. A bug in this task's own new suite, found and fixed during
   implementation: the scratch python3 checker script was created via
   `mktemp "${TMPDIR:-/tmp}/release-loop-gate-checker.XXXXXX.py"` — a
   template with a literal suffix (`.py`) *after* the `XXXXXX` run. BSD/macOS
   `mktemp` (unlike GNU `mktemp`) only substitutes a trailing run of `X`
   characters when it is the very last thing in the template; with a
   suffix present it silently returns the template unchanged
   (un-randomized) and creates that literal file. The first suite run
   therefore succeeded by accident, but every subsequent run on the same
   host failed with `mkstemp failed ... File exists`, since plain `mktemp`
   refuses to overwrite an existing path. Fixed by `mktemp -d`-ing a
   directory with a bare `XXXXXX` suffix (no literal text after it — the
   already-correct pattern this suite's own `TEST-009` canary fixture and
   `tests/bump-version-gate.tests.sh`'s fixture roots use) and writing the
   fixed-name `checker.py` inside that directory. Verified: three
   consecutive runs on this (BSD-`mktemp`) host now all succeed without
   collision (see Test Evidence).
