## Summary

Added `render-agent-frontmatter.sh`/`.ps1` (repository root): a script that
renders `model:`/`x-sdd-effort:` frontmatter into Claude `.md` agent files
and `# x-sdd-model:`/`# x-sdd-effort:` reference comments into Codex
`.toml` agent files, sourced entirely from
`contracts/agent-model-capabilities.v2.json`'s `role_defaults`. A hardcoded
`TARGETS` table (10 entries: `sdd-evaluator`/`sdd-investigator` each with
one Claude `.md` + one Codex `.toml` target; `spec-reviewer` with two
unprotected Claude `.md` targets; `impl-reviewer`/`task-reviewer` each with
two PROTECTED Claude `.md` targets) drives every render decision -- nothing
outside this table is ever a write candidate.

For unprotected Claude targets the script rewrites ONLY the `model:`
frontmatter line (computed from `role_defaults[role].minimum_tier`'s
paired `anthropic/*` registry model name) and inserts/refreshes a trailing
`<!-- x-sdd-effort: <e> -->` comment line immediately after the
frontmatter's closing `---` -- a Markdown comment outside the YAML block,
so it never participates in frontmatter parsing. A `model: inherit` target
is returned byte-unchanged (AC-018). For Codex targets it inserts/refreshes
two `# x-sdd-model:`/`# x-sdd-effort:` comment lines at the top of the
file, leaving every existing TOML key (`name`, `description`,
`sandbox_mode`, `developer_instructions`) byte-unchanged (AC-015).

For the four PROTECTED reviewer targets (`impl-reviewer-{a,b}.md`,
`task-reviewer-{a,b}.md`), the identical corrected content is computed but
written ONLY to
`specs/epic-159-pillar-c/human-copy/<repository-relative-path>`, with a
`MANIFEST.sha256` SHA-256 entry appended/updated per file -- the real
protected path is never opened for write (AC-019). The write-target
resolution is split into a testable primitive
(`resolve_write_target_raw(relpath, protected)` in the Python core;
`Resolve-WriteTargetRaw` in the PowerShell twin) that is the ENTIRE
decision: `if protected: <human-copy path> else: <real path>`, exposed
directly on the CLI (`--resolve-target-raw`/`--resolve-target` on the
`.sh` twin; `-ResolveTargetRawPath`/`-ResolveTargetRole` etc. on the `.ps1`
twin -- a documented twin divergence, `[string]` scalar flags rather than
`[string[]]` arrays, because PowerShell's native `-File` argv binding does
not reliably collect multiple bare tokens into an array parameter across a
nested subprocess invocation, verified directly while writing this task's
own PowerShell test harness) so AC-019's self-check exercises the real
resolution function, never a reimplementation.

`--check` performs the identical read-and-compute step but never writes
anywhere: it diffs computed content against on-disk content at every
target's real path -- including the four protected targets' real paths (a
read, not a write) -- and exits non-zero on any drift (AC-016). This is
wired into `tests/validate-repository.ps1` (a new step immediately after
the existing workflow-state validation) and into `.github/workflows/test.yml`
via this task's own two new suite-registration CI steps (bash + pwsh lanes
for `tests/render-agent-frontmatter.tests.sh`/`.ps1`); a SEPARATE dedicated
`--check` CI step was judged unnecessary because `tests/validate-repository.ps1`
is already invoked in every CI lane (`Validate repository` / `Validate
repository (POSIX)` steps), so wiring `--check` inside that script
propagates to CI without a new step.

`role_defaults` was seeded (T-001) from the CURRENT hardcoded values, so
the real, one-time production render performed as part of this commit
(`./render-agent-frontmatter.sh` against the live repository, no flags) is
a zero-diff no-op on every unprotected target's `model:` VALUE -- it only
ADDS the new `x-sdd-effort` comment line / Codex header lines, never
changes an existing `model:` selection (AC-017, verified both by this
task's own suite against a scratch mirror of the real content, and by the
real `git diff` after the production render: every one of the six
unprotected targets' diffs is a single added line). The four protected
targets' real files are confirmed byte-identical before/after (`git diff
--stat` against the four reviewer `.md` files and `.github/workflows/test.yml`
is empty).

**PowerShell case-sensitivity (2 layers, T-002 precedent)**: every
comparison against a registry-sourced string uses an explicit
case-sensitive operator. Layer 1 -- `canonical_tier`/`effort_control.'codex-cli'`
matching in `Get-ModelForTier`/`Get-EffortControlValue` uses `-ceq`/`-cmatch`
against a `.PSObject.Properties` enumeration, never bare dot-access. Layer
2 -- `role_defaults` key lookup in `Get-RoleValues` enumerates
`.PSObject.Properties` and matches `Name` with `-ceq`, avoiding PSObject
dot-access's default case-insensitivity. A dedicated mis-cased negative
fixture pair (a `role_defaults` key `"Sdd-Evaluator"` against a lookup for
`"sdd-evaluator"`; a `canonical_tier` value `"Strong"` against a lookup for
`"strong"`) proves both twins reject the mis-cased value with a
`RENDER_ERROR` rather than silently aliasing it -- the `.sh`/python side is
case-sensitive by construction (Python string equality), the `.ps1` side
required the explicit fix described above.

A second `.ps1`-only defect surfaced while building this task's own test
harness: an uncaught `throw` renders through pwsh 7's default
colorized/line-wrapped `ConciseView` error formatter, which breaks any
caller doing a plain substring match against the error text (including
this suite's own subprocess-based assertions). Fixed by wrapping
`render-agent-frontmatter.ps1`'s entire dispatch body in a top-level
`try`/`catch` that prints `$_.Exception.Message` via
`[Console]::Error.WriteLine()` (plain text, no ANSI, no wrapping) and exits
1 on any error.

TDD (`tdd` required workflow, risk `high`) was followed with the
write-boundary (AC-019) and read-boundary (AC-020) RED/GREEN pairs
recorded SEPARATELY inside the suite itself (not merely as two lines in
one combined check), per tasks.md's Done-When: AC-019's RED case forces
`Protected=0` on a real protected basename via
`--resolve-target-raw`/`-ResolveTargetRawPath` and confirms the resolution
function then resolves to the REAL path (proving the branch is live, not
hardcoded-safe); its GREEN case confirms `Protected=1` (both the raw
override and the shipped default `TARGETS` table) always resolves under
`human-copy/`. AC-020's RED case renders a scratch copy of a
protected-position target once (so it is correctly synced), confirms
`--check` reports `OK`, then mutates its `x-sdd-effort` value and confirms
`--check` reports `DRIFT` again (a mutation-based negative self-check,
mirroring T-001/T-002's established pattern, applied here to the
read-boundary claim specifically); the GREEN half is the pre-mutation `OK`
result. Overall-suite RED/GREEN (the whole suite failing meaningfully
because the target script does not exist yet, then passing once it does)
was additionally captured for both twins by temporarily moving
`render-agent-frontmatter.sh`/`.ps1` aside and re-running the suite.

  Deployment/CI Plan states only that `--check` "additionally joins
  `tests/validate-repository.ps1`'s existing check sequence." Since that
  script is already invoked by every CI lane (`Validate repository` /
  `Validate repository (POSIX)` steps, both already present in
  `.github/workflows/test.yml`), wiring `--check` inside
  `validate-repository.ps1` itself propagates the check to CI without a
  separate, redundant step. This task's ONLY new CI steps are its own

  `canonical_tier` resolves to on the Codex side when more than one
  `codex-cli`-`flag`-controlled model shares that tier (`strong` currently
  has both `openai/gpt-5.2-codex` and `openai/gpt-5.1-codex-max`). This
  task resolves it deterministically as "the first model in the registry's
  `models` array, in array order, matching the tier with
  `effort_control.codex-cli == "flag"`" -- for `strong` this selects
