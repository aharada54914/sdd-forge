## Summary

Added six new flags to `emit-run-record.sh`/`.ps1`: `--effort-main`,
`--effort-reviewers`, `--effort-control-main`, `--effort-control-reviewers`,
`--effort-applied-main`, `--effort-applied-reviewers` (PowerShell twin:
`-EffortMain`, `-EffortReviewers`, `-EffortControlMain`,
`-EffortControlReviewers`, `-EffortAppliedMain`, `-EffortAppliedReviewers`).
Supplying any one of them bumps the emitted `schema` to
`"sdd-run-record/v2"` and adds a sibling `effort` object (immediately after
the existing `model_ids` object, per design.md's API/Contract Plan
insertion point) with the identical two-key shape (`main`/`reviewers`),
each key carrying exactly three subfields: `effort_requested`,
`effort_applied`, `effort_degraded_reason`.

**Field-population rule** (per role slot, implemented in
`resolve_effort_slot`/`Resolve-EffortSlot`):

- If `--effort-<role>` was not supplied at all, all three subfields are
  `null` (the vacuous case never populates a reason -- Edge Cases,
  requirements.md).
- If `--effort-<role>` was supplied: `effort_requested` always carries
  that value (AC-022). `effort_applied` reaches a non-null value **only**
  when `--effort-applied-<role>` was supplied with a real (non-`"none"`)
  value **and** the paired `--effort-control-<role>` resolved to `"flag"`
  -- if a real applied value is supplied while the paired control is not
  `flag`, the script rejects with a non-zero exit and a diagnostic instead
  of silently trusting or silently nulling it (security-spec.md B4: "no
  path can report a false 'applied'", implemented structurally, not by
  caller convention). In every other reachable combination (`frontmatter`
  control, `none` control, `flag` control with an explicit `"none"`
  decline, or `flag` control with no `--effort-applied-<role>` at all),
  `effort_applied` is `null` and `effort_degraded_reason` is populated
  with one of four non-vacuous reason strings keyed on the **resolved
  `effort_control` value**, never on host identity (AC-024, AC-051):
  `effort-control-frontmatter`, `effort-control-none`,
  `effort-application-declined`, `effort-application-not-confirmed`. This
  directly implements AC-051: since `emit-run-record.sh`/`.ps1` has no
  `--host` parameter at all, a "Codex host selecting a
  `frontmatter`/`none`-control model" scenario is *structurally
  indistinguishable* from a Claude Code scenario -- both resolve through
  the identical `--effort-control-<role>` value and produce identical
  output shape (verified explicitly in the test suite by comparing the two
  scenarios' JSON sub-objects for equality).

**Backward compatibility (AC-025)**: the no-`--effort-*`-flag code path is
an intentionally untouched, unmodified duplicate of the pre-T-004
heredoc (`.sh`) / ordered-hashtable record construction (`.ps1`), selected
via an `if [ "$emit_v2" = "1" ]; then ... else ... fi` / `if ($emitV2) {
... } else { ... }` branch rather than conditionally mutating a single
shared template -- this guarantees the v1 shape can never silently drift
when v2 fields are added or edited above it. Verified two ways: (1) a
source-level `diff` of the current v1 branch's heredoc body / hashtable
body against the pre-T-004 `git show HEAD^:...` content, confirmed
byte-identical (`diff` exit 0, both twins -- see Test Evidence); (2) the
test suite runs the post-feature script with no `--effort-*` flag and
asserts `schema == "sdd-run-record/v1"` and no `effort` key at all, **and**
separately reads (never rewrites) a real, already-committed pre-feature v1
record (`reports/runs/RUN-20260705T023011Z-sdd-forge-mcp.json`) to confirm
it still reads as schema v1 with no `effort` key -- proving the validator

**PowerShell case-sensitivity (2 layers, T-002/T-003 precedent)**: layer 1
-- `--effort-control-*` values are checked against an ordinal
`[System.Collections.Generic.HashSet[string]]` (`[System.StringComparer]::Ordinal`)
entry gate at parse time, rejecting any value not exactly `flag`,
`frontmatter`, or `none` (case-sensitive) before it ever reaches
resolution logic. Layer 2 -- `Resolve-EffortSlot`'s own branch dispatch
uses `-ceq`/`-cne` exclusively, never PowerShell's default
case-insensitive `-eq`. A mis-cased negative fixture
(`-EffortControlMain "Flag"`) is asserted rejected fail-closed on both
twins (the `.sh` side's `case` pattern match is inherently case-sensitive
by construction, no glob wildcards involved). A second, T-003-precedented
fix was needed: the two new `Write-Error` call sites were replaced with
`[Console]::Error.WriteLine()` + `exit 1`, because pwsh 7's default
ConciseView terminating-error formatter (colorized, line-wrapped) breaks
any caller doing a plain substring match against the error text -- exactly
the defect T-003's report already documented and fixed the same way.

  implemented and tested against a self-contained fixture matching the

  the new check itself is correct in isolation, without attempting to
  reconcile the broader heading mismatch -- that reconciliation is outside
  T-004's Planned Files and would be an unrelated refactor.
- **Reason-string vocabulary is an implementation decision, not spelled
  out byte-for-byte in requirements.md/design.md**: requirements.md gives
  one illustrative example (`"host-no-effort-control"`), but the parent
  task instructions were explicit that the reason must be "keyed on the
  resolved `effort_control` value, not host identity." This task
  implements four distinct, non-vacuous reason strings
  (`effort-control-frontmatter`, `effort-control-none`,
  `effort-application-declined`, `effort-application-not-confirmed`),
  each directly naming the resolved `effort_control` value or the
  specific unconfirmed/declined-application condition, rather than a
  single generic `"host-no-effort-control"` string -- chosen because AC-051
  requires the SAME shape regardless of host, and a host-mentioning string
  would have been a poor fit for that requirement even as an example.
- **The `--effort-applied-*` + non-`flag`-control combination is rejected
  (non-zero exit), not silently degraded**: design.md's Data Plan
  describes the field-population rule declaratively but does not state
  what happens if a caller supplies a confirmed-applied value alongside a
  non-`flag` control (a caller-contract violation that should never occur
  from a correctly-implemented T-006 caller, since T-006 is the only
  future caller of `--effort-applied-*` with a real value). Given
  security-spec.md B4's explicit "no path can report a false 'applied'"
  language and the tasks.md Risk Rationale's "structural, not merely
  conventional" framing, this task chose fail-closed rejection over
  silent suppression -- a caller bug is surfaced immediately rather than
  silently producing a degraded-but-plausible-looking record.
