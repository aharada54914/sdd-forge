## Summary

`select-agent-model.sh`/`.ps1` now auto-detect the `--registry` file's
`schema` field: `agent-model-capabilities/v1` (including the legacy
positional `--candidate name:tier:cost` form) routes into the existing,
byte-unmodified v1 code path; `agent-model-capabilities/v2` routes into a
new branch that additionally reads `supported_efforts`, `default_effort`,
`effort_control`, `risk_effort_matrix`, and `role_defaults`.

Four new flags implement the REQ-002 effort-resolution priority order
exactly: `--effort-policy welded|matrix` (default `welded` for the
duration of Phase 1); `--requested-effort <e>` (explicit override, wins
under BOTH policies whenever supplied, clamped to the winning model's
`supported_efforts`, still gated by `--xhigh-reason` for `xhigh`);
`--role <role>` (always seeds `--minimum-tier` from
`role_defaults[role].minimum_tier` when the caller did not already set
one; under `matrix`, additionally seeds a `role-default` fallback effort
when `risk_effort_matrix` has no entry for the supplied `--risk`; under
`welded`, the effort component is structurally inert — proven by a case
that supplies `--role` on the exact AC-007 golden invocation and asserts
the output is unchanged); `--host claude-code|codex-cli` (default
`claude-code`, resolves the winning model's `effort_control` for that host
into JSON output).

`welded` reproduces v1-equivalent effort selection byte-for-byte: a
candidate's own declared `effort` (if the v2 candidates-file entry
supplies one) is used directly, exactly as v1 always required; only when a
v2 entry OMITS `effort` does the selector fall back to the winning model's
own `default_effort` (AC-013). `matrix` resolves
`risk_effort_matrix[risk]` (falling back to `role_defaults[role]
.default_effort` when `--role` is supplied and the matrix has no entry for
the risk, then to the winning model's own `default_effort` as the final
fallback), clamps to the winning model's `supported_efforts` using a
nearest-value clamp (bidirectional — both a too-high and a too-low base
value clamp to the nearest supported member, not merely a ceiling cap),
and applies a one-step escalation bump (re-clamped) only to the
`risk-matrix`-sourced value when an escalation event fires and the
registry's `risk_effort_matrix.escalation_bump` is `true`. The xhigh
eligibility gate was generalized from checking a candidate's raw declared
effort to checking its final POLICY-RESOLVED effort (`item[4]` in the
widened 6-element parsed tuple, vs. the original `item[3]`), evaluated
AFTER the escalation bump — this is a one-token rewrite of the existing
filter expression, behavior-preserving for v1/legacy because
`sort_effort == final_effort` always holds on that path; the existing sort
key's tiebreak dimension (`item[3]`) is untouched code, still driven by
each candidate's own declared/sort effort, never the policy-resolved one,
which is what makes the AC-008 sonnet-over-codex tiebreak resolve
correctly (both candidates omit a declared effort under `matrix`, so the
tiebreak falls through to cost then lexicographic name order, exactly as
today's tiebreak already does for undeclared-effort candidates).

JSON output gains two additive keys only when the v2 branch was taken:
`effort_source` (`requested`/`risk-matrix`/`role-default`/`model-default`/
`welded`) and `effort_control`; every pre-existing key
(`model`/`canonical_tier`/`effort`/`estimated_cost_per_attempt_usd`/
`available_candidates`/`xhigh_reason`/`escalation`) is unchanged in name
and type, and for a v1 or legacy-positional invocation neither new key is
ever emitted (the v1 code path is the literal original code, unreached by
any v2 logic). Malformed v2 `supported_efforts` (empty/non-array),
`effort_control` (value outside `flag`/`frontmatter`/`none`), and
`risk_effort_matrix` (non-string/out-of-vocabulary value) are each
validated eagerly against every model/matrix entry in the registry and
rejected fail-closed with a `MODEL_SELECTION_ERROR` diagnostic (AC-054).

The PowerShell twin could not name its new host-resolution parameter
`-Host`: PowerShell's `$Host` is a read-only automatic variable (verified
directly — a `param([string]$Host)` script throws `Cannot overwrite
variable Host because it is read-only or constant`), so the twin uses
`-HostName` instead (documented divergence, noted in-code and here). Every
new v2 field/value comparison in the twin uses case-sensitive operators
(`-ceq`/`-cne`/`-cin`/`-cnotin`/`-ccontains`); the dictionaries keyed by
UNTRUSTED registry/candidate strings (model name, risk key) were changed
from a bare `@{}` (PowerShell hashtables resolve string keys
case-INSENSITIVELY by default — verified directly: `@{low='x'}
.ContainsKey('LOW')` returns `True`) to ordinal
`[Collections.Generic.Dictionary[string,object]]`/`[string,string]`
instances, and the `risk_effort_matrix`/tier key-recognition gates were
changed from consulting the pre-existing (also case-insensitive)
`$matrix`/`$tierRank` hashtables to explicit case-sensitive array
membership checks. A dedicated mis-cased negative fixture pair (a
candidates-file entry naming `Anthropic/Haiku` against a registry
declaring `anthropic/haiku`; a registry declaring `effort_control:
{"claude-code":"Frontmatter"}`) proves both twins reject the mis-cased
value rather than silently aliasing it.

## Specification Differences

- **AC-054, malformed `risk_effort_matrix` category**: requirements.md's
  wording lists "missing a required key" as one malformed sub-variant, but
  REQ-002's own effort-resolution-priority clarification separately states
  that a `risk_effort_matrix` MISSING AN ENTRY FOR THE SUPPLIED RISK is an
  explicitly tolerated, non-error registry-content edge case (it triggers
  the `role-default`/`model-default` fallback, case 3/4 — exercised by
  this task's own TEST-011/TEST-012). Since AC-054 only requires ONE
  fixture per top-level category (not one per listed sub-variant --
  confirmed by both the AC text ("one fixture-driven case per
  malformed-field category") and acceptance-tests.md's TEST-054 row
  ("each independently causes... exit non-zero")), the `risk_effort_matrix`
  category's fixture uses the unambiguous "non-string value" sub-variant
  (`"high": 3` instead of `"high": "high"`) rather than "missing a key",
  which would have collided with the intentionally-tolerated
  missing-risk-entry case. This choice is implementation-visible in the
  validation code's comments and in the fixture's own name
  (`v2-malformed-risk-matrix.json`); a `risk_effort_matrix` that is
  present as a JSON value but not an object, or whose `escalation_bump`
  key is present with a non-boolean value, is ALSO rejected by the same
  validation branch, giving the "missing a required key" reading (if a
  future task interprets it as "missing `escalation_bump`" specifically)
  independent coverage without a dedicated fixture.
- **`--effort-control-*` flags**: the orchestrating instruction for this
  session listed "design が規定する --effort-control-* 系" as part of
  T-002's flag set. Re-reading design.md's API/Contract Plan confirms
  `--effort-control-main`/`--effort-control-reviewers` belong to T-004's
  `emit-run-record.sh` (design.md:472, "New decision... whether T-004's
  `--effort-applied-*` flags... require an accompanying
  `--effort-control-*` flag"), not to T-002's `select-agent-model`. T-002's
  own Planned Files (tasks.md) list only `select-agent-model.sh`/`.ps1`,
  `tests/agent-model-routing.tests.sh`, the ADR, and `CHANGELOG.md` — no
  `emit-run-record.sh` reference anywhere. No `--effort-control-*` flag was
  added to `select-agent-model`; this is a deliberate scope-boundary
  decision following tasks.md as the authoritative source, not an
  oversight, and is called out here so it is not mistaken for a gap by a
  reviewer holding the same (incorrect) initial reading.
- Design.md's older API/Contract Plan prose for T-002 (predating the
  round-2 remedy) states `--role` "provid[es] a default `--requested-effort`"
  and lists only 4 `effort_source` values (missing `risk-matrix` and
  `role-default`). This task follows requirements.md's REQ-002 round-2
  clarification instead (the authoritative, later text, per this task's
  own instructions): `--role`'s effort component seeds a
  `role_defaults[role].default_effort` FALLBACK under `matrix` only,
  attributed as its own `effort_source: "role-default"` value, never a
  `--requested-effort` substitute; the 5-value `effort_source` enum
  (`requested`/`risk-matrix`/`role-default`/`model-default`/`welded`)
  matches AC-012's explicit list.

  "computed AFTER the bump" requirement) is not the same value as its
  CANDIDATES-FILE-declared effort (needed for the tiebreak's original
  "prefer the candidate's own cheaper declared variant" meaning) — see
  AC-008's sonnet-over-codex resolution in the Summary for why conflating
  the two would have broken that specific, spec-mandated tiebreak outcome.

  hashtable resolves string keys case-INSENSITIVELY by default
  (`@{low='x'}.ContainsKey('LOW')` -> `True`) before deciding to switch the
  v2-branch's `$registered`/`$hostControlMap`/`$riskMatrix` variables to
  ordinal `Dictionary` instances; the pre-existing v1-branch `$registered`
  hashtable was deliberately left untouched (same case-insensitive
  behavior it always had) to avoid any risk of altering v1 output.
