## Summary

Added `--effort <e>` / `-Effort <e>` to `run-panelist-gpt.sh`/`.ps1`,
forwarded verbatim to the `codex` CLI invocation alongside the existing
`--model` flag (`run-panelist-gpt.sh:146` in the pre-T-006 file) only when
supplied -- omitted entirely preserves the exact pre-T-006 invocation
byte-for-byte (Breaking API: no). Added a matching `--effort` /
`-Effort` pass-through to `prepare-panelist-input.sh`/`.ps1`: since that
script prepares ONE shared sanitized bundle consumed by every panelist
vendor and never itself invokes a vendor CLI, the supplied value is
threaded through by being echoed on a second stdout line (`effort=<e>`,
after the existing `input_digest` line) for the caller to lift into
`run-panelist-gpt --effort <e>` in its own next step -- omitted entirely
preserves the exact pre-T-006 single-line stdout output.

**Injection rejection (AC-052, security-spec.md B3)**: both scripts
validate `--model`/`--effort` BEFORE assembling any `codex` invocation.
Shape checks (whitespace, leading `-`/`--`, `;`) apply to `--model`, since
neither script has its own `--registry` to enumerate valid model names
against (that deeper check is the caller's responsibility -- the caller is
expected to source `--model`/`--effort` only from `select-agent-model
--host codex-cli` output, never from unsanitized task/spec text, per B3's
"internal source only" trust boundary). `--effort` gets the same shape
checks PLUS an enumerated-vocabulary check against
`{low, medium, high, xhigh}`. A rejected value exits non-zero (2) with a
diagnostic naming the specific violated category, before the `_codex_cmd`
CLI-availability check is even reached -- so zero `codex` invocations occur
either way (proven executably in the test suite via a stub `codex` that
would leave an invocation marker if ever reached; the marker's absence
after every rejection case is the mechanical proof, not merely "the exit
code was non-zero").

**Codex-host startup wiring (AC-037/AC-038)**:
`plugins/sdd-quality-loop/skills/quality-gate/SKILL.md` step 8 (the
critical-review evaluator launch) now documents, for the Codex path: (1)
resolve `--model`/`--effort` from `select-agent-model.sh --host codex-cli
--role <role> --json` against the live v2 registry -- verified against the

- **No new script file for the "Codex-host startup path" itself**:
  requirements.md/design.md describe wiring "the Codex-host evaluator and
  investigator startup paths" (REQ-006) and tasks.md's Scope says this
  task implements "the startup-path wiring," but tasks.md's own Planned
  Files list for T-006 names only `plugins/sdd-quality-loop/skills/quality-gate/SKILL.md`
  for this surface -- no new script file. This task therefore implements
  the wiring as a documented procedure in SKILL.md (the composition
  recipe: resolve via `select-agent-model --host codex-cli`, cross-check
  against the `.toml` reference comments, reject out-of-vocabulary values,
  launch `codex`) and proves that EXACT recipe operationally in the new
  test suite (calling the real `select-agent-model.sh`/`.ps1` against the
  real registry, then feeding the result into `run-panelist-gpt`) rather
  than inventing a second, independent script that could silently drift
  from what SKILL.md documents. This interpretation is the only one
  consistent with T-006's own Planned Files list.
- **The render/selector cross-check (AC-038) is implemented as a
  documented procedure + test-suite-embedded verification function, not a
  new production-code flag on `run-panelist-gpt.sh`**: design.md's
  API/Contract Plan section for `run-panelist-gpt.sh`/`.ps1`'s `--effort`
  addition (the ONLY T-006 entry in that section) describes exactly one
  new flag, `--effort`, with no cross-check flag or parameter. Adding an
  undocumented second flag to that script would have exceeded design.md's
  literal contract for it. The cross-check logic instead lives in
  `tests/run-panelist-effort.tests.sh`/`.ps1` as a `cross_check_toml`/
  `Test-CrossCheckToml` function operationalizing the same comparison
  SKILL.md documents, tested with a GREEN (real, in-sync content) and RED
  (mutation-based negative self-check) pair per design.md's Test Strategy
  point 5's "argv-composition proof, not live-LLM proof" framing.
- **`prepare-panelist-input`'s `--effort` threading is a second stdout
  line, not a direct function call to `run-panelist-gpt`**: design.md's
  API/Contract Plan states the value is "forwarded verbatim as
  `run-panelist-gpt --effort <e>`," which could be read as a direct
  invocation. `prepare-panelist-input.sh`/`.ps1` prepares ONE shared
  sanitized bundle consumed by every panelist vendor (Claude via Agent
  tool, GPT/Gemini via separate CLI runners per
  `cross-model-verify/SKILL.md`'s existing Step 1 vs. Step 3 separation)
  and has never itself invoked a vendor CLI -- doing so now would be an
  architectural change outside this task's Planned Files (only `--effort`
  pass-through is listed for these two scripts). "Forwarded verbatim"
  is implemented as: the value is echoed on stdout for the caller (the
  skill/operator already running Step 1 then Step 3 in sequence) to lift
  unchanged into `run-panelist-gpt --effort <e>` -- proven end-to-end in
  TEST-036 by literally performing that hand-off inside the test.
- **`--effort` shape validation on `prepare-panelist-input.sh`/`.ps1` was
  intentionally NOT added**: AC-052 names only "`run-panelist-gpt.sh`/`.ps1`

## Working Notes

- **A real `codex` CLI is installed on the authoring machine** (discovered
  mid-session, at multiple `PATH` locations: `~/.nvm/versions/node/.../bin/codex`,
  `/opt/homebrew/bin/codex`, `~/bin/codex`, plus a shell alias
  `codex=codex-sync`). An early manual smoke test of the freshly-edited
  `run-panelist-gpt.ps1` WITHOUT a `PATH` override actually launched it
  (observed: `codex` rejected the unrecognized `--effort` flag with its
  own real usage text) -- confirming the script's argv assembly worked,
  but also confirming that EVERY test invocation in the new suite must
  fully override (never prepend to) `PATH`, which the suite does via a
  wrapper function (`run_gpt()` / `Invoke-RunGpt`) for every call site, a
  fact TEST-040 checks mechanically (call-site count vs. wrapper-usage
  count).
- **A stray `specs/f/verification/` directory was accidentally created**
  during the same early, PATH-unguarded manual smoke test (feature name
  `f`, default `--spec-root`, resolved relative to the real repository
  CWD). This tripped `tests/validate-repository.ps1`'s workflow-state
  check (`registry-unregistered-directory`) when re-run later in this
  session. Removed (`rmdir specs/f/verification && rmdir specs/f`) before
  any commit; confirmed absent from `git status` and from both commits'
  diffs -- this directory was never staged or committed, purely a
  same-session, pre-commit artifact of manual exploration.
- **The stub `codex`'s recorded-argv file is one token per line**
  (`printf '%s\n' "$@"` / `foreach ($a in $args) { Add-Content ... }`),
  not "flag value" pairs per line -- an early version of the test suite's
  assertions assumed the latter and failed with a correctly-composed argv
  (confirmed by manual inspection: the recorded file's content was right,
  the assertion's line-indexing was wrong). Fixed by adding an
  `argv_flat()`/`Get-ArgvFlat` helper that joins the file into one
  space-delimited line before substring/exact-match assertions -- safe
  because no accepted `--model`/`--effort` value may itself contain
  whitespace (AC-052's own rejection rule guarantees this precondition).
- **A self-referential grep false positive**: TEST-040's "no live network
  client" self-check originally grepped the whole suite file for
  `curl |wget |api\.openai\.com|...`, which always matched the check's OWN
  line (the pattern literal is textually present in the file that defines
  it). Fixed with a `SELF-CHECK-PATTERN` trailing-comment marker excluded
  from the scanned content before the pattern search runs.
- **SKILL.md's Markdown line-wrap broke exact-substring greps**: the
  first draft of the injection-rejection documentation wrapped
  `"command-separator shape"` and `"no `codex` invocation attempted"`
  across two physical lines, which `grep -F`/`Select-String` (line-based)
  cannot match as a single string. Reformatted the prose so each
  multi-word phrase this task's own tests check for stays on one physical
  line.
- **Windows-path design for the `.ps1` twin's stub `codex`**: since
  Windows' `Start-Process -FilePath "codex"` needs an OS-resolvable
  executable extension (`.ps1` is not one, by default, for direct
  process-launch resolution the way `.cmd`/`.bat` are), the `.ps1` suite's
  stub is a `codex.cmd` wrapper that delegates to a `codex-worker.ps1`
  script via the RESOLVED absolute path to `pwsh` (captured once, before
  any `PATH` override, into `$powerShellHost`) -- a bare `"pwsh"` name
  would be unresolvable once `Invoke-RunGpt`'s `PATH` override is in
  effect for the grandchild process. This branch (`else` of
  `$IsLinux -or $IsMacOS`) follows the same `codex.cmd`-on-Windows pattern
  reasoning as `tests/collection-layer.tests.ps1`'s established stub
  approach but was only exercised via the `$IsMacOS` branch in this
  sandbox (macOS); the Windows branch is analogous in structure and design
  but unexercised in this session -- flagged for the quality gate's own
  verification if a Windows CI runner is available to it.
