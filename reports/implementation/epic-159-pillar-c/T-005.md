## Specification Differences

- **Most of the TEST-027..034 case list was ALREADY implemented pre-T-005**:
  requirements.md's REQ-005 and acceptance-tests.md's own Test Target
  column both frame TEST-028..033 as "mirrors TEST-007/008/009/011"
  against T-002's Phase-1-scoped smoke, which `tests/agent-model-routing.tests.sh`
  already carried in full before this task started. This task's genuinely
  new `.sh`-side work is therefore narrower than "author a full new case
  list" reads at first glance -- it is (1) port the ENTIRE existing case
  list 1:1 onto a brand-new `.ps1` twin (the larger share of this task's
  actual work, ~450 lines of new PowerShell), (2) add the two cases neither
  twin had before (TEST-027, TEST-034), and (3) close the `test.yml`
  registration gap. This reading was cross-checked against design.md's
  Test Strategy note 7, which explicitly frames the three-part AC-027
  check as "generalizing" a pattern rather than describing wholly new
  case authorship, and is consistent with tasks.md's own Scope wording
  ("Author `tests/agent-model-routing.tests.ps1` porting the full case
  list 1:1 from the extended `.sh`").
- **`select-agent-model.ps1` JSON key order differs from the `.sh`+`jq`
  twin's**: confirmed empirically (see Summary). The `.ps1` twin therefore
  asserts JSON cases via `ConvertFrom-Json` + per-property comparison
  rather than the `.sh` twin's literal alphabetical-order JSON string
  comparison for TEST-006's legacy-positional case. This is a
  test-methodology difference only -- both twins assert the identical set
  of field values; neither the `.sh` nor the `.ps1` selector script itself
  was edited (Out of Scope: "`select-agent-model.sh`/`.ps1`'s own
  implementation (T-002)").
- **`exit 0`/`exit 1` inside an in-process (`&`) `.ps1` invocation does not
  terminate the calling test suite**: verified empirically before writing
  the full twin (three standalone probe scripts) that PowerShell's `&`
  call operator runs the target script in a nested scope where `exit`
  terminates only that scope, not the host process -- confirming
  `tests/loop-escalation.tests.ps1`'s own documented assumption
  ("`select-agent-model.ps1`'s successful (non-BLOCKED) path ... never
  calls an explicit `exit N`") does not need to hold for the BLOCKED-path
  cases this suite specifically exercises, and that in-process invocation
  (rather than spawning a `pwsh -File` subprocess per call, as
  `check-risk.ps1`/`check-terminal-tier-resume.ps1` calls also do NOT need
  to, contrary to an initial assumption) is safe and faster throughout.
- **AC-027(c) is not asserted true by this report** (see Test Evidence,
  item 3): it is a property that holds only after a human applies the
  staged `test.yml` candidate, per the task's own explicit instruction
  ("(c) は human-copy 適用後に成立する性質としてレポートに記載").
