## Summary

Extended the four installer/uninstaller scripts to place and register the
`sdd-forge-mcp` MCP server, following the acceptance-first workflow required
by this task (Required Workflow: acceptance-first). Test scenarios were
written first, run to confirm they failed for the right reason, and only then
was the installer/uninstaller logic implemented.

Key design points:

- **MCP payload copy is independent of Git tracking state.** Unlike plugin
  files (copied via `git ls-files` for local sources / `REQUIRED_PATHS` for
  the manifest check), `mcp/sdd-forge-mcp/dist/index.js` and `package.json`
  are copied directly from the filesystem (`cp -R`/`Copy-Item`) regardless of
  whether they are Git-tracked. This was intentional at the time this task
  was implemented: `mcp/sdd-forge-mcp/` was then entirely untracked (the
  `dist/` bundle was committed later, by task T-007, per
  `specs/sdd-forge-mcp/tasks.md` line 333 "dist/ Ńü«ÕłØÕø×Ńé│Ńā¤ŃāāŃāł’╝łT-001ŃĆ£T-005
  / T-009ŃĆ£T-011 Ńü«µłÉµ×£ńē®ŃéÆŃāÉŃā│ŃāēŃā½’╝ē"). Only `dist/` and `package.json` are
  copied by `place_mcp_servers`/`Install-McpServerPayloads`; `node_modules/`,
  `src/`, and `tests/` are explicitly excluded from that function.
  **Update (post-Done quality gate fix):** once `mcp/` became Git-tracked
  (63 files, committed after this task's original Done), the *general*
  staging step (which stages all `git ls-files` output for local sources,
  independently of `place_mcp_servers`) started copying the entire `mcp/`
  tree unconditionally ŌĆö bypassing `--skip-mcp` / `--mcp` selection and the
  Node >= 20 gate entirely, since that general step has no knowledge of any
  of those flags. See "Post-Done quality gate fix" under Working Notes for
  the fix (exclude `mcp/` from general staging in both the local-source and
  archive-source code paths, in both install.sh and install.ps1).
- **Placement vs. registration split mirrors the existing plugin pattern.**
  Placement happens regardless of `--target` (like plugin file staging);
  only registration (`claude mcp add` / Codex config.toml edit) is scoped by
  `--target`. This matches `design.md`'s statement that plugin registration
  presence, not placement, is what `--target` controls.
- **Node >= 20 check is MCP-only and non-fatal.** If `node` is absent or its
  major version is `< 20`, a warning is printed and MCP placement/
  registration is skipped, but plugin installation proceeds unaffected
  (exit code remains 0). Implemented via `node --version` major-version
  parsing in both bash (`node_version_ok`) and PowerShell
  (`Test-NodeVersionOk`).
- **Codex registration never creates a new config.toml.** Per the task's
  explicit requirement, if `~/.codex/config.toml` (or
  `$SDD_CODEX_HOME/config.toml` under test) does not exist, Codex MCP
  registration is skipped with a warning; the file is never created by the
  installer.
- **Idempotent marker-block registration.** Each MCP gets a delimited block
  in config.toml:
  `# >>> <name> (managed by sdd-forge installer; do not edit by hand) >>>` /
  `# <<< <name> <<<`. Re-running install strips any prior block for that name
  before appending a fresh one (via `awk` in bash, a line-filter loop in
  PowerShell), so re-registration is idempotent and unrelated config.toml
  content is preserved untouched.
- **Uninstall mirrors placement/registration split, all best-effort.**
  `claude mcp remove <name>` (best-effort, matching `try_plugin_command`
  semantics), Codex config.toml marker-block removal (no-op if the file or
  block is absent), and `INSTALL_ROOT/mcp/<name>/` removal (skipped under
  `--keep-files`/`-KeepFiles`, but otherwise idempotent ŌĆö absence is
  success). `--mcp <list>` on uninstall selects which MCP names are
  processed, following the same `VALID_MCPS`/`ValidateSet` validation as
  install.
- **bash 3.2 compatibility preserved.** No associative arrays, no `mapfile`;
  all new bash logic uses indexed arrays, `case`/`for`, and `awk` (already a
  POSIX baseline tool used elsewhere in the scripts), consistent with the
  existing installer style.

**Post-Done quality gate fix (mcp/ staging leak + bash 3.2 `--mcp ""` crash)**

Purpose: address two findings from an independent quality-gate evaluation
performed after this task's original Done, both reproduced before any fix
was applied.

Finding 1 (Critical) ŌĆö `mcp/` placed unconditionally by general staging:
`mcp/sdd-forge-mcp/` was untracked when this task was originally
implemented (see the Summary's Git-tracking note above), so the general
staging step's `git ls-files` enumeration for local sources never included
it ŌĆö only `place_mcp_servers`/`Install-McpServerPayloads` placed the
`dist/`+`package.json` payload, correctly gated by `--skip-mcp` / `--mcp` /
the Node >= 20 check. After this task's Done, `mcp/` was committed to Git in
full (63 tracked files, including `src/`, `tests/`, `package-lock.json`,
etc.), which silently changed what the *general* staging step does: it now
copies the entire `mcp/` tree into `INSTALL_ROOT` regardless of
`--skip-mcp`, `--mcp` selection, or the Node version, because that step has
no awareness of any MCP-specific flag ŌĆö it only ever inspected Git tracking
state. Verified with `git ls-files mcp/ | wc -l` ŌåÆ 63, and by installing
before the fix: `--skip-mcp` still left `mcp/sdd-forge-mcp/{src,tests,...}`
present at `INSTALL_ROOT/mcp/` (bypassing the flag entirely; only
`place_mcp_servers`'s narrower `dist/`+`package.json` copy was still gated
correctly).

Fix: excluded the `mcp/` prefix from general staging in both source-input
code paths, in both installers:
- `install.sh`, local-source path (`git ls-files -z` enumeration): changed
  to `git -C "$SOURCE_ROOT" ls-files -z -- . ':!mcp/**'` (a Git pathspec
  exclusion, evaluated by Git itself ŌĆö no bash-version-dependent string
  matching involved).
- `install.sh`, archive-source path (top-level directory copy loop): added
  `mcp` alongside `.git` to the `case "$(basename "$entry")"` skip list.
- `install.ps1`, local-source path (`git ls-files` enumeration): same
  `':!mcp/**'` pathspec appended to the `git ls-files` invocation.
- `install.ps1`, archive-source path (`Get-ChildItem` copy loop): added
  `-and $_.Name -ne "mcp"` to the existing `.git`-exclusion filter.

This makes `place_mcp_servers`/`Install-McpServerPayloads` the sole owner of
`mcp/` placement again, matching this task's original design (placement
happens regardless of `--target`, but only when `--skip-mcp` is not set,
the requested name is in `--mcp`'s list, and Node >= 20 is present).
Verified post-fix: a full local-source install places only
`mcp/sdd-forge-mcp/{dist/index.js,package.json}` (no `src/`, `tests/`,
`node_modules/`); `--skip-mcp` leaves no `mcp/` directory in `INSTALL_ROOT`
at all.

directly ŌĆö `/bin/bash install.sh --mcp ""` (macOS system bash, 3.2.57)

before the `IFS=',' read -ra` call, so an empty `--mcp` value is rejected as
an invalid name ŌĆö consistent with how every other invalid `--mcp`/`--plugins`
value is already handled ŌĆö without ever attempting to expand a
possibly-unset array. Verified under both `/bin/bash` (system 3.2.57) and
`bash` (5.3.9): both now exit 1 with the "Invalid MCP name: (empty)"
message, no "unbound variable" text in the output.

### cycle-2 quality gate fix (uninstall.sh `--mcp ""` unbound variable)

**Finding (Major, quality gate cycle-2):** `uninstall.sh --mcp ""` crashed
under bash 3.2 with `MCP_SELECTION[@]: unbound variable` (at the
`for m in "${MCP_SELECTION[@]}"` loop, uninstall.sh:112 in the pre-fix
version), for the exact same reason as install.sh's Finding 2 above:
`IFS=',' read -ra MCP_SELECTION <<< "$MCP_LIST"` on an empty `$MCP_LIST`
produces a zero-element array that bash 3.2 treats as unset under `set -u`.
`install.sh` already carried a guard for this
(`[[ -z "$MCP_LIST" ]]` early-exit before its `read -ra`, install.sh:93-96),
but that guard was never mirrored into `uninstall.sh` during the original
cycle-1 fix round ŌĆö the cycle-1 Working Notes above incorrectly stated
uninstall was unaffected by "both defects," when in fact only Finding 1 (the
staging leak) did not apply; Finding 2's root cause was present verbatim in
`uninstall.sh` and simply untested.

**Fix:** added the identical `[[ -z "$MCP_LIST" ]]` early-exit guard to
