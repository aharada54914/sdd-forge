## Outcome

Added `plugins/sdd-domain/skills/domain-reverse/SKILL.md`, an internal
(`disable-model-invocation: true`, `user-invocable: false`) skill that:

1. Invokes `investigate-codebase` (`/sdd-bootstrap:investigate-codebase feature
   <target>`, matching the invocation pattern `sdd-bootstrap-interviewer` uses)
   against the target project.
2. Reads the resulting `specs/<feature>/investigation.md`.
3. Converts its `## Findings` (INV-NNN rows), `## Open Questions`, and
   `## Risks` into a candidate domain-model seed via a documented,
   category-driven "Finding-to-Seed Mapping" table (screen/api ->
   candidate_contexts; api's state-changing verbs -> candidate_event_hints;
   business-rule/data -> candidate_terms and candidate_aggregate_hints;
   dependency/test-coverage excluded; pattern -> candidate_terms; constraint
   -> candidate_open_questions).
4. Documents the exact candidate-seed shape (five required sections:
   `candidate_contexts`, `candidate_terms`, `candidate_event_hints`,
   `candidate_aggregate_hints`, `candidate_open_questions`) that
   `domain-interviewer` (T-002, a separate, later task) will consume.

`domain-reverse` never writes into `domain/`; the seed is handed to
`domain-interviewer` in-session (optionally persisted at
`specs/<feature>/domain-reverse-seed.md` for context-fork continuity only).
It has no build-order dependency on T-001/T-002/T-004 — confirmed by the
current repository state, where `plugins/sdd-domain/` exists only as empty
plugin-manifest scaffolding (T-001 in progress) and no other sdd-domain skill
files exist yet.

Two intermediate failures were found and fixed before the green run above:

- An `unevidenced claims` phrase in the SKILL.md wraps across a Markdown line
  break; the initial single-line regex did not match. Fixed by relaxing the
  assertion to the still load-bearing substring `unevidenced claims`.
- Pester 3.4.0's `Should Contain` is a **file-content** assertion
  (`Get-Content $file | Should Contain $pattern`), not a Pester-5-style
  collection-membership assertion. Using it against an in-memory array threw
  `Get-Content : Cannot find path ... candidate_contexts` because Pester
  tried to treat the array's string form as a file path. Fixed by replacing
  those assertions with the `-contains` PowerShell operator wrapped in
  `Should Be $true` / `Should Be $false`, which is both Pester-3-safe and
  forward-compatible with later Pester versions.

## Candidate Seed Shape (for the T-002 implementer)

Documented in full in `domain-reverse/SKILL.md` under "Candidate Seed Shape."
Summary of the exact, load-bearing contract:

```markdown
# Domain Reverse Seed: <feature>

Seed-Source: reverse (investigate-codebase, <feature>)
Generated-From: specs/<feature>/investigation.md

## candidate_contexts
- name: <kebab-case>          # matches domain-contract.v1 boundedContext.name
  rationale: <string>
  evidence: [INV-NNN, ...]

## candidate_terms
- term: <string>
  definition_hint: <string>
  evidence: [INV-NNN, ...]

## candidate_event_hints
- name: <PastTense string>
  kind: event | command
  rationale: <string>
  evidence: [INV-NNN, ...]

## candidate_aggregate_hints
- name: <PascalCase>          # matches domain-contract.v1 aggregate.name
  rationale: <string>
  invariant_hint: <string>
  evidence: [INV-NNN, ...]

## candidate_open_questions
- question: <string>
  evidence: [INV-NNN, ...]
```

Binding notes: all five sections are always present (rendered `(none found)`
when empty — never omitted); `evidence` is always an `INV-NNN` id array (the
join key back to `investigation.md`, never a bare `file:line`); this
structure carries no `schema`/meta envelope and never validates against
`contracts/domain-contract.v1.schema.json` (only the later, human-confirmed
`domain/domain-contract.json` does).
