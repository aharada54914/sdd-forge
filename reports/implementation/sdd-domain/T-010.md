### Root cause

`tests/validate-repository.ps1` line 7 invokes
`plugins/sdd-quality-loop/scripts/check-workflow-state.ps1` as its very
first precondition. That script's `Test-Json -Json $RegistryText
-SchemaFile $Schema` call (line 228) uses `Test-Json`, a cmdlet **first
introduced in PowerShell 6.1** and never backported to Windows PowerShell
5.1. This host has only Windows PowerShell 5.1.22621.5697 (`pwsh` is not on
`PATH`, confirmed via `where pwsh` returning nothing, consistent with every
prior sdd-domain task's documented environment). Calling `Test-Json`
therefore throws `System.Management.Automation.CommandNotFoundException`
inside the script's own `try { $schemaOk = Test-Json ... } catch { $schemaOk
= $false }` block (lines 227-231), which is caught and treated as "schema
validation failed" — indistinguishable, from the caller's side, from a real
schema violation. `check-workflow-state.ps1` then calls
`Stop-WorkflowState "repository" "registry-schema" "registry entry violates
the bounded schema"` and exits 1, which `validate-repository.ps1` line 8-10
re-throws.

**Verified this is unconditional on this host, not caused by this task's
edits:**
- `Get-Command Test-Json -ErrorAction SilentlyContinue` returns nothing on
  this host; `$PSVersionTable.PSVersion` is `5.1.22621.5697`.
- A minimal reproduction (`Test-Json -Json '{"a":1}' -Schema
  '{"type":"object"}'`) throws the identical `CommandNotFoundException`,
  proving the failure has nothing to do with the real
  `workflow-state-registry.json` content or the `sdd-domain` entry added to
  it — it fires for *any* input, including the most trivial possible
  schema/JSON pair.
- Ran `check-workflow-state.ps1` after `git stash` (reverting every
  uncommitted change from this entire branch, including the pre-existing
  `sdd-domain` registry entry from an earlier task) and got the exact same
  `Test-Json`/`CommandNotFoundException` failure — confirming this blocker
  predates T-010 and is unrelated to any sdd-domain change on this branch.
- `reports/implementation/sdd-domain/T-001.md` independently documents
  hitting this exact same `Test-Json` PS6+-only limitation in its own test
  file and deliberately avoiding it ("PS5.1-safe `ConvertFrom-Json` only, no
  `Test-Json` (PS6+-only cmdlet, unavailable in this environment's Windows
  PowerShell 5.1)").
- No task before T-010 ever ran `validate-repository.ps1` end-to-end — it
  was explicitly out of scope for every one of T-001 through T-009 and
  T-011 (T-001: "`validate-repository.ps1` edits were made -- all explicitly
  out of scope per the task's Out of Scope section (T-010 owns
  `validate-repository.ps1`)"). This is therefore very likely the first time
  anyone has attempted to run this script to completion on this specific
  host/PowerShell-version combination.
