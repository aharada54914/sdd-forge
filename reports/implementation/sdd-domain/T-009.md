## Outcome

Extended `plugins/sdd-quality-loop/skills/workflow-retrospective/SKILL.md`
and `plugins/sdd-quality-loop/templates/retrospective-report.template.md`,
purely additively, so that when a project carries a `domain/` directory,
workflow-retrospective's metric roll-up includes domain-drift counts
(term-deviation count, boundary-violation count) sourced only from
`check-domain-conformance` findings already recorded in
`reports/quality-gate/*.md` reports — no new evidence-collection path, and
no change to `check-domain-conformance` itself (T-008's territory,
untouched).

1. **New "Domain-drift metrics (when `domain/` exists)" subsection** added
   to `workflow-retrospective/SKILL.md`, inserted after the existing "Also
   derive dataset quality indicators" block and before the pre-existing
   "For spec-review, task-review, and impl-review metrics" section (an
   insertion point chosen to keep the file's existing three metric-family
   blocks — per-task metrics, dataset-quality indicators, review-gate
   metrics — contiguous and undisturbed). The subsection documents:
   - The precondition: only runs `domain/` exists; otherwise the subsection
     (and its report table) is skipped entirely — "do not emit a
     zero-filled table" is stated explicitly, so an unconditional zero row
     can never be mistaken for "zero drift observed."
   - The source: scan each retained quality-gate report (using the same
     `(task ID, Run ID)` de-duplication and retention rule already defined
     in rule 3 of "Deterministic artifact rules") for
     `check-domain-conformance WARN (<n> finding(s)):` or `... FAILED (<n>
     finding(s)):` blocks — the exact two block headers
     `check-domain-conformance.sh`/`.ps1` (T-008) can emit — and collect
     each block's `- <finding text>` lines.
   - The classification rule: each finding line is matched against the
     fixed set of five finding-message shapes T-008's script can actually
     produce (verified by reading both script twins in full; see
     "Verification against T-008's real output format" below):
     - **Term deviation**: `unrecognized term '...'`,
       `aggregate reference '...' not found in domain-contract.json
       aggregates`, `aggregate reference '...' has no
       domain/aggregates/....md card`.
     - **Boundary violation**: `Bounded-Context '...' not found in
       domain-contract.json`, `Bounded-Context lists two contexts (...)
       with no declared relation in context map` (the AC-015 undeclared-
       relation case).
     - Neither pattern (the script's two file-not-found input-error lines,
       `requirements.md not found: ...` / `design.md not found: ...`) is
       not counted in either bucket — these report a missing input file to
       the check, not a domain-drift finding.
   - **Term-Deviation Count**, **Boundary-Violation Count**, and a
     **Domain-Drift Trend** comparison against the previous retrospective's
     combined total, explicitly folded into the existing friction-pattern /
     WFI recurrence discipline (never draft a WFI from one period's counts
     alone).
2. **New "Domain Drift Metrics" report section** added to both the
   in-skill `## Output` Markdown template (inside `SKILL.md`) and the
   standalone `retrospective-report.template.md`, placed after `## Metrics`
   and before `## Friction Patterns` in both files. The table carries
   Term-Deviation Count, Boundary-Violation Count, and Combined
   Domain-Drift Count, each with This-Period / Previous-Period / Trend
   columns, matching the existing "Comparison With Previous Retrospective"
   table's column convention. A one-line note above the table states the
   section is included only when `domain/` exists, mirroring the SKILL.md
   subsection's own precondition.

### One intermediate failure found and fixed before the green run

Two failures appeared on the first run, both test-harness authoring issues
in the new test file, not defects in the SKILL.md/template edits:

1. A structural-contract assertion built its expected regex with a
   double-backtick (`` `` ``) around `domain/`, intending PowerShell
   escaping, but the actual template text wraps `domain/` in a single pair
   of backticks; the mismatched escaping made the `[regex]::Escape(...) +
   "..." + [regex]::Escape(...)` concatenation not match the real line.
   Fixed by asserting the literal phrase `"Include this section only when
   domain/ exists"` directly via `[regex]::Escape(...)` instead of
   constructing it from concatenated fragments.
2. `Get-DomainDriftCounts -ReportPaths @()` (the "no domain/ / empty
   report set" scenario) failed with a `ParameterBindingValidationException`
   because `[Parameter(Mandatory)][string[]]$ReportPaths` rejects an empty
   array as if it were unbound in this PowerShell 5.1/Pester 3.4.0
   combination. Fixed by adding `[AllowEmptyCollection()]` to the parameter
   declaration; the function's mandatory-but-possibly-empty contract now
   matches the actual test intent (an empty list of retained reports is a
   valid, real input — e.g., no quality-gate reports have run yet — not a
   caller error).

Both fixes were made directly in `tests/sdd-domain/drift-metrics.Tests.ps1`
and confirmed by the full green re-run shown above.

## Scope notes and deviations

- **How a domain-drift finding is "recorded" in a quality-gate report**: the
  task brief asked me to determine "the exact JSON path/shape
  workflow-retrospective would need to read from" by inspecting real
  quality-gate reports on disk. I read every existing quality-gate report
  under `reports/quality-gate/` (via `grep -rl "check-design-system\|
  check-domain-conformance"`) and found **zero** existing reports reference
  either check — both are new, warn-phase scripted gates (`check-design-
  system` landed in an earlier feature; `check-domain-conformance` landed
  in this same branch's T-008, immediately before this task) that have
  never actually fired in a real historical run yet. Quality-gate reports
  in this repository are free-form prose Markdown (confirmed by reading
  several real examples: `reports/quality-gate/T-006.md`,
  `20260629T111150Z.md`, `bootstrap-interviewer-enhancement-T-008.md`) —
  there is no fixed JSON schema or fixed "Gates run" table shape that every
  report follows; `quality-gate/SKILL.md` step 15 says "Create
  `reports/quality-gate/<timestamp>.md`" with no enforced sub-schema for
  gate output. Given that, this task documents the aggregation rule against
  the one shape that **is** deterministic and testable: the literal stdout
  block `check-domain-conformance` itself prints
  (`check-domain-conformance WARN|FAILED (<n> finding(s)):` followed by
  `- <finding text>` lines), which `quality-gate/SKILL.md`'s own prose
  ("Warn-phase: findings are recorded in the report ... but do not block")
  already implies is what gets pasted/quoted into the report body when the
  gate fires. This keeps the classification rule anchored to T-008's real,
  verified script output (never an invented JSON path) and makes the rule
  testable today even though no historical fixture with a real recorded
  finding exists yet in this repository.
- **No new evidence-collection path**: confirmed the edit never reads
  `domain/` directly, never invokes `check-domain-conformance`, and never
  adds a new "Must Read" source file to the Metrics Collection list — the
  only new read is a text-pattern scan over the retained quality-gate
  reports already listed in that section's existing bullet list.
- **Domain-Drift Trend and the friction-pattern discipline**: rather than
  inventing a new, separate escalation path straight to a WFI, the
  Domain-Drift Trend comparison is explicitly folded into the existing
  Improvement Loop's friction-pattern recurrence and confidence rules
  (Section 1 of `workflow-retrospective/SKILL.md`) — a sustained increase
  is a friction-pattern *candidate*, never an automatic WFI trigger from a
  single period, matching this repository's established "never overfit to
  one observation" convention used everywhere else in the skill.
