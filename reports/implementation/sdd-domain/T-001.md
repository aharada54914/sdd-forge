## Scope notes and deviations

- `Get-Content -Encoding Utf8` combined with a BOM-less `.ps1` file initially
  corrupted a non-ASCII Japanese fixture literal (`"ja": "..."`) during
  parsing on this Windows PowerShell 5.1 (ANSI-interpreted) environment,
  producing cascading quote-parsing errors well past the offending line. Per
  the feature's Global Constraints (PS1 hygiene), the fixture's `ja` field
  uses an ASCII placeholder value (`"chuumon"`) instead of literal Japanese
  text; this does not affect schema coverage since the `ja` field carries no
  format constraint in `contracts/domain-contract.v1.schema.json` beyond
  `type: string`.
- Two PowerShell/Pester-specific pitfalls were found and fixed during
  verification, not anticipated in the original plan: (1) a function
  returning an empty `System.Collections.Generic.List[string]` is unrolled to
  `$null` by PowerShell's output stream, so every call site wraps the result
  in `@(...)` to force array semantics; (2) Pester 3.4.0's legacy `Should
  Contain` operator checks file contents, not collection membership (that is
  `-contains`), so pattern-enum-membership assertions use the native
  `-contains` operator instead.
- `Set-StrictMode -Version Latest` throws on a missing property access for a
  `PSCustomObject` (unlike a hashtable, which returns `$null`), and the
  corrupt fixtures round-trip through `ConvertTo-Json`/`ConvertFrom-Json` into
  `PSCustomObject` specifically to exercise missing-field paths. A
  `Get-PropSafe` helper was added to the test file so every field read
  returns `$null` instead of throwing when a field is legitimately absent.
- No skill files, hook-guard changes, quality-gate wiring, or
  `validate-repository.ps1` edits were made — all explicitly out of scope per
  the task's Out of Scope section (T-010 owns `validate-repository.ps1`).
- `specs/sdd-domain/traceability.md` already lists T-001 against REQ-001,
  REQ-002, REQ-009 (via the corresponding requirement rows) from Phase 2
  planning; no edit was needed.
