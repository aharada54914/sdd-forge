## Target

Hook-guard extension: reject an agent-set `Domain-Model-Status` field of
`Approved`.

## Summary

`plugins/sdd-quality-loop/scripts/sdd-hook-guard.{js,py,ps1,sh}` are
self-protected: the running guard's own `PROTECTED_GATE_SUFFIXES` list
blocks direct agent tool-call `Edit`/`Write` to these four files during a
live agent session. Per the established procedure for this file class
(tasks.md Global Constraints; prior R-10 precedent), the finished versions
of all four files — plus an extended copy of the existing
`tests/guard-parity.tests.sh` fixture suite — were written to
`scratchpad-T006/` at the repository root instead of the real paths. **The
real protected files were not touched.** A human must apply the scratchpad
files with `cp` (exact commands in `scratchpad-T006/APPLY-INSTRUCTIONS.md`)
before this task can leave "Implementation Complete."

The new guard rejects a net increase in occurrences of
`Domain-Model-Status: Approved` when the edit targets (a path ending in)
`domain/context-map.md` and the edit is agent-authored. It mirrors the
existing `tasks.md` `Approval: Approved` guard exactly: same net-increase
counting logic (count occurrences before vs. after; deny only a net
increase), and the same `SDD_SUDO` bypass behavior (a valid signed token
permits the write). This is explicitly **not** the never-sudo-bypassable
WFI/Second-Approval pattern — per requirements.md's Security Boundaries and
the task's own Risk Rationale, this guard is one tier below the
second-approver control, the same class as the routine tasks.md approval
gate.
