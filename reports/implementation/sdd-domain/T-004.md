## Outcome

Added `plugins/sdd-domain/skills/domain-model/SKILL.md`, the plugin's one
public entry skill (frontmatter omits `user-invocable: false`, carries
`disable-model-invocation: true`), that:

1. Documents `/sdd-domain:domain-model [new|update|reverse]` mode dispatch,
   with `new` as the default (bare invocation or a bare seed argument both
   mean `new`).
2. Routes `new` mode directly to `domain-interviewer` (T-002) with whatever
   seed was supplied (or none).
3. Routes `reverse` mode to `domain-reverse` (T-003) first, then hands its
   candidate seed off to `domain-interviewer`, relaying any seed-failure
   error from `domain-reverse` instead of falling through with fabricated
   content.
4. Implements `update` mode's own targeted re-interview algorithm directly
   (the one mode this skill does not wholesale-delegate): identify the
   edited stage `N` (asked of the human, not git-diffed), snapshot upstream
   stages `1..N-1`, re-run stage `N` and every downstream stage `N..7` in
   confirmation mode (existing artifacts re-presented for human approval,
   never blindly regenerated) via per-stage `domain-interviewer` calls,
   verify upstream stages remain byte-identical against the snapshot,
   regenerate `domain/domain-contract.json`, and reset
   `Domain-Model-Status` to `Pending` in `domain/context-map.md` regardless
   of its prior value (documents the `N = 4` convergence case, where the
   status reset and stage 4's own confirmation-mode write are the same
   file operation).

`domain-model` never writes any `domain/` artifact itself except the one
`Domain-Model-Status:` line reset in `update` mode (folded into the single
`context-map.md` write that mode already performs); every other write
remains `domain-interviewer`'s sole responsibility, and `domain-model` never
sets `Approved` under any circumstance.
