## Design.md Assumptions verification: does the tasks.md-mediated consent gate really not apply to `domain/`?

**Confirmed accurate, with the fallback path independently verified by real
execution, not merely by reading source.**

design.md's Assumptions section (lines 166-181) cites
`prepare-panelist-input.sh:27-59` (argument parsing accepts any non-empty
`--task`/`--input`, no existence lookup) and states the residual risk that
the scripts' consent gate (`prepare-panelist-input.sh:91-95` per that
citation; the actual line range in the current file is 89-95 for the
consent-gate comment block, with the tasks.md-flag scan at lines 96-117 and
the `SDD_SUDO` fallback at lines 119-219) reads `tasks.md` for
`Cross-Model: enabled`, which `domain/` does not produce, and concludes
`domain-review-loop` must invoke the underlying scripts directly under human

Reading the actual script confirms this exactly: `prepare-panelist-input.sh`
scans `$tasks_file` (default `<spec-root>/<feature>/tasks.md`) line-by-line
for a `## <task_id> ...` heading and a `Cross-Model: enabled` line inside
that section (lines 96-117); if the file does not exist or the section is
never found, `consent_kind` stays empty and control falls through
unconditionally to the `SDD_SUDO` check (line 120 onward). There is no third
path and no special-case for a project-level tree with no `tasks.md`.

This was then verified **by real execution**, not just by reading source: a
fixture project root with a `domain/` tree and no `specs/sdd-domain-model/tasks.md`
was built, and `prepare-panelist-input.sh --task DM-001 --feature
sdd-domain-model --input domain/` was run against it with no `SDD_SUDO` file


confirming both that the tasks.md path is checked (and fails, since the file
does not exist) and that the `SDD_SUDO` path is the only remaining option
(see "Test Evidence" below for the corresponding Pester assertions and a

**Conclusion: design.md's Assumptions citation was accurate as written; no

## A second, independently discovered environment finding (not a script defect)

While building real-execution test fixtures, `prepare-panelist-input.sh`
initially failed with `sanitization failed` (exit 2) even after a valid
`SDD_SUDO` token was in place and consent was granted. Root-caused to two
compounding, pre-existing host conditions, neither introduced by this task:

1. **The `python3` name on this host's `PATH` resolves to a non-functional
   Windows Store app-execution-alias stub**
   (`...\WindowsApps\python3.exe`), which prints the single word `Python`
   and exits nonzero instead of running any script, regardless of caller
   (confirmed identically whether invoked from a plain shell or from
   `bash.exe` launched via PowerShell's `&` operator). A real, working
   interpreter is installed at
   `...\Programs\Python\Python314\python.exe`, but neither
   `prepare-panelist-input.sh` nor `check-cross-model.sh` (correctly, for a
   portable script) provide any override for the bare `python3` command
   name. Worked around for test purposes only by building a small PATH-shim
   directory containing a copy of the real interpreter named `python3.exe`
   and prepending it to `PATH` for the duration of each `bash.exe` child
   process invoked by the test file — this does not touch the system `PATH`
   or either script under test.
2. **PowerShell 5.1's `Set-Content -Encoding UTF8` always writes a UTF-8
   byte-order-mark.** Fixture Markdown/JSON files for the sanitizer were
   initially written this way; `prepare-panelist-input.sh` reads them with
   `cat` (raw bytes, BOM preserved) and pipes them into its python3
   sanitizer subprocess, whose default stdout encoding on this
   Japanese-locale Windows host is the ANSI code page (cp932) rather than
   UTF-8. cp932 cannot encode the BOM character (U+FEFF), so the
   sanitizer's own `sys.stdout.write(text)` raised `UnicodeEncodeError` and
   the script correctly reported `sanitization failed` and exited 2 — this
   is the *scripts working correctly*, surfacing a real fixture-authoring
   defect (BOM-prefixed input), not a bug in `prepare-panelist-input.sh`
   itself; a UTF-8-locale CI host would not hit this. Fixed by writing all
   fixture files with `[IO.File]::WriteAllText(path, text,
   [Text.UTF8Encoding]::new($false))` instead of `Set-Content -Encoding
   UTF8`, eliminating the BOM.

Both are recorded in the test file's own comments (`Set-Utf8NoBom`'s
docblock and the file-header "Environment constraint" note) so a future
reader does not mistake either for a defect in the scripts under test. This
is the same class of finding as `sdd-forge-windows-jq-gate-scripts` (project
memory) for `jq`, now extended to `python3` app-execution-alias shadowing
and PowerShell 5.1's default BOM behavior.

Neither finding required any change to `prepare-panelist-input.sh` or
`check-cross-model.sh` — both scripts are out of this task's scope (T-011
adds no script changes, per its own Goal: "no script change is required")
and neither was edited.
