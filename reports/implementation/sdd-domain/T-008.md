## Outcome

Added `plugins/sdd-quality-loop/scripts/check-domain-conformance.sh` and its
`.ps1` twin, a deterministic, warn-phase-default quality-gate script that
replicates `check-design-system.sh`/`.ps1`'s exact CLI and exit-code
conventions:

1. **Skip-when-absent**: when `<project-root>/domain` does not exist, prints
   `check-domain-conformance skipped: no domain/ directory.` and exits 0. A
   second skip case (missing `domain/domain-contract.json`, or a contract
   that fails to parse as JSON on the PowerShell twin) also exits 0 with a
   one-line note, matching the never-block guarantee already established by
   `domain-sync` (T-007) for the same class of precondition failure.
2. **Warn-phase default**: findings (unrecognized `Bounded-Context:` names,
   unrecognized structured-field terms, aggregate references absent from
   `domain/aggregates/`, and undeclared two-context relations) print as
   `check-domain-conformance WARN (<n> finding(s)):` and exit 0.
3. **`SDD_DOMAIN_ENFORCE=error` escalation**: the same findings print as
   `check-domain-conformance FAILED (<n> finding(s)):` and exit 1, mirroring
   `SDD_DESIGN_SYSTEM_ENFORCE`.
4. **Bad invocation**: a missing/non-existent `<project-root>` always exits 1
   regardless of warn/enforce mode.
5. **AC-015 two-context relation check**: when `Bounded-Context:` lists
   exactly two contexts, the script reads `domain-contract.json`'s
   `relations[]` array and passes only when a `from`/`to` pair (in either
   direction) matches the two named contexts; otherwise it records a warn
   finding naming both contexts and "no declared relation in context map."
6. **Term-matching v1 scope** (design.md Assumptions / OQ-R1): term
   conformance matches only exact canonical terms from
   `domain-contract.json`'s `contexts[].terms[].canonical` against
   structured-field usage — a `[[term:Name]]` marker inside a Markdown
   heading line in `requirements.md`. No lexical-variant or synonym matching
   is implemented, matching the documented v1 scope exactly.

CLI shape (positional args for `.sh`, named params for `.ps1`, matching
`check-design-system`'s own positional-vs-named split):

- `.sh`: `check-domain-conformance.sh <project-root> [<requirements-md>] [<design-md>]`

### One intermediate failure found and fixed before the green run

The `Bad invocation` test (`ProjectRoot` does not exist) initially failed with
a `RemoteException` thrown from inside the test helper function
`Invoke-CheckDomainConformance`, not a genuine script defect. Root cause: the
test file sets `$ErrorActionPreference = "Stop"` at file scope (matching this
repository's existing Pester convention); the fixture script's
`check-domain-conformance: project root not found: ...` message is written
via `Write-Error`, and when the external `powershell.exe -File ...` process's
stderr is captured with `2>&1` inside a caller whose `$ErrorActionPreference`
is `Stop`, PowerShell treats the redirected stderr `ErrorRecord` as a
terminating error and aborts the function before `$LASTEXITCODE` can be read
— even though the external process itself exited 1 correctly (confirmed
independently: `$LASTEXITCODE` was already 1 when checked directly, outside
the `Stop`-scoped helper). Fixed by setting `$ErrorActionPreference =
"Continue"` for the scope of `Invoke-CheckDomainConformance` only (restored
implicitly on function exit, since PowerShell scopes are per-call), leaving
the file-level `Stop` preference intact for every other assertion in the
suite. This is a test-harness artifact of capturing an external process's
stderr under `-Stop`, not a defect in `check-domain-conformance.ps1` itself
(the script's own `exit 1` on bad invocation was correct throughout, verified
independently before and after the fix).

## Scope notes and deviations

- **CLI shape decision**: `check-design-system.sh`/`.ps1` takes
  `<project-root> [<design-md>] [<changed-file>...]` — a single Markdown
  file plus an open-ended changed-file list, because its job is scanning
  arbitrary changed source files for raw style values. `check-domain-
  conformance` has a different, narrower job (validate exactly two Phase-1
  spec artifacts against a contract), so its signature is
  `<project-root> [<requirements-md>] [<design-md>]` — two optional named
  Markdown paths instead of a changed-file list. This preserves the
  precedent's spirit (positional project-root first, optional Markdown
  paths, warn/enforce/skip semantics identical) while fitting the actual
  inputs this check needs, rather than forcing an unused changed-file
  parameter to exist for surface-level signature matching alone.
- **Term-matching implementation (`[[term:Name]]` marker convention)**: the
  task's design.md Assumptions describe the v1 scope as "exact
  canonical-term matching on structured fields only
  (`Bounded-Context:` values, headings, aggregate names)" but does not
  specify an exact marker syntax for how a term reference is spelled inside
  a heading. No existing artifact in this repository (domain-sync,
  domain-interviewer, or any fixture) defines one either — `domain-sync`'s
  "Canonical terms" section explicitly describes term preference as "a
  drafting guideline for the interview, not a separate injected field," with
  no machine-checkable marker. To make Check 2 concretely implementable and
  testable rather than an unfalsifiable prose-matching heuristic, this task
  introduces `[[term:Name]]` as the structured-field marker a heading can
  carry to declare an explicit term reference — analogous in spirit to this
  repository's existing `{{placeholder}}` template-field convention. This is
  additive and opt-in: a `requirements.md`/`design.md` file that never uses
  the marker (the common case today, since nothing generates it yet) never
  triggers Check 2 findings, so this choice cannot introduce false positives
  against any existing spec. The two required checks with concrete,
  spec-mandated data shapes — `Bounded-Context:` field matching (Check 1)
  and aggregate-reference matching via Markdown links to `aggregates/` paths
  (Check 3, matching the exact link shape `domain-sync`'s SKILL.md documents
  for `design.md` cross-references) — are implemented against the literal
  formats already defined by T-007's `domain-sync` and T-002's
  `domain-interviewer`, with no invented convention. Flagging for the
  orchestrator/human: if a different structured-term-marker convention is
  preferred, Check 2's regex (`\[\[term:([^\]]+)\]\]` in both twins) is the
  single, isolated place to change it.
