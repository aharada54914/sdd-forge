## Problem Evidence

Two independent instances within epic-159-pillar-c, both attributable to the
same root mechanism and both traced directly to the same concurrently-merging
sibling branch (`epic-136-phase2-gates`, "Phase 2"):

  Assumptions section stated as an unconditional fact that
  `.github/workflows/test.yml` "remain[s] outside the protected-gate table
  for the duration of this feature," and `AC-027`/`TEST-027` and Main
  Workflows T-001/T-005 built a direct-write registration plan on that
  premise. Spec-review round 2
  (`reports/spec-review/epic-159-pillar-c/attempt-1/round-2/reviewer-b.json`,

## Root Cause Hypothesis

Both instances share the identical shape: a requirements/design document
asserts, as a settled fact at authoring time, a claim about
repository-wide, git-tracked, globally-shared, mutable state that this
feature's own branch does not own exclusively -- a protected-file guard's
current membership set, or the next-free number in a shared sequential
namespace (`docs/adr/NNNN-*.md`). Both kinds of state can be silently
changed by any OTHER branch that merges to `main` between the assumption's
authoring time and the point a reviewer or implementer relies on it,
because neither is scoped to this feature's own files. This repository
already has the correct mitigating pattern in two isolated places -- the A3
assumption's explicit re-verification clause in this same
`requirements.md`, and the ad hoc "re-verify at drafting time" instruction
epic-159-pillar-c's own T-002 task added after the ADR collision was caught
-- but neither is a stated, generalizable rule an author would apply
proactively to a NEW claim of this shape before a reviewer (constrained to
a fixed read-only allowlist that cannot see live guard-configuration files
or run `ls docs/adr/`) has to catch it reactively, or before a human has to
notice the drift and manually supply it as "directing context" to an
otherwise-blind reviewer.

## Proposed Change

| Target File | Change Description |
|---|---|
| `AGENTS.md` § Rules | Add a rule: any `requirements.md` or `design.md` assumption or declaration that asserts a currently-true fact about repository-wide, git-tracked, shared/global state this feature's own branch does not exclusively own (a protected-file/guard membership list, the next-free number in a shared sequential namespace such as `docs/adr/NNNN-*.md`, or an equivalent shared registration surface) must include an explicit re-verification instruction to be executed at the point closest to when the claim is actually consumed -- at spec/design-review time for a claim that gates a reviewer's conclusion, and at implementation/drafting time for a claimed-free identifier -- modeled directly on the existing A3 assumption pattern in `requirements.md` and the re-verify-and-renumber-if-collided instruction this feature's own T-002 task added for `docs/adr/0012`. A bare, unconditional claim about this class of state, with no re-verification instruction, should be treated as a structural gap by spec-review's `ASSUMPTIONS-RESOLVABLE` check, not accepted on the strength of prose alone. |

Reduce the count of spec-review or impl-review findings (Critical or Major)
attributable to "a requirements.md/design.md claim about shared, git-tracked,
globally-scoped repository state was invalidated by a concurrently-merging
sibling branch between authoring time and review/implementation time" from 2

least one other feature branch is concurrently active -- by making the
re-verification instruction an explicit, checkable authoring expectation

incidental awareness of a sibling branch's changes to catch the drift
reactively.
