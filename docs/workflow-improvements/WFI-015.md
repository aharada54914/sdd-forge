## Problem Evidence

Two independent instances, in two different features, of the identical
mechanism: a change alters the condition that previously kept one branch of
an existing test/verification suite from ever executing for real (an
OS/platform capability gate, or a tool-availability SKIP), the newly-active
branch's own environmental assumption was never independently exercised
before, and the gap surfaces only in the real environment the gate exists to
protect (a CI runner's actual OS, or a real release run) -- not on the
developing session's own local host.

- **Instance 1 (quality-loop-fixes, 2026-07-20, commit `ac26cac`, issue
  #203):** `tests/lib/loop-driver.sh`'s `loop_validator_capability_probe`
  had, since before `#199`, silently gated `tests/loop-consistency.tests.sh`'s
  impl/task legs off on `windows-latest` because
  `validate-review-context-set.sh`'s `jq -r` reads emitted CRLF-contaminated
  output there, tripping the probe into "unhealthy" and skipping the
  dependent checks -- a status quo nobody had to look at because the probe's
  own skip made the suite green regardless. `quality-loop-fixes` T-004
  (issue #179, commit `fe7dc81`) fixed the CRLF contamination at its root
  (`validate-review-context-set.sh`'s 12 `jq -r` sites), which flipped the
  capability probe to "healthy" on `windows-latest` for the first time --
  and the previously-gated impl/task legs, now actually executing there for
  the first time, immediately hit "the same CRLF consumption class one layer
  deeper" (`ac26cac`'s own commit message, issue #203), because nothing had
  ever verified those specific legs' own Windows behavior while the probe
  was masking them. `ac26cac` fixed it with a second, behavior-based,
  self-healing probe that gates exactly the 6 dependent checks with a named
  reason instead of silently skipping the whole leg
  (`tests/lib/loop-driver.sh`, `tests/loop-consistency.tests.sh`).
- **Instance 2 (epic-159-pillar-c T-007, this period, 2026-07-21/22, commit
  `c59a247` on the squash-merged PR branch, PR #205):** `tests/bump-version-gate.tests.sh`'s
  TEST-001 (AC-001) exercises `scripts/bump-version.sh`'s real
  `CHANGELOG.md` mutation path, which requires a leading `## Unreleased`
  heading to rewrite. On this dev host, TEST-001 SKIPs itself with a named
  reason ("this host's sed is BSD-style; `scripts/bump-version.sh`'s
  unedited mutation section ... requires GNU sed"), so the fixture-level
  proof that a release actually needs a standing `## Unreleased` heading
  had never run for real locally since the gate suite landed (pillar-b
  #148, commit `306f2b2`, 2026-07-16) -- the ONE actual `bump-version.sh`
  release since then is T-007's own v1.11.0 (this period; the prior release,
  v1.10.0, predates the gate suite by a week). CI's GNU sed lane runs
  TEST-001 for real and would have failed the release: v1.11.0's own
  CHANGELOG.md left no standing `## Unreleased` heading (v1.10.0-era
  convention), a gap invisible on the implementer's own macOS host precisely
  because the SKIP there had never let that specific assumption be
  exercised. Fixed by `c59a247` ("keep a standing Unreleased heading in
  CHANGELOG"), landed inside the same squash-merged PR branch before merge,
  confirmed by `reports/quality-gate/20260721T153827Z-epic-159-pillar-c-T-007.md`'s
  AC-046/AC-050 evidence row and this period's own live re-verification (CHANGELOG.md
  line 3, `## Unreleased`, unchanged since).

## Root Cause Hypothesis

Both instances share the identical shape: an existing test/verification
suite has a branch that is legitimately, honestly gated behind an
environment or platform condition (an OS-specific capability probe; a
tool-portability SKIP) -- not a defect, a correct acknowledgment that the
branch cannot run everywhere. But nothing in this repository's authoring
practice treats "a change is about to flip that gate from closed to open
for the first time in a real environment" as its own re-verification
trigger. Because the gated branch has never executed for real in the
environment where it will now run, any hidden assumption specific to that
environment (Windows's `jq` CRLF behavior; the real CHANGELOG heading
convention GNU sed's `bump-version.sh` mutation depends on) rides
unexercised through the change and is discovered only when the gate finally
opens for real -- at CI time or at release time, the most expensive and
most visible place to discover it, rather than proactively by the author
who changed the gating condition. This is the same class of "claim about
state this branch does not fully control, invalidated only when actually
exercised" that WFI-013 already targets for spec/design claims about
shared repository state -- but the trigger here is a code change altering
a test's own SKIP/capability-gate condition, not a requirements/design
document's prose claim.

## Proposed Change

| Target File | Change Description |
|---|---|
| `AGENTS.md` § Rules | Add a rule: when a change alters the condition that gates an existing test/verification suite's environment- or platform-SKIPped branch (an OS capability probe, a `sed`/toolchain-portability SKIP, or an equivalent conditional gate) such that the gated branch will newly execute for real somewhere it previously did not (a CI runner's actual OS, or a real release run rather than a dry run), the author must, before considering the change complete: (1) name explicitly, in the implementation report, which previously-gated branch is now newly reachable and in which environment; and (2) either exercise that branch for real in a matching environment before merging, or explicitly flag it as "pending first real execution at CI/release time" so the quality-gate reviewer and the next retrospective can trace any resulting failure back to this class rather than treating it as an unrelated surprise. Modeled directly on the two remediations above: `ac26cac`'s named-reason gate on the newly-reachable Windows legs, and this task's own AC-045/AC-046 two-stage design (spec-time attestation plus a mandatory real re-run at release time) as the pattern already proven to catch exactly this class before it reaches CI. |

## Expected Effect

Reduce the count of features whose CI run or release step fails because a
change altered an existing test/verification suite's environment- or
platform-SKIP/capability-gate condition without the newly-reachable branch's
own environmental assumption being independently exercised or explicitly
flagged first, from 2 in this lineage's history (quality-loop-fixes:
issue #203, the Windows capability-probe leg; epic-159-pillar-c T-007: the
bump-version-gate TEST-001 CHANGELOG-heading gap) toward 0 in the next 2
features where a change alters such a gating condition -- by making "this
gate is about to open for the first time in a real environment" an explicit
authoring checkpoint instead of relying on the gate's own historical silence
to stand in for correctness.
