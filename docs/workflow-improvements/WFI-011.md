## Root Cause Hypothesis

epic-159-pillar-a's investigation and requirements-authoring phase asserted
two specific factual claims about existing repository behavior ("N loops
are script-enforced with grep-able numeric caps"; "this script has no
existing direct test driver") without citing the same kind of file:line
grep evidence that `implement-task`'s own acceptance-first discipline later
required, and got, for the *implementation* claims in T-001 and T-004. Both
false premises were caught only because each task's Scope/Done-When
sections independently mandated implementation-time grep verification of a
related but different claim (cap values from driver sources; python3/
resume-script coverage) — the premise-level claim was verified as a side
effect, not because investigation-phase claims are themselves required to
carry the same evidentiary bar. In a task whose Scope did not happen to
require an adjacent grep pass, an equivalent premise error could ship
unnoticed into `requirements.md`/`design.md` and only surface, if at all,
as spec drift discovered by chance during unrelated implementation work.

## Proposed Change

| Target File | Change Description |
|---|---|
| `AGENTS.md` § Rules | Add a rule: when `investigation.md`, `requirements.md`, or `design.md` assert a specific, checkable factual claim about existing repository behavior (e.g., "N scripts enforce X", "script Y has no existing test driver", "script Z contains a numeric limit"), the assertion must cite the specific grep/file:line evidence it rests on in the document itself. Spec-review and task-review should treat an uncited factual claim of this kind as a structural gap, not accept it on the strength of prose alone. |
