## Problem Evidence

All facts below reproduced 2026-07-23 on `main` at `7f152ce5`.

1. The authoring-time validator
   `plugins/sdd-implementation/scripts/validate-implementation-report.sh`
   requires, for any v2 report, a `## Output Paths And Hashes` section
   (`required_headings` tuple, lines 47-54) whose entries are bullets of the
   exact shape `` - **Path**: `x`; **SHA-256**: `h` `` (`output_pattern`,
   lines 114-129).
2. WFI-005 (commit `c3a7a16d`, docs/workflow-improvements/WFI-005.md,
   Status: Verified) REPLACED that section in
   `plugins/sdd-implementation/templates/implementation-report.template.md`
   with a `## Outputs` two-column table (template lines 32-42) — the form the
   independent evaluator launch boundary
   (`plugins/sdd-quality-loop/scripts/validate-review-context-set.sh`)
   parses. WFI-005's Proposed Change row 1 described the template change as
   "additive, so pre-existing consumers are unaffected", but its Result
   section says "replaced the bullet-list output section"; the validator was
   never updated.
3. Consequence, reproduced: a report authored exactly from the current
   template is rejected —
   `bash plugins/sdd-implementation/scripts/validate-implementation-report.sh
   reports/implementation/epic-159-pillar-c/T-004.md` prints
   `IMPLEMENTATION_REPORT_FIELD: missing ## Output Paths And Hashes` and
   exits 1.
4. Committed-report census (heading-line grep over
   `reports/implementation/`, 2026-07-23): 6 v2 reports carry ONLY the old
   bullet section (`agent-cost-context-isolation` T-001/T-002/T-003/T-004/
   T-006/T-008); 23 v2 reports carry BOTH headings (sdd-domain x3,
   local-env-mcp x10, ci-mcp x8, epic-136-phase2-gates x2 — the era that
   satisfied validator and evaluator boundary simultaneously by writing both
   sections); every report from `epic-136-phase1-guards` onward is
   table-only and is rejected by the validator.
5. `tests/turn-first-workflow.tests.sh:288-312` asserts the TEMPLATE
   contains `## Output Paths And Hashes`; the suite currently fails with
   `not ok: implementation report template omits: ## Output Paths And
   Hashes` (reproduced). Neither that suite nor `tests/run-all.sh` is
   registered in `.github/workflows/test.yml`, so the failure never reached
   CI; issue #211 tracks the failure as a symptom with the cause marked
   要調査.
6. The WFI-005 parity suite `tests/template-validator-parity.tests.sh`
   binds the template only to the two consumers named in WFI-005's evidence
   — `validate-review-context-set.sh` and `check-evidence-bundle.sh` (lines
   15-16). `validate-implementation-report.sh` has no parity leg, so exactly
   the drift class WFI-005 was written to prevent recurred against the
   consumer WFI-005 itself broke.
7. The validator has no live invocation site: a repository-wide grep finds
   only a prose reference in
   `plugins/sdd-quality-loop/skills/quality-gate/SKILL.md:199` (describing
   the Model/Effort two-line requirement) and three test drivers
   (`tests/turn-first-workflow.tests.sh`,
   `tests/emit-run-record-feature-scope.tests.sh` / `.ps1`), whose fixtures
   still use the old bullet form because they mirror the validator's current
   behavior.
8. The drift survived a later direct modification of the validator:
   epic-159 Pillar C (commit `825d6c66`, PR #185) added the
   `- Model:` / `- Effort:` checks (validator lines 78-86, AC-026) without
   noticing that its own implementation reports — including
   `reports/implementation/epic-159-pillar-c/T-004.md`, the report for the
   task that modified this validator — fail the validator's outputs check.

## Root Cause Hypothesis

WFI-005 aligned the template with the two consumers its evidence inventory
named (evaluator launch boundary, evidence-bundle validator) and pinned that
alignment with a parity suite scoped to those two — but
`validate-implementation-report.sh`, the same-plugin authoring-time
validator created in `9ad373e5` in agreement with the then-bullet template,
was absent from that consumer inventory, and the "additive" wording in the
Proposed Change concealed that the executed change was a replacement. The
only drivers that exercise the validator live in test suites not registered
in CI (`turn-first-workflow` runs only via `run-all.sh`, itself
unregistered), so the break produced no signal, while dual-heading-era
reports papered over the contradiction at authoring time until the
template-only form became universal.

## Proposed Change

| Target File | Change Description |
|---|---|
| `plugins/sdd-implementation/scripts/validate-implementation-report.sh` | The v2 outputs contract becomes: at least one of (a) `## Outputs` (exactly once) containing >=1 two-column row `` \| `path` \| `sha256` \| `` — validated with the same canonical-repository-path rule, a malformed-row guard (every non-header/non-separator table line must parse), and duplicate-path rejection that the bullet rules apply today; (b) legacy `## Output Paths And Hashes` (exactly once) validated by the existing bullet rules, retained solely so the 6 bullet-only and 23 dual-form committed v2 reports remain valid. When both sections are present both are validated and the duplicate-path check spans both. `required_headings` drops the unconditional `Output Paths And Hashes` entry in favor of this either/both rule; `v2_indicators` is unchanged (no reclassification of schema-less historical reports). All other checks (Model/Effort, Test Evidence, Iteration And Escalation, Isolation Evidence, Session Handoff, legacy pre-v2 acceptance) are unchanged. |
| `tests/template-validator-parity.tests.sh` / `.ps1` | New third leg: render `implementation-report.template.md` with a full substitution set that satisfies every validator rule (Test Result enum, attempt count, escalation vacuity, isolation mode, handoff status — the existing `render` helper fills only the placeholders the other legs need) and run `validate-implementation-report.sh` against the result, expecting `IMPLEMENTATION_REPORT_OK`; plus a validator-pin grep that the Outputs-table row parser remains present. This closes the WFI-005 gap class for this validator at commit time — the parity suite is already registered in `.github/workflows/test.yml` (lines 75/79), so CI coverage requires NO protected-file change. |
| `tests/turn-first-workflow.tests.sh` | The template needle list (line 290) expects `## Outputs` instead of the removed heading; the `current.md` fixture (lines 317-359) converts to the table form and gains the `- Model:` / `- Effort:` lines the epic-159-pillar-c checks require (a second latent fixture failure hidden behind the needle failure); the first `missing_cases` element becomes the table row; two acceptance fixtures are added — a bullet-only v2 report (historical form) and a dual-form v2 report (no duplicate path) — both expected `IMPLEMENTATION_REPORT_OK`; the pre-v2 legacy fixture and its byte-preservation check are unchanged. |
| `tests/emit-run-record-feature-scope.tests.sh` / `.ps1` | Fixture report bodies convert from the bullet form to the current table form (they mirror the validator's contract); assertion logic is unchanged. |

No protected file is touched. CI registration of

   unchanged by this WFI.

   after they were written — the epic-159-pillar-c `- Model:`/`- Effort:`
   checks (29 pre-pillar-c v2 reports lack the lines) and the Test Result

   committed history has never been a property of this validator (it has

   proposal starts from known data.
