## Problem Evidence

- `reports/retrospective/2026-07-12T1530Z-epic-136-phase1-guards.md`, Data
  Notes: "its `gate_reports.total` is 0 because the deterministic counter
  associates gate reports by `Task:`/`Run ID:` header lines while this
  feature's reports use a `Task ID:` header with the evaluator run id in the
  body; the true count is 6 ... Single-instrument observation, low
  confidence, recorded without a WFI; if it recurs next feature it becomes a

- `reports/retrospective/2026-07-14T193907Z-epic-159-pillar-a.md`, FP-03: the
  prediction recurred. Running the emitter for epic-159-pillar-a produced
  `reports/runs/RUN-20260714T193722Z-epic-159-pillar-a.json` with
  `gate_reports.total: 0` (true count: 4, per that report's own Metrics
  table) and `gate_reports.blocked: 1` (true count: 0 — all four gate
  reports read `VERDICT: PASS`).
- Root-cause line citations (both defects, `plugins/sdd-quality-loop/scripts/emit-run-record.sh`):
  - Line 78: `grep -q "Task: ${tid}\b" "$gf"` requires the literal substring
    `Task: T-00N`; this repository's gate-report template header line reads
    `Task ID: T-00N` instead, so the association never matches, for any
    feature using that template.
  - Line 87: `grep -q 'BLOCKED' "$gf"` is an unscoped whole-file substring
    search, not anchored to a `VERDICT:` line. It matched
    `reports/quality-gate/2026-07-14T191311Z-epic-159-pillar-a-T-001.md`'s
    own Critical Review Cycles prose ("design.md:136 example JSON shows
    terminal.state BLOCKED ...", a finding *about design.md's example JSON*,
    unrelated to that report's own `VERDICT: PASS`), producing a false
    `blocked` count.

## Root Cause Hypothesis

The deterministic run-record emitter's gate-report counting logic was
authored against an older or assumed quality verification gate report
header convention (`Task:` / `Run ID:` as separate lines) that does not
match the report template this repository's quality verification gate
skill actually emits (`Task ID:` as a single combined header line, with the
run id embedded in the report body instead of a dedicated header). Because
the counting logic silently returns 0 rather than failing loudly, this
mismatch has now produced identically wrong output across two consecutive
retrospective periods without surfacing as an error. Separately, the
gate-report "blocked" count is derived from a whole-file keyword search
rather than from the report's own verdict field, so any report whose prose
discusses the word "blocked" for an unrelated reason (describing test
behavior, quoting another document, etc.) is miscounted as a blocked gate
run.

Separately, AGENTS.md § Rules "Evidence report identity fields" (added and
Verified by WFI-003, 2026-07-06) currently mandates that quality verification
gate reports "must carry a `Task: T-NNN` line and a `Run ID:` line whose
value equals the evaluator run id reserved in the identity ledger." WFI-003's
Verified Result recorded local-env-mcp's gate reports (2026-07-06) as
compliant with this rule, carrying both `Task:` and `Run ID:` lines in
addition to the template's native `Task ID:` line. The epic-136-phase1-guards
and epic-159-pillar-a gate reports cited as this WFI's own evidence carry
neither line — report authoring has regressed away from WFI-003's Verified
compliance state since 2026-07-06, rather than the emitter's `Task:`/`Run
ID:` expectation having always been merely an incorrect assumption. This
WFI's fix retunes the emitter to accept the now-noncompliant format rather
than restoring AGENTS.md/WFI-003 compliance.

header-association row of the GitHub-Issue Lane is withdrawn under (b), and

(baseline 1, true 0, same horizon). The epic-136-phase1-guards period's six
gate reports remain non-compliant on the main branch history; their
retroactive correction is a separate human decision.
