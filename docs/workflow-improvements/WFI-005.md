## Problem Evidence

The 2026-07-07 retrospective for `sdd-domain`
(`reports/retrospective/2026-07-07T0633Z-sdd-domain.md`, FP-05) records four
template-vs-validator contract gaps in which the written policy or the
canonical artifact template disagrees with the deterministic tool that
enforces it:

1. `plugins/sdd-implementation/templates/implementation-report.template.md`
   emits the heading `## Output Paths And Hashes` with bullet-list entries and
   contains no `- Task ID: T-NNN` line, while the independent evaluator launch
   boundary (`plugins/sdd-quality-loop/scripts/validate-review-context-set.sh`,
   `quality:sdd-evaluator` role) requires line 1 to be exactly
   `# Implementation Report: T-NNN`, a full-line `- Task ID: T-NNN` match, and
   a `## Outputs` section with two-column `` | `path` | `sha256` | `` table
   rows for every authorized non-spec input. All 11 sdd-domain implementation
   reports required manual retrofit before any evaluator could launch, and no
   historical feature's reports satisfy the boundary as generated
   (checked: `agent-cost-context-isolation`, `workflow-state-integrity`).
2. `plugins/sdd-quality-loop/templates/quality-report.template.md` has no
   `Feature:` line, while `check-evidence-bundle.sh` requires the quality
   verification gate report to contain exactly one `^Feature:` line whose
   value equals the verification contract's `feature` field. All 11 sdd-domain
   gate reports added the line manually.
3. `plugins/sdd-quality-loop/references/deterministic-check-policy.md`
   ("check-placeholders Scope and Waivers") documents `required: false` plus a
   non-empty `waiver_reason` as the sanctioned remedy for placeholder-scan
   false positives, but `check-contract.py`'s risk-tier enforcement only
   permits waiving the compile-check set (`lint`, `typecheck`, `build`) on a
   non-code stack — `placeholder-scan` cannot be waived at any tier, so the
   documented remedy is unusable. Observed as a hard blocker on two Risk:high
   task contracts (T-005, T-011 cycle-1 gate records), resolved only by
   rewording shipped prose.
4. Same defect class, adjacent evidence: `check-workflow-state`'s
   stage-provenance check derived the recorded repository root from the
   current directory's basename, which its own release-validation fixture
   (`tests/repository-release-validation.tests.sh`, copies the tree into
   `.../repository/`) violates — every CI OS rejected valid provenance until
   fixed in commit 2c8af66 on 2026-07-07.

## Root Cause Hypothesis

Deterministic enforcement tools for the quality verification gate and the
independent evaluator launch boundary were hardened in separate changes
without updating the artifact templates and policy documents that produce or
describe the artifacts they consume, and no deterministic check binds
template output to validator expectations. The twin guard scripts have a
parity test suite that catches exactly this divergence class between
implementations, but no equivalent exists between templates and the
validators that parse their output — so template/validator drift is only
discovered at gate time, as a hard failure, by whoever runs the gate next.

## Proposed Change

| Target File | Change Description |
|---|---|
| GitHub Issue (plugin-improvement lane; created by the audit cycle and recorded in the GitHub-Issue field above) | Propose three plugin-side fixes, carried verbatim into the Issue body: (1) the implementation report template gains a `- Task ID: {{task_id}}` line directly under its heading and a `## Outputs` section whose entries are two-column table rows (`| path | sha256 |`, backtick-quoted values) for every produced file, matching what the independent evaluator launch boundary parses -- additive, so pre-existing consumers are unaffected; (2) the quality verification gate report template gains a `Feature: {{feature}}` line directly under its `Task ID:` line, matching the evidence-bundle validator's requirement that the gate report name its feature exactly as the verification contract does; (3) the deterministic-check policy resolves the placeholder-scan waiver contradiction in the stricter direction -- the policy text is rewritten to state that the placeholder scan is required at every risk tier and cannot be waived, the remedy for a false positive being to fix or reword the flagged content; this documents the behavior the enforcement tool already has, so no tool behavior changes. |
| tests/template-validator-parity.tests.sh | New deterministic parity test (POSIX shell twin, following the existing twin-test convention): render each canonical template with fixture values and run the actual enforcing parsers against the result -- the evaluator launch boundary's report-heading / `- Task ID:` / `## Outputs`-table checks and the evidence-bundle validator's `Feature:` line check -- failing whenever a template stops satisfying the validators that consume its output, so this drift class is caught at commit time instead of gate time. |
| tests/template-validator-parity.tests.ps1 | PowerShell twin of the same parity test, byte-equivalent in coverage per the twin-test convention. |
