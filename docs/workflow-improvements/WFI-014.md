## Problem Evidence

Two independent instances within epic-159-pillar-d's single spec-review loop,
at two consecutive rounds, both under the identical check ID
(`EDGE-CASE-COVERAGE`) and the identical underlying mechanism:

- **Instance 1 (round 1, Major):** `requirements.md` AC-009 explicitly claims
  the locking suite "locks the fetch-failure (AC-006), diff-detected
  (AC-007), no-diff, and dedup branches" -- four named branches.
  `reports/spec-review/epic-159-pillar-d/attempt-1/round-1/reviewer-b.json`,
  `EDGE-CASE-COVERAGE`, Major, FAIL: "acceptance-tests.md's TEST-009 (the
  sole TEST-ID mapped to AC-009) describes only CI-resilience assertions ...
  it contains no assertion for the 'no-diff' branch ... No TEST-ID in the
  table anywhere exercises 'successful fetch, no divergence -> no issue
  filed.' Downstream failure mode: an implementation bug causing the script
  to file an issue unconditionally regardless of divergence ... would ship
  with all stated acceptance tests green." Remediated at round 2 by adding
  AC-020 (a concrete no-diff zero-invocation assertion), confirmed by round
  2 reviewer-a's `AC-OBSERVABLE` finding.
- **Instance 2 (round 2, Major):** `requirements.md` AC-006 requires
  fail-soft behavior "when either the Anthropic or the OpenAI fetch fails"
  (REQ-002: "on ANY fetch failure").
  `reports/spec-review/epic-159-pillar-d/attempt-1/round-2/reviewer-b.json`,
  `EDGE-CASE-COVERAGE`, Major, FAIL: "acceptance-tests.md TEST-006 scopes
  its fixture only to 'both vendor sources' failing together. No AC/TEST
  fixture exercises the asymmetric case (one vendor succeeds, the other
  fails) ... an implementation that only fail-softs on total dual-vendor
  failure would violate AC-006 with no test to catch it." Remediated at
  round 3 (commit `86cec1f`, "TEST-006 three-scenario fail-soft fixture"),
  later confirmed live by T-003's implementation report's Test Evidence
  ("AC-006 (3 fetch-failure scenarios x2 lanes = 6 scenario runs)") and by
  the T-003 quality-gate report's AC-006 evidence row (all 3 scenarios
  re-run by the independent evaluator).

Both remediations added the missing branch's own fixture/assertion rather
than narrowing the AC's claim -- confirming the gap was genuinely in
acceptance-tests.md's coverage, not in the requirement's wording.

## Root Cause Hypothesis

Both instances share the identical shape: an AC/REQ statement's own language
enumerates or implies an exhaustive set of branches or conditions -- a named
list ("locks the fetch-failure, diff-detected, no-diff, and dedup branches")
or an any/either quantifier ("on ANY fetch failure"; "when either ... fetch
fails") -- but the TEST-ID row(s) acceptance-tests.md actually maps to that
AC cover a narrower subset than the AC's own stated scope.
acceptance-tests.md's TEST-ID table is drafted by paraphrasing each AC's
stated scope, and nothing in the authoring process cross-checks that every
branch/condition the AC's own language names or quantifies has a
corresponding TEST-ID assertion. The gap is therefore caught only reactively
by spec-review's `EDGE-CASE-COVERAGE` check -- here it took two independent
reviewer passes across two rounds to converge, because each pass surfaced
only the instance visible in the artifact state it was given.

## Proposed Change

| Target File | Change Description |
|---|---|
| `AGENTS.md` § Rules | Add a rule: when drafting or amending `acceptance-tests.md`, for every AC whose own language enumerates a set of branches (a named list such as "locks the A, B, C, and D branches") or quantifies over conditions ("any", "either", "each", "every"), the author must expand that language into its individual branches/conditions and verify each one has a TEST-ID row with its own concrete assertion (or an explicit, stated reason a branch is covered elsewhere) BEFORE submitting for spec-review. A branch enumeration whose TEST coverage is a strict subset of the AC's own stated scope should be treated by the author as an authoring defect to fix proactively, not left for spec-review's `EDGE-CASE-COVERAGE` check to catch reactively -- modeled directly on the two epic-159-pillar-d remediations (AC-020's zero-invocation assertion for the no-diff branch; TEST-006's three-scenario fail-soft split for the any/either quantifier). |

own exhaustive-branch/condition language (a named list, or an any/either
quantifier) not fully mirrored by acceptance-tests.md's TEST-ID coverage"
(the `EDGE-CASE-COVERAGE` class) from 2 this period (epic-159-pillar-d:
AC-009's no-diff branch gap at round 1 + AC-006's asymmetric-fetch-failure
gap at round 2) toward 0 in the next 2 features whose spec-review loop
reaches 2 or more rounds -- by making the branch-expansion cross-check an
explicit authoring expectation instead of relying on the reviewer's
reactive check, which in this feature needed two consecutive rounds to
converge.
