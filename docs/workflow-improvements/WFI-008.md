## Problem Evidence

Discovered 2026-07-12 while preparing `chore/track-verification-evidence-logs`
(commit `7d0ea44`, which rescued the sibling features whose logs still
existed on disk): the quality-gate raw logs referenced by the verification
contracts and evidence bundles of features `ci-mcp` and
`evidence-deep-verify` no longer exist anywhere.

- `git log --all --diff-filter=A -- 'specs/ci-mcp/verification/qg/*.log'` is
  empty (likewise for `specs/evidence-deep-verify/verification/qg/`): the
  files were never committed in any branch's history.
- Cause of non-tracking: at the time these features ran their quality
  verification gates, `.gitignore` contained the blanket `*.log` rule with
  only the flat re-include `!specs/**/verification/*.log`; the nested
  re-include `!specs/**/verification/**/*.log` (which covers
  `verification/qg/T-NNN/*.log`) was introduced later on
  `feature/epic-136-phase1-guards` (`fcd13ff`) and is not yet on main.
- Cause of disk loss: the working worktrees `sdd-forge-p4` / `sdd-forge-p5`
  where the gates ran were deleted after the features completed.
- Blast radius (measured 2026-07-12 on this checkout): all 21 evidence
  bundles fail validation — `ci-mcp` T-001..T-013 and `evidence-deep-verify`
  T-001..T-008; 116 unique artifact paths are missing; risk tiers: high 15,
  medium 4, low 2, critical 0 (no HMAC signature involvement).
- What is intact: every `reports/quality-gate/*.md` quality report and every
  `*.contract.json` referenced by the 21 bundles exists with matching
  SHA-256, and every recorded `git_commit` exists and is an ancestor of
  HEAD. Only the raw gate stdout logs (`build.log`, `tests.log`,
  `typecheck.log`, `check-placeholders.log`, `check-task-state.log`,
  `check-traceability.log`, `install-tests.log`, `uninstall-tests.log`) are
  lost.
- Observable failures today: `check-task-state.sh specs/ci-mcp/tasks.md` and
  `check-task-state.sh specs/evidence-deep-verify/tasks.md` FAIL on every
  Done task ("evidence file missing: specs/<feature>/verification/qg/..."),
  and `check-evidence-bundle.sh` FAILs on each of the 21 bundles
  ("artifact ... missing"). A fresh clone reproduces all of it.
- Regeneration cannot restore validity: `check-evidence-bundle.sh` binds
  every artifact to its recorded SHA-256
  (`plugins/sdd-quality-loop/scripts/check-evidence-bundle.sh`, sha256
  mismatch check), and re-run logs cannot reproduce the original bytes.
- Near-miss harm this period: the failures initially presented as
  regressions during 2026-07-12 verification work and required a dedicated
  investigation session to re-attribute to the known loss; nothing currently
  marks these two features' verification failures as known and explained.
- Precedent for recording known verification failures:
  `mcp/sdd-forge-mcp/tests/golden/fixtures/sdd-forge-refactor.expected.json`
  already pins `exitCode: 1` with five "evidence bundle failed validation"
  messages as the expected golden state for feature `sdd-forge-refactor`.

## Root Cause Hypothesis

Two independent conditions combined:

1. **Tracking gap.** The `.gitignore` re-include for verification evidence
   only covered `specs/**/verification/*.log` (flat), while the quality
   verification gate writes its raw logs one level deeper under
   `verification/qg/T-NNN/`. Every gate log for `ci-mcp` and
   `evidence-deep-verify` was therefore silently ignored at commit time,
   while the evidence bundles that reference those logs (with SHA-256
   hashes) were committed. `check-evidence-bundle` passed at generation time
   because the files existed on disk in the working worktree — nothing
   verifies that referenced artifacts are actually git-tracked.
2. **Ephemeral storage.** The only copies lived in throwaway worktrees
   (`sdd-forge-p4` / `sdd-forge-p5`) that were deleted after feature
   completion, converting the tracking gap into permanent loss.

The SHA-256 binding in the evidence design is working as intended: it makes
the loss tamper-evident and non-repairable-by-fabrication. The failure is in
artifact retention, not in the integrity design. A deterministic
"referenced artifacts must be git-tracked" check at bundle-generation time
would close condition 1 categorically, but that is a gate-logic
(measurement-lane) change and is explicitly out of this WFI's scope; it is
noted as a candidate follow-up for a separate WFI.

## Proposed Change

Disposition decision (approved direction from 2026-07-12 analysis): do NOT
edit or regenerate the 21 evidence bundles or their contracts — rewriting
Done-task evidence to reach PASS is the exact tampering class the threat
model defends against (docs/THREAT-MODEL.md, "Agent modifies evidence"),
and the risk-tier enforcement in check-contract.py makes a waiver route
technically impossible for `stack: code` contracts anyway. Instead, record
the loss honestly, pin the known-failure state so regressions stay
distinguishable, and close the tracking gap.

| Target File | Change Description |
|---|---|
| `specs/ci-mcp/verification/EVIDENCE-LOSS.md` (new) | Record the loss for ci-mcp T-001..T-013: what is missing (per-task qg log list), why (gitignore nesting gap + p4/p5 worktree deletion), what remains intact (quality reports, contracts, git_commit ancestry), why regeneration is excluded (SHA-256 binding + tamper-evidence rationale), and a pointer to this WFI. |
| `specs/evidence-deep-verify/verification/EVIDENCE-LOSS.md` (new) | Same record for evidence-deep-verify T-001..T-008. |
| `mcp/sdd-forge-mcp/tests/golden/task-state-golden.test.ts` | Add `"ci-mcp"` and `"evidence-deep-verify"` to the `GOLDEN_FEATURES` array so the parser-vs-shell parity suite covers both features' known-failure state. |
| `mcp/sdd-forge-mcp/scripts/record-golden-fixtures.ts` | Add the same two features to its `GOLDEN_FEATURES` array so `npm run golden:record` can (re)record their fixtures. |
| `mcp/sdd-forge-mcp/tests/golden/fixtures/ci-mcp.expected.json` (new, recorded) | Record via `npm run golden:record`: expected `exitCode: 1` plus the exact per-task "evidence bundle failed validation" message set — following the existing `sdd-forge-refactor.expected.json` known-failure precedent. |
| `mcp/sdd-forge-mcp/tests/golden/fixtures/evidence-deep-verify.expected.json` (new, recorded) | Same recording for evidence-deep-verify. |

Dependency (not a direct target of this WFI, no duplicate application): the
`.gitignore` nested re-include `!specs/**/verification/**/*.log` and the 46
rescued log files for the sibling features are already authored on branch
`chore/track-verification-evidence-logs` (commit `7d0ea44`, unpushed).
Merging that branch to main is the recurrence-prevention half of this
disposition and is tracked in the Verification Plan.

## Expected Effect

1. Misdiagnosis of the known loss as a new regression drops from 1
   investigation this period (2026-07-12) to 0: any check-task-state or
   evidence-bundle failure for `ci-mcp` / `evidence-deep-verify` is
   explained in-place by `EVIDENCE-LOSS.md`, and any CHANGE in the failure
   set (including a silent flip to PASS) breaks the golden fixtures in the
   3-OS CI matrix.
2. Bundle-referenced verification artifacts missing from a fresh clone for
   newly completed features drops from 116 unique paths accumulated this
   period (ci-mcp + evidence-deep-verify, unrecoverable) to 0, once the
   `7d0ea44` re-include lands on main.
3. Zero weakening of verification: no gate, grader, threshold, contract, or
   bundle is modified; the 21 bundles continue to FAIL deterministic
   validation, by design.

## Verification Plan

At the next completed feature's retrospective analysis:

1. **Recurrence check (primary metric).** In a fresh-clone-equivalent
   worktree of main, run
   `sh plugins/sdd-quality-loop/scripts/check-evidence-bundle.sh` over every
   `specs/<new-feature>/verification/T-*.evidence.json` of the newly
   completed feature: all must PASS, and
   `git ls-files --error-unmatch <artifact>` must succeed for every artifact
   path listed in those bundles. The proposal succeeds on this axis only if
   the missing count is 0.
2. **Known-failure pinning check.** Confirm
   `mcp/sdd-forge-mcp/tests/golden/fixtures/ci-mcp.expected.json` and
   `evidence-deep-verify.expected.json` exist and the golden suite
   (`npm test` under `mcp/sdd-forge-mcp`, mcp-tests job) passes on the 3-OS
   CI matrix — i.e., the recorded failure sets still match the live shell
   output exactly.
3. **Misdiagnosis check.** Grep `docs/review-tickets/` and the retrospective
   report for any new ticket attributing a `ci-mcp` /
   `evidence-deep-verify` bundle-validation failure to a code regression:
   expected count 0.
4. **Dependency landed (precondition for step 1's Target-Metric).**
   Confirm `.gitignore` on main contains
   `!specs/**/verification/**/*.log` (via the `7d0ea44` merge or an
   equivalent commit). Merging `chore/track-verification-evidence-logs`
   (commit `7d0ea44`) before the next completed feature's retrospective is a
   human action owned by the repository operator (the branch is authored and
   verified but unpushed; approving this WFI implies scheduling that merge).
   If the re-include has NOT landed on main by the horizon, the
   Target-Metric comparison in step 1 is deferred — not scored as a miss
   against this WFI — until the dependency lands, and the deferral is noted
   in that retrospective.

## Rollback-Plan

Revert the commit(s) whose message contains `WFI-008`. They touch only the
two new `EVIDENCE-LOSS.md` records, the two `GOLDEN_FEATURES` array entries
(test + record script), and the two recorded fixture JSON files — no gate
script, schema, hook, contract, or evidence bundle is modified. Reverting
restores the prior state exactly: both features' bundles fail validation
identically before and after (the gates never depended on the added files),
and the golden suite simply stops covering the two features. The
`.gitignore` dependency rides `7d0ea44` and has its own revert path.
