## Problem Evidence

On 2026-07-22, during the epic-191 impl-review (session working directory
outside the repository, e.g. `~/Setup`), the Impl-Review-Status guard
falsely denied writing `Impl-Review-Status: Passed` into
`specs/<feature>/design.md` even though a genuine
`integrated-verdict.json` with verdict PASS existed and
check-workflow-state was green (user report, this filing's trigger).

The defect is confirmed in all three logic twins at the same function:

- `plugins/sdd-quality-loop/scripts/sdd-hook-guard.py:141-154`
  (`_impl_review_verdict_exists`): docstring says "CWD-relative path
  resolution (matches JS behavior, ADR-004)";
  `glob.glob("reports/impl-review/<feature>/attempt-*/round-*/integrated-verdict.json")`
  is resolved against the process CWD only.
- `plugins/sdd-quality-loop/scripts/sdd-hook-guard.js:1303-1338`
  (`implReviewVerdictExists`): `fs.readdirSync('reports/impl-review/' +
  feature)` -- CWD-relative only.
- `plugins/sdd-quality-loop/scripts/sdd-hook-guard.ps1:888-917`
  (`Test-ImplReviewVerdictExists`): `Test-Path "reports/impl-review/$feature"`
  -- comment says "(CWD-relative, ADR-004)".

Deterministic reproduction (recorded in this session, 2026-07-22, macOS,
live guards at main `eea908c`): with a fixture repository containing a PASS
verdict and the process CWD outside it, the payload
`{"tool_name":"write","tool_input":{"file_path":"<abs>/specs/feat-y/design.md","content":"Impl-Review-Status: Passed\n"}}`
run with `CLAUDE_PROJECT_DIR=<abs repo>` exits 2 (deny) on all three live
twins (py, js, ps1) and exits 0 (allow) on the staged fixed twins. The same
reproduction is now locked as guard-parity Scenario 37.

Why the existing parity suite missed it: `tests/guard-parity.tests.sh`
Scenarios 19-22 exercise the verdict gate via `parity_check_in "$WORK"`,
which sets BOTH the CWD and `CLAUDE_PROJECT_DIR` to the fixture repository,
so the CWD-relative glob always succeeded in CI.

Cross-check of the sibling gates (this session): the CWD-only resolution is
unique to the impl-review verdict check. No spec-review or task-review
verdict file is read by the guard scripts (`reports/spec-review` /
`reports/task-review` are not referenced), and the guards' other root
resolutions (`_resolve_project_root`, kill-switch C-08 parent walk) already
use the multi-candidate order. The `.sh` twin is a dispatcher with no
verdict logic.

## Root Cause Hypothesis

The impl-review verdict check was implemented with CWD-relative resolution
(explicitly documented as matching ADR-004's then-current JS behavior) and
was never migrated to the multi-candidate root-resolution pattern
(`CLAUDE_PROJECT_DIR` -> git-root walk -> CWD) that the same scripts already
apply to SDD_SUDO and AGENT_STOP lookups. The parity suite reinforced the
blind spot because its verdict scenarios pin CWD to the repository, so the
only run configurations that fail are ones CI never exercises -- they occur
in real sessions whose CWD is outside the repository (multi-project
sessions such as `~/Setup`).

## Expected Effect

False denials of `Impl-Review-Status: Passed` writes in sessions whose CWD
is outside the repository drop from 1 observed occurrence this period
(epic-191, 2026-07-22) to 0 in subsequent impl-review completions, and the
guard-parity deterministic check count rises from 36 to 39 scenarios (three
added, none removed or loosened; Scenario 39 pins deny-on-FAIL from an
outside CWD).
