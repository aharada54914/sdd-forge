## Problem Evidence

The 2026-07-09 retrospective for `evidence-deep-verify`
(`reports/retrospective/2026-07-09T0245Z-evidence-deep-verify.md`, FP-02)
records a path-convention contradiction between installed plugin
documentation and the deterministic validator that gates independent
evaluator launch:

- The independent evaluator launch boundary requires the canonical
  per-feature directory path: `plugins/sdd-quality-loop/scripts/validate-review-context-set.sh:265`
  matches only `reports/implementation/$feature/$task_id.md` for the
  `quality:sdd-evaluator` role, failing launch (`PATH 'sdd-evaluator requires
  the current task implementation report'`) on any other layout.
- Eight installed plugin skill/reference/template locations instead document
  the pre-canonicalization flat form `reports/implementation/<task-id>.md`
  (no per-feature directory segment):
  `plugins/sdd-implementation/skills/implement-task/SKILL.md:100`,
  `plugins/sdd-implementation/skills/implement-task/references/agent-delegation-policy.md:30`,
  `plugins/sdd-implementation/skills/implement-task/references/implementation-policy.md:38`,
  `plugins/sdd-implementation/skills/implement-tasks/SKILL.md:191`,
  `plugins/sdd-implementation/skills/visual-verify-loop/SKILL.md:58`,
  `plugins/sdd-quality-loop/references/deterministic-check-policy.md:140`,
  `plugins/sdd-bootstrap/skills/sdd-bootstrap-interviewer/templates/AGENTS.template.md:34`
  (the shared project-constitution scaffold every bootstrapped project
  starts from), and
  `plugins/sdd-lite/skills/lite-gate/SKILL.md:29`.
- Concrete harm this period: all 8 of `evidence-deep-verify`'s implementation
  reports were first committed at the flat path
  `reports/implementation/evidence-deep-verify-T-NNN.md`, following the
  implementation-phase plugin's own documented convention verbatim. The
  independent evaluator's launch-boundary reservation failed on `PATH` for
  the flat layout, and the quality verification gate's own reviewer
  corroborates the cause: `reports/quality-gate/2026-07-09T021500Z-evidence-deep-verify-T-007.md:40`
  ("Report-path move to the canonical reports/implementation/<feature>/T-NNN.md
  layout was an orchestrator gate-phase normalization required by
  validate-review-context-set."). The fix required a bulk 9-file `git mv` in
  a single gate-phase commit (`e3c148e`: `T-001`-`T-008` plus the `T-006`
  traceability addendum) rather than the reports landing correctly at their
  first commit.

## Root Cause Hypothesis

The independent-evaluator launch boundary was hardened to require the
canonical per-feature report directory, but the implementation-phase
plugin's own skill and reference files -- and the shared bootstrap
constitution template every project starts from -- were never updated to
match; they still instruct implementers to write to the pre-canonicalization
flat path. Because every one of these locations is read verbatim by the
implementation-phase workers that author the reports, the contradiction is
structural rather than an individual drafting slip: it reproduces on every
task of every full-track feature until the documentation itself is
corrected, and is only ever caught at gate time, as a hard launch-boundary
failure, by whoever runs the gate.

The lightweight track's own gate-state validation is deliberately
path-agnostic (it locates a task's implementation report by recursive
task-ID search rather than requiring a specific path), and the lightweight
track's own gate skill does not invoke the independent-evaluator launch
boundary at all -- so the lightweight track's documented flat-path
convention is not part of this contradiction and is out of this WFI's scope.
