## Root Cause Hypothesis

Both gaps share the same shape: a translation- or detection-layer authoring
step relies on the author's (or a single review pass's) manual vigilance for
a property that is easy to state as a rule but easy to miss case-by-case —
"every case-sensitive `.sh`/`sed`/`jq` comparison must have a case-sensitive
`.ps1` equivalent" and "a detection suite's own source must never contain the
literal substring it detects." Neither property currently has a checklist
step in `AGENTS.md` or the implementation-phase skills that an author (or a
single reviewer) can mechanically walk before declaring a task
Implementation Complete. For FP-01 specifically, the gap is compounded by
PowerShell having *two independent* layers of default case-insensitivity
(operators such as `-match`/`-eq`, and cmdlets such as `Select-String`/
`Get-ChildItem -Filter`) that do not share a single sweep methodology, so a
checklist naming only "check for case-sensitivity" without both layers
explicit would reproduce exactly the cycle-1-then-cycle-2 escalation this
feature's quality-gate needed. For FP-02, the gap is that "do not embed the
marker literal" is an easy rule to forget specifically because embedding it
is the natural way to write a human-readable assertion message or grep
pattern — the failure mode is invisible until a deterministic scanner
(placeholder-scan) or grep self-check actually runs against the suite's own
source.

## Proposed Change

| Target File | Change Description |
|---|---|
| `AGENTS.md` § Rules | Add a rule: any `.sh`→`.ps1` full-parity translation port must be swept for case-sensitivity weakening at **two independent layers** before being reported Implementation Complete — (1) an **operator-level** sweep covering every `-match`/`-notmatch`/`-eq`/`-ne`/`-contains`/`-notcontains`/`-replace`/`-like`/`-notlike` site against a value the `.sh` original compares case-sensitively (`[[ =~ ]]`, `==`, `sed`, `jq ==`/`test()`), narrowed to its `-c`-prefixed case-sensitive variant; and (2) a separate **cmdlet/language-feature-level** sweep covering `Select-String`, `Get-ChildItem -Filter`/`-Include`/`-Exclude`, `-split`, `[regex]::Match`/`IsMatch`/`Replace`, `switch -wildcard`/`-regex`, `Sort-Object`, and raw string-instance methods — each cmdlet/operator defaults independently and one sweep does not imply the other. Each port's acceptance evidence must include at least one **mis-cased negative fixture per layer** (e.g. an uppercase-variant input the `.sh` original rejects) proving the `.ps1` port rejects it identically. |
| `AGENTS.md` § Rules | Add a second rule: any test suite whose acceptance checks assert against a literal marker/vocabulary string (grep pattern, output assertion, or human-readable `ok`/`fail` message) must assemble that marker at runtime from non-contiguous literals (or otherwise avoid embedding the contiguous banned substring in its own source, comments, or messages) whenever the same marker vocabulary is also plausibly scanned by a deterministic detection gate the suite's own repository runs (e.g. `check-placeholders`, a `gh`-invocation grep self-check) — so the suite's own source is never a false-positive target of the detection mechanism it exists to test. |

task or one marker-detection test-suite-authoring task:

   PowerShell case-sensitivity weakened-port finding, from 4 this period

   them.

   for, from 1 this period (T-002's `0438a52` remediation) toward 0.
