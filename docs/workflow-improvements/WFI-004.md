## Problem Evidence

- `docs/review-tickets/RT-20260705-001.yml` (feature local-env-mcp, severity
  major, requires_human_decision: true): the first end-to-end run of the
  full-profile persisted-state validation after the implementation phase
  surfaced five contradictions; two are unresolvable without either evidence
  fabrication or a workflow change.
- Validator diagnostics observed in sequence on 2026-07-05 (each blocking the
  quality verification gate for all 10 tasks of the feature):
  `implementation design hash is stale`, `task reviewer outputs or integrated
  summary contradict the final PASS`, `task traceability hash is stale`,
  `task reviewer manifests omit layer inputs`
  (`plugins/sdd-quality-loop/scripts/check-workflow-state.sh` exit 1).
- Schema contradiction: `plugins/sdd-review-loop/agents/task-reviewer-a.md`
  lines 222-256 document the reviewer output as `stage: "task"`,
  `role: "task-reviewer-a"`, `allowed_input_manifest`, `checks[].result`,
  while `check-workflow-state.sh` lines 354-372 and 386-397 accept only
  `stage: "task-review"`, `role: "reviewer-a"`, `manifest`,
  `checks[].status`, and a `findings` array. A reviewer that follows its
  role file can never satisfy the validator.
- Re-review deadlock: `task-reviewer-a.md` lines 125-130 (INITIAL-STATE)
  mandate that the approval field "must not be" the approved value and that
  `Status:` stay in {Planned, In Progress, Blocked} for any round after
  round 1 attempt 1, while every post-implementation state legitimately
  carries an approved-with-audit-mark approval field and
  `Status: Implementation Complete` (specs/local-env-mcp/tasks.md). Any
  truthful post-implementation re-review therefore returns NEEDS_WORK
  forever.
- Artifact-freeze contradiction: `check-workflow-state.sh` line 595 requires
  the design review gate to be Passed whenever tasks.md exists, and
  stage-provenance (lines 471-474, 503-505) binds design.md and
  traceability.md content hashes with only status-header normalization.
  Yet the task decomposition review gate PASSED tasks whose approved scopes
  update those very artifacts (specs/local-env-mcp/tasks.md T-005: design
  open-question resolution; T-010: traceability verification-status
  finalization). The sanctioned edits had to be reverted (commit a059524)
  to restore validator green-ability, leaving two approved Done When items
  unsatisfiable as written.

## Root Cause Hypothesis

The full-profile persisted-state validation path was authored against a
canonical task-stage evidence schema and post-implementation invariants that
had never been exercised end-to-end: every prior feature was registered as
legacy or lightweight-track and bypassed these checks. The task decomposition
review gate's reviewer role definitions were written against an earlier
evidence schema and only cover pre-implementation review states, and the
stage-provenance design assumes review-bound artifacts never change after
their gate passes — an assumption the specification generator's own
open-question resolution pattern and the task decomposition review gate's own
approved task scopes contradict. The drift is structural (validator vs. role
definitions vs. sanctioned task scopes), not an individual authoring mistake.

## Proposed Change

| Target File | Change Description |
|---|---|
| AGENTS.md | Add two project-side rules under Sources Of Truth. (1) "Post-review artifact freeze": once a review gate passes, its hash-bound artifacts (the design document after the design review gate; the task plan body and traceability document after the task decomposition review gate) are content-frozen except for the normalized status/approval fields. Sanctioned later updates — open-question resolutions, verification-status finalization — are recorded in non-frozen addenda (implementation reports, `specs/<feature>/verification/`, user documentation) instead of the frozen artifact, and task authors must scope Done When items accordingly. When an already-approved task's Done When names a frozen artifact, the Done When wording is amended to name the equivalent addendum record — a spec change requiring explicit human authorization, recorded in the task plan and re-bound by a post-implementation provenance re-review (human decision of 2026-07-05 for the blocked feature: wording amendment, not deemed satisfaction). (2) "Post-implementation provenance re-review": when task-stage review evidence must be re-bound after the implementation phase (evidence-schema drift, incomplete reviewer input manifests), the task decomposition review gate runs a new attempt in which both reviewers receive the complete input set including all four layer specification files, emit the persisted-state validator's canonical task output schema (reviewer A: top-level `feature`/`attempt`/`round`, `stage: "task-review"`, `role: "reviewer-a"`, `manifest` array of path+sha256 pairs, `checks[].status`, and a `findings` array with one severity-bearing entry per FAIL; reviewer B: top-level `feature`/`attempt`/`round`, `manifest.allowed_inputs`, `checks[].result`, and the same `findings` array), and evaluate task state by lifecycle validity — an approved approval field bearing a valid human or workflow-bypass-mode audit mark and statuses {Planned, In Progress, Blocked, Implementation Complete, Done} are valid — instead of the pre-implementation initial-state rule. The mismatch between the review gate plugin's shipped role definitions and the validator's canonical schema is tracked as a linked plugin-maintainer follow-up in this WFI's GitHub Issue; this WFI does not author plugin-internal changes. |

## Expected Effect

Reduce the persisted-state validator exit code for the blocked feature
(local-env-mcp) from 1 (failing with the layer-manifest diagnostic) to 0,
with zero reduction in the number of deterministic checks, gates, or hash
bindings enforced. Qualitatively: the persisted-state validator becomes
satisfiable for full-profile features after the implementation phase — the
task decomposition review gate can run a truthful post-implementation
provenance re-review (unblocking features whose review evidence needs
re-binding), reviewer outputs validate without orchestrator field-encoding
migration, and sanctioned post-review updates no longer create
frozen-artifact drift. The validator script, its thresholds, and its hash
bindings are unchanged.
