## Problem Evidence

- Retrospective `reports/retrospective/2026-07-05T022733Z-sdd-forge-mcp.md`, FP-002: all 11 implementation reports lack `Task Attempt Count`/`Run ID`; all 11 quality-gate reports use `Task ID:` without a `Run ID:` field, so the retrospective's deterministic artifact rules cannot associate them, producing 33 `N/A` metric cells for 11 fully-evidenced tasks.
- Run record `reports/runs/RUN-20260705T023011Z-sdd-forge-mcp.json`: `gate_reports.total: 0` and `first_pass_gate.passed_first_try: 0/11` despite 11 PASS reports on disk; `review_tickets.major: 1` counts `RT-20260623-001` from a previous feature. The emitter's association rules cannot see this feature's evidence.
- The two measurement consumers (retrospective analysis and the run-record emitter) silently under-count real evidence, which makes every future WFI verification comparison unreliable.

## Root Cause Hypothesis

Report authors follow the quality verification gate's report conventions (`Task ID:` header line, evaluator run ids recorded in the identity ledger), while the measurement pipeline expects an additive identity schema on the artifacts themselves (`Task:` plus `Run ID` on gate reports; `Task Attempt Count` plus `Run ID` on implementation reports). Nothing in the project's workflow files states these required fields, so every artifact is written in the older shape and the measurement pipeline cannot associate evidence deterministically — the drift is uniform across all authors and tasks, confirming a missing-specification cause rather than individual authoring mistakes.

## Proposed Change

| Target File | Change Description |
|---|---|
| AGENTS.md | Add an "Evidence report identity fields" subsection under AGENTS.md § Rules (the section that governs artifact conventions): implementation reports must carry `Run ID:` and `Task Attempt Count:` lines; quality verification gate reports must carry a `Task: T-NNN` line and a `Run ID:` line whose value equals the evaluator run id reserved in the identity ledger. State that these fields are additive (existing consumers such as check-task-state and evidence-bundle generation are unaffected) and exist so that retrospective analysis and run-record emission can count evidence deterministically. |

## Expected Effect

Retrospective metric cells reported as `N/A` due to schema drift drop from 33 (this period) to 0, and the run-record `gate_reports.total` equals the number of on-disk PASS gate reports for the feature instead of 0. No graders or thresholds change; only the artifacts gain the identity fields the existing measurement logic already expects.
