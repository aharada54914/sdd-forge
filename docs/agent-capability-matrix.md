# Agent Capability Matrix — SDD Quality Loop

A read-only summary of agent roles, tool scopes, and which actions are enforced by the hook guard vs. by convention.

---

## Agent Roles and Capabilities

### sdd-investigator
**Purpose:** Read-only codebase investigation for problem decomposition.

**Sandbox Mode:** `read-only` (TOML setting)

| Capability | Allowed | Enforced By | Notes |
|---|---|---|---|
| **Read files** | ✓ | Convention + sandbox | Bash: `cat`, `ls`, `grep`, `find`, `head`, `tail`, `stat` permitted; any write command denied by sandbox |
| **Run tests** | ✓ | Convention | Bash: `npm test`, `python -m pytest`, etc. for verification only |
| **Write to code/spec files** | ✗ | Sandbox mode | sdd-investigator role has `sandbox_mode = "read-only"` |
| **Write `developer_instructions` in .codex/agents/*.toml** | ✗ | Hook guard + Convention | Agent role file write is rejected if `developer_instructions` is removed or absent (T-001 guard) |
| **Set Approval / Second Approval / WFI Approved** | ✗ | Hook guard | sdd-hook-guard blocks any write that increases these fields in tasks.md |
| **Write SDD_SUDO** | ✗ | Hook guard | Guard denies any write to SDD_SUDO file (shell commands targeting it) |
| **Mint evidence HMAC signatures** | ✗ | External key + policy | Evidence signing requires external `SDD_EVIDENCE_KEY` (agent cannot read it) |
| **Generate SDD_SUDO tokens** | ✗ | External key + policy | Sudo tokens require external `SDD_SUDO_KEY` (agent cannot read it) |

**Output:** Markdown reports with findings, file:line references, no code modifications.

---

### sdd-evaluator
**Purpose:** Independent review of implementation against approved spec; return PASS/NEEDS_WORK with classified findings.

**Sandbox Mode:** `read-only` (TOML setting)

| Capability | Allowed | Enforced By | Notes |
|---|---|---|---|
| **Read spec and code** | ✓ | Convention + sandbox | Read requirements.md, design.md, acceptance-tests.md, implementation files, test output |
| **Run tests and builds** | ✓ | Convention | Bash: `npm test`, `npm build`, linters permitted for verification |
| **Write evaluation report** | ✓ | Convention | Output is a plain-text verdict (PASS/NEEDS_WORK) and findings; not written to tasks.md |
| **Write to code/spec files** | ✗ | Sandbox mode | sdd-evaluator role has `sandbox_mode = "read-only"` |
| **Set Approval / Second Approval / WFI Approved** | ✗ | Hook guard | sdd-hook-guard blocks any write that increases these fields in tasks.md |
| **Set review_verdict in evidence bundle** | ✗ | Policy + hook guard | Review verdict is generated by the evaluator but recorded by an external skill, not by the agent directly writing to JSON |
| **Write SDD_SUDO** | ✗ | Hook guard | Guard denies any write to SDD_SUDO file |
| **Mint evidence HMAC signatures** | ✗ | External key + policy | Signature generation is out-of-process (generate-evidence-bundle script); agent cannot invoke it |
| **Generate SDD_SUDO tokens** | ✗ | External key + policy | Sudo tokens require external `SDD_SUDO_KEY` (agent cannot read it) |

**Output:** Verdict and findings (Critical/Major/Minor); attestation of checked behaviors. Verdict is ingested by quality-gate skill, not written by agent to approval fields.

### sdd-panelist-gpt
**Purpose:** Independent blind verification of a task's implementation from an OpenAI vendor perspective; emits a structured verdict JSON as advisory signal.

**Vendor / Invocation:** `openai` — invoked via codex CLI (`run-panelist-gpt.{sh,ps1}`).

**Sandbox Mode:** `read-only` (TOML setting; `disallowedTools: Write, Edit, NotebookEdit`)

| Capability | Allowed | Enforced By | Notes |
|---|---|---|---|
| **Read sanitized task input** | ✓ | Convention + sandbox | Receives only the `prepare-panelist-input` sanitized bundle (no secrets, no `.env`, no absolute paths, no private URLs) |
| **Emit verdict JSON** | ✓ | Convention | Outputs `cross-model-verdict/v1` JSON with `verdict` (PASS/NEEDS_WORK), `findings`, `blind:true`, `input_digest`, `consent` |
| **Write code or spec files** | ✗ | Sandbox mode (`disallowedTools`) | Panelist role is read-only; cannot modify the repository |
| **Approve tasks or set Approval fields** | ✗ | Hook guard | sdd-hook-guard blocks any write that increases Approval/Second Approval counts |
| **Sign evidence bundles** | ✗ | External key + policy | No access to `SDD_EVIDENCE_KEY`; cannot invoke `generate-evidence-bundle` |
| **Read evaluator verdict or prior review tickets** | ✗ | Convention (blind execution) | `prepare-panelist-input` provides only the task spec; evaluator context is excluded |
| **Receive secrets, keys, or absolute paths** | ✗ | Sanitization step | `prepare-panelist-input` strips these before any external send |

**Output:** `specs/<feature>/verification/T-NNN.panelist-openai.verdict.json` (`cross-model-verdict/v1`).
**External API send:** Only after explicit consent (`Cross-Model: enabled` in `tasks.md` OR valid `SDD_SUDO`). Never invoked in CI.

**Agent role files:**
- `.codex/agents/sdd-panelist-gpt.toml` (developer_instructions 必須)
- `plugins/sdd-quality-loop/agents/panelist-gpt.md`

## Turn-First Model Routing — M-04

Turn-first routing optimizes expected implementation/review iteration count
before token price. Expected iterations are optimized first, then the weakest sufficient tier, then token price.
Concrete model IDs are examples only; check availability at invocation and
record the selected provider/model.

### Predicted Attempts Matrix

| Risk | lightweight | standard | strong |
|---|---:|---:|---:|
| low | 1 | 1 | 1 |
| medium | 2 | 1 | 1 |
| high | 3 | 2 | 1 |
| critical | 3 | 2 | 1 |

The selector chooses the minimum predicted-attempt count. Ties choose the lower
vendor-neutral tier. Only equal-tier routes compare invocation-supplied
`estimated_cost_per_attempt_usd`; the route also records `cost_estimate_source`
and `cost_estimate_timestamp` so checked-in policy never embeds stale prices.
Equal cost is resolved by the lexicographically smaller provider/model
identifier, written as the lexicographically smaller `provider/model`.

### Provider Tier Mapping

| Tier | Provider | Model family | Reasoning effort | 最終確認日 | 参照ソース |
|---|---|---|---|---|---|
| lightweight | Anthropic | Haiku | low | 未確認 | Anthropic official docs (models overview) |
| standard | Anthropic | Sonnet | medium | 未確認 | Anthropic official docs (models overview) |
| strong | Anthropic | Opus | high | 未確認 | Anthropic official docs (models overview) |
| lightweight | OpenAI/Codex | `gpt-5.1-codex-mini` | low | 未確認 | OpenAI developers docs (Codex) |
| standard | OpenAI/Codex | `gpt-5.1-codex` | medium | 未確認 | OpenAI developers docs (Codex) |
| strong | OpenAI/Codex | `gpt-5.2-codex` (`gpt-5.1-codex-max` fallback) | high or xhigh | 未確認 | OpenAI developers docs (Codex) |

Codex agent model selection is a runtime control (`--model` / `--effort` flags
passed to the Codex CLI at invocation), not a field in the role TOML. The
`.codex/agents/*.toml` files intentionally do not pin a `model` key because the
role schema has historically rejected malformed role definitions. Routing
therefore happens at the call site after model availability is checked.
The caller filters candidates through the host capability registry. A
substitute is eligible only when the registry assigns it to the same canonical
tier; the canonical tier does not change and the concrete provider/model is
recorded. If no available candidate satisfies an exact tier or role floor,
routing fails closed as `model-tier-unavailable`. For strong Codex routing,
high effort is the default. Xhigh is used only when the capability registry or
a task-specific evaluator contract requires it, with the reason recorded.

### Effort Policy: `welded` vs. `matrix`

`select-agent-model`'s `--effort-policy` flag chooses HOW the winning
model's reasoning effort is resolved, as a second, independent axis from
tier selection (`docs/adr/0012-effort-tier-decoupling.md`). As of T-007
(Issue #155, epic-159-pillar-c Phase 2), the default effort policy is `matrix`:
effort resolves from `contracts/agent-model-capabilities.v2.json`'s
`risk_effort_matrix[risk]` (low→low, medium→medium, high→high,
critical→high), with a one-step escalation bump on a repeated
same-classified failure, clamped to the winning model's own
`supported_efforts`; when the matrix has no entry for the supplied risk,
`role_defaults[role].default_effort` is used instead, falling back to the
winning model's own `default_effort` as a last resort.
`--requested-effort <e>` always wins over either policy when explicitly
supplied, subject to the same clamp and `--xhigh-reason` gate.

`welded` (the sole policy through Phase 1, T-001..T-006) remains fully
supported indefinitely via an explicit `--effort-policy welded` flag — no
deprecation timer (OQ-004). It reproduces the pre-Phase-2 byte-identical
effort selection: a candidate's own declared effort, or the winning
model's own `default_effort` when none is declared, with no risk-based
computation at all.

### Role Floors

| Role | Required tier | Anthropic route | OpenAI/Codex route |
|---|---|---|---|
| sdd-investigator | lightweight | Anthropic Haiku | OpenAI/Codex `gpt-5.1-codex-mini`, effort low |
| spec-reviewer-a/b | standard minimum | Anthropic Sonnet or stronger | OpenAI/Codex `gpt-5.1-codex`, effort medium or stronger |
| impl-reviewer-a/b | standard minimum | Anthropic Sonnet or stronger | OpenAI/Codex `gpt-5.1-codex`, effort medium or stronger |
| task-reviewer-a/b | standard minimum | Anthropic Sonnet or stronger | OpenAI/Codex `gpt-5.1-codex`, effort medium or stronger |
| sdd-evaluator | strong | Anthropic Opus | OpenAI/Codex `gpt-5.2-codex` (`gpt-5.1-codex-max` fallback), effort high or xhigh |

### Escalation

Failure classes are the closed enum `test`, `lint`, `typecheck`, `build`,
`review-major`, and `review-critical`. A tier becomes ineligible for the next
attempt only when the same classified failure occurs twice consecutively for the
same task; different failure classes do not accumulate. Escalation is a
one-tier increase and records prior tier, next tier, failure class, attempt
number, and reason. At `strong`, recurrence blocks the task with
`terminal-tier-recurrence` for human diagnosis.

Terminal-tier resume is fail-closed. The paired
`check-terminal-tier-resume.sh` and `.ps1` validators require a changed,
hash-bound task contract, a hash-bound diagnosis reference, and an explicit
human authority/timestamp matching the task's `Diagnosis Reference:` and
`Terminal Reapproval:` fields. Chat history cannot replace these artifacts.

Deterministic parsing, validation, hashing, and state transitions use scripts rather than model routing.

## Hook Guard Enforcement Matrix

The sdd-hook-guard.py script enforces the following checks at the PreToolUse event:

1. **Kill Switch (AGENT_STOP)** — Lines 192–223
   - If file exists at `$CLAUDE_PROJECT_DIR` or git root, deny all tool calls
   - Never bypassed by sudo mode
   - Fail-closed: any OSError is treated as "cannot verify, block"

2. **Approval Guard** — Lines 65–104
   - For any write to a file ending in `tasks.md`:
     - Count `Approval: Approved` on `+` lines minus count on `−` lines
     - Subtract `Second Approval: Approved` occurrences (substring match) to avoid double-counting
     - If net increase > 0: deny (unless valid, unexpired SDD_SUDO exists)
   - Applies to Edit, Write, apply_patch, and Bash commands

3. **WFI Approval Guard** — Lines 112–124
   - For any write to a file matching `docs/workflow-improvements/*.md`:
     - Count `Status: Approved` lines on `+` vs `−`
     - If net increase > 0: deny (NO BYPASS, even under sudo)
   - Applies to Edit, Write, apply_patch, and Bash commands

4. **Second Approval Guard** — Lines 126–130
   - For tasks.md: count `Second Approval: Approved` lines on `+` vs `−`
   - If net increase > 0: deny (NO BYPASS, even under sudo)
   - Applies to Edit, Write, apply_patch, and Bash commands
   - Note: check-task-state also verifies approver names are distinct

5. **Agent-Role Guard** — Lines 133–145
   - For any write to a file matching `.codex/agents/*.toml`:
     - If the resulting file does NOT contain `developer_instructions` field, deny
   - Applies to Edit, Write, apply_patch, and Bash commands targeting agent role paths

6. **SDD_SUDO Write Guard** — Lines 54–60 (Bash pattern) and parallel logic in Python
   - For any Bash/shell command on the command line:
     - If it contains write operators (`>`, `>>`, `tee`, `cp`, `mv`, `rm`, `Set-Content`, etc.) AND targets SDD_SUDO, deny
   - Applies to Bash, shell, exec_command, and exec tool calls

## Convention-Based Constraints (Not Hook-Guarded)

These are enforced by agent instructions in the TOML `developer_instructions` field, not by the hook guard. Agents are expected to follow them; violations are detected by code review or downstream verification:

| Constraint | Where It Matters | Notes |
|---|---|---|
| **Do not invoke generate-evidence-bundle or check-* gate scripts directly** | sdd-investigator, sdd-evaluator | Agents should read spec/implementation and return findings; gates are invoked by skills, not agents. Agents might call `npm test` or `python -m pytest` to gather evidence, but they do not invoke gate scripts. |
| **Do not attempt to read signing keys** | sdd-investigator, sdd-evaluator | `developer_instructions` forbids reading `~/.sdd/*` or env vars `SDD_SUDO_KEY`, `SDD_EVIDENCE_KEY`. Bash sandbox also blocks file reads outside the project tree (convention). |
| **Do not modify evidence bundles or quality-gate reports** | sdd-investigator, sdd-evaluator | Bundles are generated and signed by out-of-process scripts (generate-evidence-bundle, quality-gate skill). Agents produce input (findings) that skills incorporate, not the other way around. |
| **Do not self-approve or self-evaluate** | sdd-investigator, sdd-evaluator (evaluator especially) | Evaluator is positioned as independent; it should not read its own previous output or the investigator's findings and cross-contaminate. Separate contexts enforced by skill orchestration, not by hook guard. |

## Summary: Hook-Guard vs. Convention

**Hook-Guard Enforces (Fail-Closed):**
- Kill switch (AGENT_STOP)
- Approval/Second Approval/WFI approval count increases in tasks.md
- SDD_SUDO file writes
- Agent role file completeness (`developer_instructions` present)

**Convention Enforces (Agent Instructions + Code Review):**
- Read-only behavior in sandbox-mode agents (no file writes at all)
- Not invoking gate scripts directly
- Not reading signing keys
- Not self-approving or mixing contexts

The two layers work together: the hook guard catches direct attempts to set approval/sudo fields; agent instructions guide the agent to not attempt them in the first place. This defense-in-depth approach ensures both prompt-injection resistance and deterministic enforcement.
