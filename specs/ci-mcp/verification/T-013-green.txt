## Cycle 2 (evaluator finding fix)

Independent quality-gate evaluator confirmed a Major: security-spec.md B2's
DoS mitigation for job logs ("リングバッファ + 256 KiB 上限(末尾優先)") was
declared but not implemented — `githubGetText` read the entire upstream
job-log body via `.text()` before any truncation, so peak memory scaled with
the full log size (unbounded), and `truncateLogTail` made a second full copy.
Fixed by giving `githubGetText` (github-client.ts) a bounded streaming read
path (`readBoundedTail`) that consumes `response.body` chunk-by-chunk and
retains only a tail window bounded by 262144 + 65536 (margin) bytes,
regardless of upstream body size; `truncateLogTail` (tools/actions.ts) still
owns the final byte-exact, UTF-8-safe cut to <= 262144 bytes, now operating
on the already-bounded tail instead of the full body. New regression tests
added: a >1 MiB streaming body (tail-priority truncation still holds), a
multibyte-character body crossing the cut boundary (no replacement
characters), a small streaming body (whole log, untruncated), and a direct
unit test of `readBoundedTail` proving the retention buffer's
`maxRetainedBytes` never exceeds capBytes + marginBytes + one chunk even for
a multi-MiB stream. Full suite grows from 140 to 146 tests, all green below.
